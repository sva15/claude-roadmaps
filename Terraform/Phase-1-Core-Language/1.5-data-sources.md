> **Priority:** 🔴 Must-Know (Used in virtually every Terraform project)  
> **Previous:** [1.4 — count and for_each](./1.4-count-and-for-each.md)

## 1. First Principles — Start Simple

### What Problem Does This Solve?

Not everything in your infrastructure is created by Terraform. You need to reference things that already exist:
- The **latest Amazon Linux AMI** — AWS publishes new AMIs regularly, you don't want to hardcode IDs
- **Availability zones** in the current region — different regions have different AZs
- A **VPC created by another team** — your compute resources deploy into it
- The **current AWS account ID** — used for ARN construction
- An **EKS cluster** — your Kubernetes provider needs its endpoint and CA certificate

Data sources let Terraform **read** existing infrastructure without managing it.

### The Fundamental Distinction

```
RESOURCE:     "I want this thing to EXIST. Create it, track it, update it, delete it."
DATA SOURCE:  "This thing already EXISTS. Just TELL ME about it."
```

| | Resource | Data Source |
|---|---------|------------|
| Prefix in code | `resource` | `data` |
| Creates infrastructure? | ✅ Yes | ❌ No (read-only) |
| Managed by Terraform? | ✅ Yes (in state, lifecycle managed) | ❌ No (just a snapshot) |
| Can be destroyed? | ✅ `terraform destroy` removes it | ❌ Nothing to destroy |
| Appears in state? | ✅ Full attributes stored | ✅ Read values cached (for plan comparison) |
| API call type | Create/Update/Delete | Describe/Get/List (read-only) |

### Real-World Analogy

| Concept | Analogy |
|---------|---------|
| **Resource** | Building a house — you design it, construct it, maintain it, and can demolish it |
| **Data Source** | Looking up a neighbor's house on Google Maps — you can see its address and location, but you don't own or control it |

## 2. Terraform Mental Model

### Where Data Sources Fit in the Reconciliation Loop

```
terraform plan:
  1. REFRESH data sources → call AWS APIs to fetch current values
  2. REFRESH resources → call AWS APIs to check current resource state
  3. COMPARE desired (code) vs known (state)
  4. Generate plan

KEY: Data sources are ALWAYS refreshed on every plan.
     They represent the LATEST state of external infrastructure.
     This is different from resources, which are only refreshed
     to check for drift.
```

### Data Flow

```
data "aws_ami" "latest"          →  data.aws_ami.latest.id
  (query AWS for latest AMI)           │
                                       ▼
resource "aws_instance" "web"    ←  ami = data.aws_ami.latest.id
  (create instance with that AMI)
```

## 3. Core Concept — Deep Dive

### The 6 Data Sources You'll Use in Every Real Project

#### 1. `aws_ami` — Find the Right Machine Image

```hcl
# Find the latest Amazon Linux 2023 AMI
data "aws_ami" "al2023" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-x86_64"]
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }

  filter {
    name   = "architecture"
    values = ["x86_64"]
  }
}

# Usage
resource "aws_instance" "web" {
  ami           = data.aws_ami.al2023.id     # Always the latest AMI
  instance_type = "t3.micro"
}
```

**Why not hardcode the AMI ID?**
- AMI IDs are **region-specific** — `ami-abc123` in us-east-1 doesn't exist in eu-west-1
- AWS **publishes new AMIs** with patches — hardcoding means you're always on old images
- Data source automatically picks the newest matching AMI on every plan

#### 2. `aws_availability_zones` — Region-Agnostic Code

```hcl
data "aws_availability_zones" "available" {
  state = "available"

  # Optional: exclude local zones
  filter {
    name   = "opt-in-status"
    values = ["opt-in-not-required"]
  }
}

# Usage: deploy subnets across available AZs
resource "aws_subnet" "public" {
  for_each          = { for i, az in 
    slice(data.aws_availability_zones.available.names, 0, 2) : az => i 
  }
  availability_zone = each.key
  cidr_block        = cidrsubnet(var.vpc_cidr, 8, each.value)
  vpc_id            = aws_vpc.main.id
}
```

**Why?** Your code works in **any region** without changes. `us-east-1` has 6 AZs, `ap-south-1` has 3 — the data source adapts automatically.

#### 3. `aws_caller_identity` — Who Am I?

```hcl
data "aws_caller_identity" "current" {}

# Usage
locals {
  account_id = data.aws_caller_identity.current.account_id
}

# Build ARNs dynamically
resource "aws_iam_policy" "app" {
  policy = jsonencode({
    Statement = [{
      Effect   = "Allow"
      Action   = ["s3:GetObject"]
      Resource = "arn:aws:s3:::my-bucket-${local.account_id}/*"
    }]
  })
}
```

**Why?** Avoids hardcoding account IDs — the same code works across dev/staging/prod AWS accounts.

#### 4. `aws_region` — Current Region

```hcl
data "aws_region" "current" {}

# Usage
output "ecr_url" {
  value = "${local.account_id}.dkr.ecr.${data.aws_region.current.name}.amazonaws.com"
}
```

#### 5. `aws_vpc` — Reference an Existing VPC

```hcl
# Look up a VPC by tag
data "aws_vpc" "existing" {
  filter {
    name   = "tag:Name"
    values = ["production-vpc"]
  }
}

# Or by ID
data "aws_vpc" "by_id" {
  id = "vpc-abc123"
}

# Usage: deploy into someone else's VPC
resource "aws_subnet" "new" {
  vpc_id     = data.aws_vpc.existing.id
  cidr_block = "10.0.100.0/24"
}
```

**When to use:** Your team deploys compute resources into a VPC managed by the networking team's Terraform config.

#### 6. `aws_subnets` — Find Subnets by Filter

```hcl
# Find all private subnets in a VPC
data "aws_subnets" "private" {
  filter {
    name   = "vpc-id"
    values = [data.aws_vpc.existing.id]
  }
  filter {
    name   = "tag:Tier"
    values = ["private"]
  }
}

# Usage: deploy RDS across private subnets
resource "aws_db_subnet_group" "main" {
  name       = "my-db-subnet-group"
  subnet_ids = data.aws_subnets.private.ids
}
```

### Filter Blocks — Querying by Attributes

```hcl
filter {
  name   = "tag:Environment"     # Tag-based filter
  values = ["prod"]
}

filter {
  name   = "state"               # Attribute-based filter
  values = ["available"]
}

filter {
  name   = "name"                # Wildcards supported
  values = ["al2023-ami-*"]
}
```

**Multiple filters = AND logic** — all must match.  
**Multiple values in one filter = OR logic** — any value matches.

### Data Sources in Modules

```hcl
# PATTERN: Use data sources inside modules for self-contained queries
module "compute" {
  source = "./modules/compute"
  vpc_id = module.vpc.vpc_id
  # No need to pass AMI ID — module finds it internally
}

# modules/compute/main.tf
data "aws_ami" "al2023" {
  most_recent = true
  owners      = ["amazon"]
  filter { name = "name"; values = ["al2023-ami-*-x86_64"] }
}

resource "aws_instance" "web" {
  ami = data.aws_ami.al2023.id    # Self-contained
}
```

## 4. Under the Hood — Critical

### Data Sources and the Plan Cycle

```
EVERY terraform plan:
  1. Terraform executes ALL data source queries (API calls)
  2. Results are stored temporarily in state
  3. These values feed into resource planning
  4. On next plan, data sources are queried AGAIN (fresh data)

IMPLICATION: Data source values can CHANGE between plans!
  Plan 1: data.aws_ami.al2023.id = "ami-111"
  (AWS publishes new AMI)
  Plan 2: data.aws_ami.al2023.id = "ami-222"
  Plan: aws_instance.web will be REPLACED (AMI changed!)
```

### Data Sources in State

```json
{
  "mode": "data",           // ← "data" vs "managed" for resources
  "type": "aws_ami",
  "name": "al2023",
  "provider": "provider[\"registry.terraform.io/hashicorp/aws\"]",
  "instances": [{
    "attributes": {
      "id": "ami-0abc123def",
      "name": "al2023-ami-2023.5.20241001.0-kernel-6.1-x86_64",
      "creation_date": "2024-10-01T00:00:00.000Z"
    }
  }]
}
```

### What Happens When a Data Source Returns Nothing

```hcl
data "aws_vpc" "missing" {
  filter {
    name   = "tag:Name"
    values = ["does-not-exist"]
  }
}

# Error: no matching VPC found
# Data sources FAIL if they return zero results (for singular queries)
# Data sources return empty list if they return zero results (for plural queries like aws_subnets)
```

## 5. Failure Scenarios — Production Focus

### Scenario 1: Data Source Returns Multiple Results

| | Details |
|--|---------|
| **Symptom** | `Error: multiple VPCs matched; use additional constraints to reduce matches` |
| **Root Cause** | Filter is too broad — multiple VPCs have the same tag |
| **Fix** | Add more specific filters, or use `most_recent = true` for AMIs |

### Scenario 2: AMI Change Triggers Instance Replacement

| | Details |
|--|---------|
| **Symptom** | Plan shows `forces replacement` on your EC2 instance |
| **Root Cause** | AWS published a new AMI matching your filter → `data.aws_ami` returns a new ID |
| **Fix (option 1)** | Accept the replacement (instances should be replaceable) |
| **Fix (option 2)** | `lifecycle { ignore_changes = [ami] }` — manage AMI updates separately |
| **Fix (option 3)** | Pin a specific AMI name pattern instead of using `most_recent` |

### Scenario 3: Cross-Account Data Source Failure

| | Details |
|--|---------|
| **Symptom** | `Error: Access Denied` when querying data source |
| **Root Cause** | IAM role lacks `Describe*` permissions for the resource type |
| **Fix** | Add `ec2:DescribeImages`, `ec2:DescribeVpcs`, etc. to the IAM policy |

## 6. Data Sources vs terraform_remote_state

| | Data Source | `terraform_remote_state` |
|--|-----------|--------------------------|
| What it reads | AWS API directly | Another Terraform state's outputs |
| Requires | IAM permissions to AWS | Access to the state bucket |
| Coupling | Loose (just AWS) | Tight (depends on other config's outputs) |
| When to use | Query existing resources | Reference outputs from another Terraform config |
| Staleness | Always current (API call on every plan) | Current as of last apply of the source config |

## 10. Interview Preparation

### Q1: *"What's the difference between a resource and a data source?"* (Screening — asked in virtually every Terraform interview)

**Strong Answer:**
"A resource declares something Terraform should create and manage — it has a full lifecycle of create, update, and destroy. A data source reads something that already exists, either because it's managed by another system, another Terraform config, or was created manually. Data sources are read-only queries. Both appear in state, but data sources are re-queried on every plan to get fresh values, while resources are only refreshed to detect drift."

### Q2: *"Why would you use `data "aws_ami"` instead of hardcoding the AMI ID?"* (Mid)

**Strong Answer:**
"Three reasons: portability, freshness, and maintenance. AMI IDs are region-specific — hardcoding breaks when you deploy to a different region. Data sources automatically pick the latest AMI matching your filters, meaning you get security patches. And you don't have to manually update AMI IDs across configs when a new version is published. The trade-off: a new AMI might trigger instance replacement, so for sensitive workloads I'd use `lifecycle { ignore_changes = [ami] }` to control when updates happen."

### Q3: *"How do you reference infrastructure managed by another team?"* (Mid)

**Strong Answer:**
"Two options: data sources or `terraform_remote_state`. Data sources query AWS directly — `data 'aws_vpc' 'existing'` with filters to find the VPC by tag or ID. This is loosely coupled — I just need IAM permissions to describe the VPC. `terraform_remote_state` reads the other team's state outputs — more structured but creates a dependency on their state format. I prefer data sources for stable, well-tagged resources and remote state when I need multiple outputs that are specifically curated by the other team."

### Q4: *"A `data.aws_ami` query returns a different AMI ID than last time. What happens?"* (Senior)

**Strong Answer:**
"Terraform re-queries data sources on every plan. If the AMI ID changes, any resource using `data.aws_ami.x.id` will show a plan to update or replace. For EC2 instances, changing the AMI forces replacement — Terraform will destroy and recreate the instance. To control this: use `lifecycle { ignore_changes = [ami] }` to decouple AMI updates from instance lifecycle, or pin a more specific filter pattern to reduce unexpected changes."

### Q5: *"Data sources refresh on every plan. Can this cause problems?"* (Senior)

**Strong Answer:**
"Yes — two main issues. First, non-deterministic plans: if a data source returns different values between plan and apply (because the underlying data changed in between), the apply may produce different results than what was shown in the plan. Second, performance: data sources that query large datasets (like all AMIs matching a broad filter) can slow down plan. For critical infrastructure, I sometimes use `terraform_remote_state` instead, which gives a stable snapshot rather than a live query."

## 13. Cheat Sheet

```hcl
# Data source syntax
data "<provider>_<type>" "<name>" {
  filter { name = "..."; values = ["..."] }
  # OR specific lookup
  id = "..."
}

# Reference
data.<type>.<name>.<attribute>
data.aws_ami.al2023.id
data.aws_caller_identity.current.account_id
data.aws_availability_zones.available.names

# The 6 essentials:
data "aws_ami"                    # Find AMI by filters
data "aws_availability_zones"     # List AZs in current region
data "aws_caller_identity"        # Current account ID/ARN
data "aws_region"                 # Current region name
data "aws_vpc"                    # Look up existing VPC
data "aws_subnets"                # Find subnets by filter
```

```
RULE: If Terraform creates it → resource
RULE: If it already exists   → data source
RULE: Data sources refresh on EVERY plan
RULE: Singular data sources (aws_vpc) FAIL if no match
RULE: Plural data sources (aws_subnets) return empty list if no match
```
