# Topic 1.4 — count and for_each: Creating Multiple Resources

> **Phase:** 1 — Core Language  
> **Topic:** count, for_each, and managing resource multiplicity in state  
> **Priority:** 🔴 Must-Know (Foundation)  
> **Previous:** [1.3 — Providers and Dependencies](./1.3-providers-and-dependencies.md)

---

## 1. First Principles — Start Simple

### What Problem Does This Solve?

Real infrastructure is almost never "one of each." You need:
- **3 subnets** across 3 availability zones
- **5 S3 buckets** for different teams
- **10 IAM users** with similar policies
- **2 environments** (dev, prod) with identical structure

Without `count` or `for_each`, you'd copy-paste the same resource block N times. This violates DRY (Don't Repeat Yourself) and becomes unmaintainable at scale.

But here's the critical question: **when you create multiple resources, how does Terraform track them in state?** This is where `count` and `for_each` differ fundamentally — and choosing wrong causes production disasters.

### Real-World Analogy

Think of **apartment buildings**:

| `count` | `for_each` |
|---------|-----------|
| Apartments numbered 1, 2, 3, 4, 5 | Apartments named by tenant: "Smith", "Jones", "Patel" |
| Remove apartment 3 → apartments 4 and 5 renumber to 3 and 4 | Remove "Jones" → "Smith" and "Patel" are untouched |
| Renumbering = chaos (wrong mail, wrong keys) | Named keys = stable (everyone keeps their apartment) |

**This is the single most important thing in this topic:** `count` uses numeric indexes, `for_each` uses stable keys. Using `count` when you need stable identity will cause Terraform to destroy and recreate resources during refactoring.

## 2. Terraform Mental Model

### How count and for_each Change the State

**Without count/for_each (single resource):**
```
State:  aws_subnet.public → subnet-abc123
```

**With count:**
```
State:  aws_subnet.public[0] → subnet-abc123
        aws_subnet.public[1] → subnet-def456
        aws_subnet.public[2] → subnet-ghi789
```

**With for_each:**
```
State:  aws_subnet.public["us-east-1a"] → subnet-abc123
        aws_subnet.public["us-east-1b"] → subnet-def456
        aws_subnet.public["us-east-1c"] → subnet-ghi789
```

**The key insight for the plan → apply cycle:**

When you remove an item:
- **count**: All items after the removed one shift down. Terraform sees new index → new resource. Plans destroy + recreate for every shifted resource.
- **for_each**: Only the removed key is destroyed. All other keys remain mapped to the same real resources.

### Impact on Idempotency

Both `count` and `for_each` maintain idempotency — unchanged configurations produce no changes. But **modifying the collection** has very different behaviors.

## 3. Core Concept — Deep Dive

### count — Numeric Index-Based Multiplicity

```hcl
resource "aws_subnet" "public" {
  count = 3

  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(var.vpc_cidr, 8, count.index)
  availability_zone = data.aws_availability_zones.available.names[count.index]

  tags = {
    Name = "${local.name_prefix}-public-${count.index + 1}"
  }
}

# Reference: aws_subnet.public[0].id, aws_subnet.public[1].id
# Splat:     aws_subnet.public[*].id  (returns list of all IDs)
```

**How count works internally:**

```
count = 3 tells Terraform:
  1. Create aws_subnet.public[0] with count.index = 0
  2. Create aws_subnet.public[1] with count.index = 1
  3. Create aws_subnet.public[2] with count.index = 2

State records:
  aws_subnet.public[0] → subnet-aaa
  aws_subnet.public[1] → subnet-bbb
  aws_subnet.public[2] → subnet-ccc
```

**The count trap — why order matters:**

```
BEFORE: count = 3
  [0] → subnet-aaa (us-east-1a, 10.0.0.0/24)
  [1] → subnet-bbb (us-east-1b, 10.0.1.0/24)
  [2] → subnet-ccc (us-east-1c, 10.0.2.0/24)

AFTER: Remove the middle one (change count to 2)
  [0] → subnet-aaa (us-east-1a, 10.0.0.0/24)  ← stays
  [1] → ???

  Terraform's plan:
    ~ aws_subnet.public[1] will be UPDATED
      (changes AZ from us-east-1b to us-east-1c, CIDR changes)
    - aws_subnet.public[2] will be DESTROYED

  Result: subnet-bbb gets MODIFIED (potentially destructive!)
          subnet-ccc gets DESTROYED
          This is NOT what you wanted!
```

**When count IS appropriate:**
```hcl
# Identical resources where order doesn't matter
resource "aws_nat_gateway" "main" {
  count = var.enable_nat ? 1 : 0    # Toggle resource on/off
  # ...
}

# Truly identical copies
resource "aws_eip" "nat" {
  count  = var.az_count
  domain = "vpc"
}
```

### for_each — Key-Based Stable Multiplicity

```hcl
# With a MAP
resource "aws_subnet" "public" {
  for_each = {
    "us-east-1a" = { cidr = "10.0.0.0/24", index = 0 }
    "us-east-1b" = { cidr = "10.0.1.0/24", index = 1 }
    "us-east-1c" = { cidr = "10.0.2.0/24", index = 2 }
  }

  vpc_id            = aws_vpc.main.id
  cidr_block        = each.value.cidr
  availability_zone = each.key

  tags = {
    Name = "${local.name_prefix}-public-${each.key}"
  }
}

# Reference: aws_subnet.public["us-east-1a"].id
# All IDs:   values(aws_subnet.public)[*].id
#   or:      [for k, s in aws_subnet.public : s.id]
```

**How for_each works internally:**

```
for_each = { "us-east-1a" = {...}, "us-east-1b" = {...} }

Terraform creates:
  aws_subnet.public["us-east-1a"] with each.key = "us-east-1a"
  aws_subnet.public["us-east-1b"] with each.key = "us-east-1b"

State records:
  aws_subnet.public["us-east-1a"] → subnet-aaa
  aws_subnet.public["us-east-1b"] → subnet-bbb
```

**The for_each advantage — stable keys:**

```
BEFORE:
  ["us-east-1a"] → subnet-aaa
  ["us-east-1b"] → subnet-bbb
  ["us-east-1c"] → subnet-ccc

AFTER: Remove "us-east-1b"
  ["us-east-1a"] → subnet-aaa  ← UNTOUCHED
  ["us-east-1c"] → subnet-ccc  ← UNTOUCHED
  
  Plan: - aws_subnet.public["us-east-1b"] will be DESTROYED
  
  Only the removed key is destroyed. Everything else is stable.
```

**for_each with a SET:**
```hcl
# When you don't need values, just unique keys
resource "aws_iam_user" "team" {
  for_each = toset(["alice", "bob", "charlie"])
  name     = each.key    # each.key == each.value for sets
}
```

**for_each with a MAP derived from a variable:**
```hcl
variable "subnets" {
  type = map(object({
    cidr = string
    tier = string
  }))
  default = {
    "public-1a"  = { cidr = "10.0.0.0/24",  tier = "public" }
    "public-1b"  = { cidr = "10.0.1.0/24",  tier = "public" }
    "private-1a" = { cidr = "10.0.10.0/24", tier = "private" }
    "private-1b" = { cidr = "10.0.11.0/24", tier = "private" }
  }
}

resource "aws_subnet" "this" {
  for_each = var.subnets

  vpc_id     = aws_vpc.main.id
  cidr_block = each.value.cidr

  tags = {
    Name = "${local.name_prefix}-${each.key}"
    Tier = each.value.tier
  }
}
```

### for Expressions — Transforming Collections

for expressions are **not** for creating resources — they transform data:

```hcl
# List → List (filter public subnets)
output "public_subnet_ids" {
  value = [for k, s in aws_subnet.this : s.id if s.tags["Tier"] == "public"]
}

# Map → Map (transform keys/values)
output "subnet_map" {
  value = { for k, s in aws_subnet.this : k => s.id }
  # Result: { "public-1a" = "subnet-aaa", "public-1b" = "subnet-bbb", ... }
}

# List → Map
locals {
  az_list = ["us-east-1a", "us-east-1b", "us-east-1c"]
  az_map  = { for i, az in local.az_list : az => i }
  # Result: { "us-east-1a" = 0, "us-east-1b" = 1, "us-east-1c" = 2 }
}
```

### The Decision Rule

```
┌────────────────────────────────────────────────────┐
│        WHEN TO USE count VS for_each               │
│                                                    │
│  ┌─────────────────────┐  ┌─────────────────────┐  │
│  │       count          │  │      for_each        │  │
│  │                     │  │                     │  │
│  │ • Toggle on/off     │  │ • EVERYTHING else   │  │
│  │   (count = 0 or 1)  │  │                     │  │
│  │                     │  │ • Resources with     │  │
│  │ • Truly identical   │  │   distinct identity  │  │
│  │   copies where      │  │                     │  │
│  │   position doesn't  │  │ • Named resources   │  │
│  │   matter            │  │   (subnets, users,   │  │
│  │                     │  │    buckets)          │  │
│  └─────────────────────┘  └─────────────────────┘  │
│                                                    │
│  Default: USE for_each                             │
│  count is the exception, not the rule              │
└────────────────────────────────────────────────────┘
```

## 4. Under the Hood — Critical

### State Representation

**count resources in state:**
```json
{
  "resources": [
    {
      "module": "module.vpc",
      "mode": "managed",
      "type": "aws_subnet",
      "name": "public",
      "instances": [
        {
          "index_key": 0,
          "attributes": {
            "id": "subnet-aaa",
            "cidr_block": "10.0.0.0/24"
          }
        },
        {
          "index_key": 1,
          "attributes": {
            "id": "subnet-bbb",
            "cidr_block": "10.0.1.0/24"
          }
        }
      ]
    }
  ]
}
```

**for_each resources in state:**
```json
{
  "resources": [
    {
      "module": "module.vpc",
      "mode": "managed",
      "type": "aws_subnet",
      "name": "public",
      "instances": [
        {
          "index_key": "us-east-1a",
          "attributes": {
            "id": "subnet-aaa", 
            "cidr_block": "10.0.0.0/24"
          }
        },
        {
          "index_key": "us-east-1b",
          "attributes": {
            "id": "subnet-bbb",
            "cidr_block": "10.0.1.0/24"
          }
        }
      ]
    }
  ]
}
```

**The difference:** `index_key` is a number for count, a string for for_each. This is why for_each keys are stable — they're named anchors, not positions.

### How the Dependency Graph Handles Multiple Instances

```
With count = 2:
  aws_subnet.public[0]  ─┐
                          ├──▶ aws_vpc.main
  aws_subnet.public[1]  ─┘

Each index is a SEPARATE node in the dependency graph.
Terraform can create [0] and [1] in parallel (both depend on VPC).
```

### for_each Requires Known Values at Plan Time

```hcl
# THIS FAILS:
resource "aws_subnet" "public" {
  for_each = toset(aws_availability_zones.available.names)
  # Error: for_each depends on resource attributes not yet known
}

# FIX: Use a data source (known at plan time)
data "aws_availability_zones" "available" {
  state = "available"
}

resource "aws_subnet" "public" {
  for_each = toset(data.aws_availability_zones.available.names)
  # Works: data sources are resolved during refresh, before plan
}
```

This is a common gotcha: **for_each keys must be known before apply** — they can't depend on values that are only `(known after apply)`.

## 5. Project Implementation — Evolving Project

### Refactoring the Project to Use for_each

**Before (count — from Topics 1.1-1.3):**
```hcl
resource "aws_subnet" "public" {
  count                   = 2
  vpc_id                  = aws_vpc.main.id
  cidr_block              = cidrsubnet(var.vpc_cidr, 8, count.index)
  availability_zone       = data.aws_availability_zones.available.names[count.index]
  map_public_ip_on_launch = true

  tags = {
    Name = "${local.name_prefix}-public-${count.index + 1}"
  }
}
# State: aws_subnet.public[0], aws_subnet.public[1]
```

**After (for_each — stable keys):**
```hcl
locals {
  # Build a map of AZ name → subnet config
  public_subnets = {
    for i, az in slice(data.aws_availability_zones.available.names, 0, local.az_count) :
    az => {
      cidr  = cidrsubnet(var.vpc_cidr, 8, i)
      index = i
    }
  }

  private_subnets = {
    for i, az in slice(data.aws_availability_zones.available.names, 0, local.az_count) :
    az => {
      cidr  = cidrsubnet(var.vpc_cidr, 8, i + 10)
      index = i
    }
  }
}

resource "aws_subnet" "public" {
  for_each = local.public_subnets

  vpc_id                  = aws_vpc.main.id
  cidr_block              = each.value.cidr
  availability_zone       = each.key
  map_public_ip_on_launch = true

  tags = {
    Name = "${local.name_prefix}-public-${each.key}"
    Tier = "public"
  }
}

resource "aws_subnet" "private" {
  for_each = local.private_subnets

  vpc_id            = aws_vpc.main.id
  cidr_block        = each.value.cidr
  availability_zone = each.key

  tags = {
    Name = "${local.name_prefix}-private-${each.key}"
    Tier = "private"
  }
}

# State: aws_subnet.public["us-east-1a"], aws_subnet.public["us-east-1b"]
```

**Updated route table associations:**
```hcl
resource "aws_route_table_association" "public" {
  for_each       = aws_subnet.public
  subnet_id      = each.value.id
  route_table_id = aws_route_table.public.id
}
```

**Updated outputs:**
```hcl
output "public_subnet_ids" {
  description = "Map of AZ to public subnet ID"
  value       = { for k, s in aws_subnet.public : k => s.id }
}

output "private_subnet_ids" {
  description = "Map of AZ to private subnet ID"
  value       = { for k, s in aws_subnet.private : k => s.id }
}

output "public_subnet_id_list" {
  description = "List of public subnet IDs (for modules that need a list)"
  value       = [for s in aws_subnet.public : s.id]
}
```

### Migration: count → for_each

> ⚠️ **WARNING**: You cannot simply change `count` to `for_each` and apply. Terraform sees different state keys and will plan to **destroy all count-indexed resources** and **create new for_each-keyed resources**.

**Safe migration process:**
```bash
# 1. Move the existing state entries to the new keys
terraform state mv 'aws_subnet.public[0]' 'aws_subnet.public["us-east-1a"]'
terraform state mv 'aws_subnet.public[1]' 'aws_subnet.public["us-east-1b"]'

# 2. Update the code from count to for_each

# 3. Run plan — should show NO changes (if state mv was correct)
terraform plan

# 4. If plan shows changes, fix the code to match the real attributes
```

Alternatively, in Terraform 1.1+, use **moved blocks**:
```hcl
moved {
  from = aws_subnet.public[0]
  to   = aws_subnet.public["us-east-1a"]
}

moved {
  from = aws_subnet.public[1]
  to   = aws_subnet.public["us-east-1b"]
}
```

## 6. Real-World Mapping

### In AWS

| Terraform Pattern | AWS Use Case |
|-------------------|-------------|
| `for_each` with AZ map | Subnets across AZs — stable, named |
| `count = var.enable_x ? 1 : 0` | NAT Gateway, Bastion host — toggle on/off |
| `for_each` with IAM users | Team members — add/remove without affecting others |
| `for_each` with S3 buckets | Per-team or per-service buckets |
| `count = var.az_count` | Identical EIPs or NAT gateways per AZ |

### In CI/CD Pipelines

```yaml
# Plan output with for_each is much clearer:
# aws_subnet.public["us-east-1a"] will be created
# aws_subnet.public["us-east-1b"] will be created
#
# vs count (less readable):
# aws_subnet.public[0] will be created
# aws_subnet.public[1] will be created
#
# Reviewers can immediately tell WHICH subnet is being modified
```

### In Multi-Team Environments

```hcl
# Common pattern: for_each over team configurations
variable "teams" {
```

## 7. Failure Scenarios — Production Focus

### Scenario 1: The count Reindex Disaster

| | Details |
|--|---------|
| **Symptom** | `terraform plan` shows destroy+recreate for resources you didn't change |
| **Root Cause** | Removed an item from a `count`-based list. Index [2] became [1], causing Terraform to see a "new" resource at index [1]. |
| **Terraform Behavior** | Plans to modify/destroy shifted resources. If applied, real resources get recreated — causing outage. |
| **How to Fix** | Migrate to `for_each` with `terraform state mv` or `moved` blocks. |

### Scenario 2: for_each Key Must Be Known

| | Details |
|--|---------|
| **Symptom** | `Error: Invalid for_each argument. The for_each value depends on resource attributes that cannot be determined until apply.` |
| **Root Cause** | for_each keys depend on a value that's `(known after apply)` — like a resource ID |
| **How to Fix** | Use data sources (known at plan time) or static maps/sets for keys. |

### Scenario 3: Duplicate for_each Key

| | Details |
|--|---------|
| **Symptom** | `Error: Two different items produced the key "us-east-1a" in this for_each expression.` |
| **Root Cause** | A `for` expression produced duplicate keys |
| **How to Fix** | Ensure keys are unique. Use `toset()` to deduplicate or add a unique suffix. |

### Scenario 4: Changing a for_each Key

| | Details |
|--|---------|
| **Symptom** | Plan shows destroy of resource with old key + create with new key |
| **Root Cause** | You changed a map key (e.g., "subnet-a" to "public-a"). Terraform treats this as deletion of one resource + creation of another. |
| **How to Fix** | Use `terraform state mv` or a `moved` block to rename the key in state. |

## 8. Debugging Playbook

```
PROBLEM: Unexpected destroy+recreate in plan

STEP 1: Check if using count
  └── Are resources indexed by number? [0], [1], [2]?
  └── Did you recently add/remove an item from the list?
  └── If yes → this is the count reindex problem

STEP 2: Check state addresses
  └── terraform state list | grep resource_name
  └── Are they indexed or keyed?
  └── Compare with what your code would produce

STEP 3: Fix with state mv
  └── terraform state mv 'resource[0]' 'resource["key"]'
  └── Or use moved blocks in code

STEP 4: Verify
  └── terraform plan → should show no changes
  └── If changes remain, check attribute values match

---

PROBLEM: "for_each value depends on resource attributes"

STEP 1: Identify the dynamic value
  └── Which attribute is "known after apply"?

STEP 2: Replace with static/data values
  └── Use a data source or variable for the keys
  └── Only VALUES can be dynamic, KEYS must be known

STEP 3: If impossible, use count as fallback
  └── count = length(var.list) with numeric indexes
  └── Accept the trade-off of index instability
```

```hcl
# Step 1: Create 3 S3 buckets with count
variable "bucket_names" {

resource "aws_s3_bucket" "counted" {
  count         = length(var.bucket_names)
  bucket_prefix = "${var.bucket_names[count.index]}-"
  force_destroy = true
}

# Step 2: Apply. Then remove "data" from the list:
# default = ["logs", "backup"]

# Step 3: Run terraform plan
# Observe: bucket[1] changes from "data" to "backup", bucket[2] is destroyed
```

```hcl
# Step 2: Apply. Then remove "data":
# for_each = toset(["logs", "backup"])
```

## 10. Interview Preparation

### Q1: *"What's the difference between count and for_each?"* (Screening)

**Strong Answer:**
"Both create multiple instances of a resource, but they differ in how they identify instances in state. `count` uses numeric indexes — changing the list order causes index shifts that trigger destroy+recreate of unmodified resources. `for_each` uses stable string keys — removing one key only affects that specific resource. In production, I default to `for_each` and only use `count` for toggling resources on/off with `count = condition ? 1 : 0`."

### Q2: *"You have 3 subnets created with `count = 3`. You need to remove the middle one. What happens and how do you handle it?"* (Mid)

**Strong Answer:**
"If I just reduce count to 2 and remove the middle list element, Terraform renumbers: index [2] becomes [1], causing the last subnet to be modified to match the middle one's position, and the old [2] is destroyed. This is the count reindex problem. To handle it safely, I'd migrate to `for_each` first using `terraform state mv` to remap the indexes to named keys, then remove the unwanted key. Or if I must stay with count, I'd use `terraform state rm` on the specific index, then adjust the code and count value."

### Q3: *"Why do for_each keys need to be known at plan time?"* (Senior)

**Strong Answer:**
"Terraform builds its dependency graph and plan before any API calls. It needs to know the full set of resource instances to plan creates, updates, and deletes. If keys depend on `(known after apply)` values, Terraform can't determine which resources will exist, making it impossible to produce a plan. This is a fundamental constraint of declarative planning — the graph must be fully known before execution."

### Q4: *"How would you iterate over both public and private subnets without code duplication?"* (Senior)

**Strong Answer:**
"I'd use a single variable that defines all subnets as a map with a 'tier' attribute, then use `for_each` on that map. If I need separate resource types or behaviors per tier, I'd filter with a for expression: `for_each = { for k, v in var.subnets : k => v if v.tier == 'public' }`. This keeps the data DRY while allowing tier-specific resource configuration."

### Q5: *"Walk me through migrating from count to for_each without downtime."* (Senior)

**Strong Answer:**
"First, I'd identify the mapping: which numeric index corresponds to which named key. Then I'd run `terraform state mv` for each resource to rename it in state — for example, `terraform state mv 'aws_subnet.public[0]' 'aws_subnet.public[\"us-east-1a\"]'`. After all state moves, I'd update the code from count to for_each with the matching keys. Finally, `terraform plan` should show zero changes, confirming the migration is correct. Alternatively, in Terraform 1.1+, I'd use `moved` blocks in code to declaratively express the remapping."

### Q6: *"When is count actually the right choice?"* (Trap)

**Strong Answer:**
"Two scenarios: First, toggling a resource on/off with `count = var.enabled ? 1 : 0` — the simplest conditional create. Second, creating truly identical copies where individual identity doesn't matter and you'll never remove one from the middle — like NAT Gateway EIPs where you always want exactly N. If you're ever unsure, default to `for_each` — it's strictly safer. The cost of using `for_each` when `count` would suffice is zero. The cost of using `count` when you need `for_each` is a production outage."

## 11. Advanced Insights

### Trade-Offs

| Aspect | count | for_each |
|--------|-------|----------|
| Simplicity | ✅ Simpler syntax | Requires map/set construction |
| Stability | ❌ Index-based, fragile | ✅ Key-based, stable |
| Refactoring safety | ❌ Reindex causes destroy+recreate | ✅ Only affected keys change |
| Conditional create | ✅ `count = 0/1` is idiomatic | Possible but less clean |
| Dynamic keys | N/A (always numeric) | ❌ Keys must be known at plan time |
| Plan readability | `resource[0]` — opaque | `resource["name"]` — self-documenting |

### Scaling Challenges

- **Large for_each maps (100+ items)**: Plan output gets very long. Use `-target` to plan specific resources during debugging.
- **Dynamic for_each keys from data sources**: If the data source returns different results (e.g., new AZ added), Terraform will plan to create the new resource automatically.
- **Nested for_each (modules)**: When a module has for_each and resources inside also use for_each, state addresses become deeply nested: `module.vpc["prod"].aws_subnet.public["us-east-1a"]`.

### patterns with flattened maps

```hcl
# Complex: multiple AZs × multiple tiers = many subnets
locals {
  subnets = merge([
    for tier in ["public", "private"] : {
      for i, az in local.azs :
      "${tier}-${az}" => {
        cidr = cidrsubnet(var.vpc_cidr, 8, tier == "public" ? i : i + 10)
        az   = az
        tier = tier
      }
    }
  ]...)
}

resource "aws_subnet" "all" {
  for_each          = local.subnets
  vpc_id            = aws_vpc.main.id
  cidr_block        = each.value.cidr
  availability_zone = each.value.az

  tags = {
    Name = each.key
    Tier = each.value.tier
  }
}
```

## 12. Common Mistakes

### Beginner Mistakes

| Mistake | Why It's Wrong | Fix |
|---------|---------------|-----|
| Using `count` for named resources | Index shift causes destroy+recreate | Use `for_each` |
| Not understanding `each.key` vs `each.value` | Confusing what to reference | `each.key` = map key, `each.value` = map value |
| Using `for_each` with a list | `for_each` requires map or set, not list | Wrap with `toset()` for simple lists |

### Intermediate Mistakes

| Mistake | Why It's Wrong | Fix |
|---------|---------------|-----|
| Dynamic for_each keys from resources | Fails: "depends on resource attributes" | Use data sources or static values for keys |
| Changing for_each keys | Treats as delete old + create new | Use `state mv` or `moved` blocks |
| `count = length(var.list)` with non-identical items | Combines count fragility with distinct resources | Use `for_each = toset(var.list)` instead |

### Production Mistakes

| Mistake | Why It's Wrong | Fix |
|---------|---------------|-----|
| Migrating count→for_each without `state mv` | All resources destroyed and recreated | Always use `state mv` or `moved` blocks |
| Not pinning for_each to known values | Data source changes can add/remove resources unexpectedly | Use variables, not data sources, for critical keys |
| No plan review after collection changes | A key removal destroys real infrastructure | Always review plan when modifying for_each maps |

## 13. Cheat Sheet

### Quick Decision

```
Need 0 or 1 of something? → count = condition ? 1 : 0
Need N identical things?   → count = N (if you'll never remove from middle)
Need named things?         → for_each = map or set
Unsure?                    → for_each (always safe)
```

### Syntax Reference

```hcl
# count
resource "r" "name" {
  count = 3
  # count.index → 0, 1, 2
}
# Reference: r.name[0].id
# All:       r.name[*].id

# for_each with map
resource "r" "name" {
  for_each = { key1 = val1, key2 = val2 }
  # each.key   → "key1", "key2"
  # each.value → val1, val2
}
# Reference: r.name["key1"].id
# All:       values(r.name)[*].id

# for_each with set
resource "r" "name" {
  for_each = toset(["a", "b", "c"])
  # each.key = each.value → "a", "b", "c"
}

# for expression (data transformation, not resource creation)
[for item in list : item.id]                          # list → list
{ for k, v in map : k => v.id }                       # map → map
[for k, v in map : v.id if v.public]                  # map → filtered list
```

### State Commands for count/for_each

```bash
# List all instances
terraform state list | grep aws_subnet

# Show a specific count instance
terraform state show 'aws_subnet.public[0]'

# Show a specific for_each instance
terraform state show 'aws_subnet.public["us-east-1a"]'

# Move count to for_each
terraform state mv 'aws_subnet.public[0]' 'aws_subnet.public["us-east-1a"]'
```
