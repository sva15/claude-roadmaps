# Topic 1.1 — How Terraform Actually Works: The Execution Model

> **Phase:** 1 — Core Language  
> **Topic:** The Execution Model (Refresh → Plan → Apply)  
> **Priority:** 🔴 Must-Know (Foundation)  
> **Terraform Version:** 1.11+

---

## 1. First Principles — Start Simple

### What Problem Does This Solve?

Imagine you have 200 servers, 50 databases, 30 load balancers, and hundreds of security rules across 3 AWS accounts. How do you:

- Know **exactly** what exists right now?
- Make a change to one thing **without accidentally breaking** another?
- **Undo** a bad change at 2 AM during an outage?
- Let **5 engineers** make changes without stepping on each other?

Before Terraform, you either:
1. **Clicked through the AWS console** — unrepeatable, unauditable, error-prone
2. **Wrote shell scripts** with AWS CLI — fragile, imperative ("do this, then do that"), and no rollback
3. **Used CloudFormation** — AWS-only, verbose, slow rollbacks

Terraform solves this by being a **declarative state reconciliation system**.

### Real-World Analogy

Think of a **GPS navigation system**:

| GPS | Terraform |
|-----|-----------|
| You tell it **where you want to go** (destination) | You write **what infrastructure you want** (.tf files) |
| It checks **where you are right now** (GPS signal) | It checks **what actually exists** (API calls to AWS) |
| It calculates the **route** (directions) | It calculates the **plan** (what to create/change/destroy) |
| You **drive** the route | It **applies** the changes |
| If you take a wrong turn, it **recalculates** | If someone manually changes infra, it **detects drift** and corrects |

The critical insight: **you never tell the GPS the exact turns to make**. You tell it the destination, and it figures out the path. Terraform works the same way — you describe the destination (desired state), not the journey (imperative steps).

### Why Terraform Needs This Concept

Without the execution model, Terraform would be just another scripting tool. The execution model is **the reason** Terraform is reliable:

- **Refresh** ensures Terraform isn't working with stale knowledge
- **Plan** gives you a preview (a contract) before anything changes
- **Apply** makes exactly the changes in the plan — nothing more, nothing less
- **Idempotency** means running it twice produces no extra changes

## 2. Terraform Mental Model

### How This Fits Into the Core Framework

```
┌─────────────────────────────────────────────────────────┐
│                  THE TERRAFORM TRIANGLE                  │
│                                                         │
│         DESIRED STATE              ACTUAL STATE          │
│        (your .tf files)          (real AWS infra)        │
│              │                         │                │
│              │         PLAN            │                │
│              └────── (the diff) ───────┘                │
│                        │                                │
│                   KNOWN STATE                           │
│                 (terraform.tfstate)                      │
│                                                         │
│  Plan = Desired - Known (after refreshing Known from    │
│         Actual)                                         │
└─────────────────────────────────────────────────────────┘
```

### Three States You Must Always Track

| State | Where It Lives | What It Represents |
|-------|---------------|-------------------|
| **Desired State** | `.tf` files in your repo | What you *want* to exist |
| **Known State** | `terraform.tfstate` file | What Terraform *last saw* in the real world |
| **Actual State** | AWS / cloud provider | What *actually* exists right now |

### The Plan → Apply Cycle

```
terraform plan:
  1. READ desired state from .tf files
  2. REFRESH known state by querying actual APIs
  3. COMPARE desired vs known
  4. OUTPUT the diff (what will be created/changed/destroyed)

terraform apply:
  1. EXECUTE the plan (make API calls)
  2. UPDATE known state after each successful resource operation
  3. REPORT results
```

### Dependency Graph

Terraform doesn't execute top-to-bottom like a script. It:
1. Parses all `.tf` files (order doesn't matter)
2. Builds a **directed acyclic graph (DAG)** of dependencies
3. Executes **independent resources in parallel**
4. Executes **dependent resources in order**

Example: If your EC2 instance references `aws_vpc.main.id`, Terraform knows the VPC must be created first. You don't need to specify this — Terraform infers it from the reference.

### Idempotency

This is the property that makes Infrastructure as Code reliable:

> Running `terraform apply` on unchanged code produces **zero changes**.

Why this matters:
- **Safety**: You can re-run apply after a failure without creating duplicates
- **Confidence**: "No changes" means your code matches reality
- **Drift detection**: If apply shows changes you didn't write, something changed outside Terraform

## 3. Core Concept — Deep Dive

### The Four Operations

#### 1. `terraform init`

**What it does:**
- Downloads provider plugins (e.g., AWS provider v5.x)
- Initializes the backend (where state is stored)
- Downloads modules from registries
- Creates the `.terraform/` directory and `.terraform.lock.hcl`

**When to run:**
- First time in a new Terraform directory
- After adding/changing providers or modules
- After changing backend configuration
- After cloning a repo (`.terraform/` is gitignored)

```bash
$ terraform init

Initializing the backend...
Initializing provider plugins...
- Finding hashicorp/aws versions matching "~> 5.0"...
- Installing hashicorp/aws v5.82.0...
- Installed hashicorp/aws v5.82.0 (signed by HashiCorp)

Terraform has been successfully initialized!
```

#### 2. `terraform plan` (The Refresh → Diff Cycle)

**Step-by-step internal process:**

```
Step 1: PARSE
  └── Read all .tf files in the current directory
  └── Build the resource graph
  └── Validate syntax and references

Step 2: REFRESH (happens automatically)
  └── For each resource in state:
      └── Call the AWS API to get current attributes
      └── Update the in-memory state to match reality
      └── This is how drift is detected

Step 3: DIFF
  └── Compare desired state (.tf) vs refreshed known state
  └── For each resource:
      ├── In code but NOT in state → WILL CREATE (+)
      ├── In state but NOT in code → WILL DESTROY (-)
      ├── In both but attributes differ → WILL UPDATE (~)
      └── In both and identical → NO CHANGE

Step 4: DISPLAY
  └── Show the execution plan with all changes
```

**Reading plan output:**

```hcl
# aws_vpc.main will be created
+ resource "aws_vpc" "main" {
    + arn                  = (known after apply)
    + cidr_block           = "10.0.0.0/16"    # From your code
    + id                   = (known after apply)
    + enable_dns_support   = true
    + tags                 = {
        + "Environment" = "dev"
        + "ManagedBy"   = "terraform"
      }
  }

# aws_instance.web will be updated in-place
~ resource "aws_instance" "web" {
    ~ tags = {
        ~ "Environment" = "dev" -> "staging"   # Changed
          "Name"        = "web-server"          # Unchanged
      }
  }

# aws_s3_bucket.old will be destroyed
- resource "aws_s3_bucket" "old" {
    - bucket = "my-old-bucket" -> null
  }

Plan: 1 to add, 1 to change, 1 to destroy.
```

**Symbols to know:**
| Symbol | Meaning |
|--------|---------|
| `+` | Create (new resource) |
| `-` | Destroy (remove resource) |
| `~` | Update in-place |
| `-/+` | Destroy and recreate (forced replacement) |
| `<=` | Read (data source) |

#### 3. `terraform apply`

**What actually happens:**

```
Step 1: Generate or use saved plan
Step 2: For each resource in dependency order:
  ├── Make the API call (e.g., CreateVpc)
  ├── Wait for the resource to reach ready state
  ├── Record the resource's real-world ID and attributes in state
  └── If error:
      ├── Mark resource as TAINTED in state
      ├── Stop dependent resources
      └── Continue independent resources (parallel execution)
Step 3: Write updated state file
Step 4: Display results
```

**Critical behavior on partial failure:**
- If resource 3 of 10 fails, resources 1 and 2 are **already created and in state**
- Resources 4-10 that depend on resource 3 are **skipped**
- Resources 4-10 that are independent of resource 3 may **still be created**
- The state file reflects the partial reality
- Running `terraform plan` again will show a plan to create only the missing resources

#### 4. `terraform destroy`

```
$ terraform destroy
# is equivalent to:
$ terraform apply -destroy

# Always preview first:
$ terraform plan -destroy
```

Destroy is **just an apply** where the desired state is "nothing." Terraform:
1. Reads the state file to know what exists
2. Plans to delete everything in reverse dependency order
3. Applies the deletions
4. Empties the state file

## 4. Under the Hood — Critical

### What Terraform Actually Does Internally

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   .tf files  │────▶│   Terraform  │────▶│  AWS APIs     │
│ (desired     │     │   Core       │     │ (actual       │
│  state)      │     │              │     │  state)       │
└──────────────┘     │  ┌────────┐  │     └──────────────┘
                     │  │ State  │  │
                     │  │ File   │  │
                     │  │(known) │  │
                     │  └────────┘  │
                     │              │
                     │  ┌────────┐  │
                     │  │Provider│  │
                     │  │Plugin  │  │
                     │  │(aws)   │  │
                     │  └────────┘  │
                     └──────────────┘
```

### State Interaction During Each Operation

| Operation | State Read? | State Write? | API Calls? |
|-----------|-------------|-------------|------------|
| `init` | No | No (creates backend config) | Downloads providers |
| `plan` | Yes (refresh) | No (in-memory only) | Yes (read-only refresh) |
| `apply` | Yes (refresh) | Yes (after each resource) | Yes (create/update/delete) |
| `destroy` | Yes | Yes (removes entries) | Yes (delete calls) |

### Provider Interaction

The AWS provider is a **separate binary** (plugin) that Terraform downloads during `init`. It:
- Translates Terraform resource definitions into AWS API calls
- Maps `aws_vpc` → `CreateVpc`, `ModifyVpcAttribute`, `DeleteVpc`
- Handles API retries, eventual consistency, and pagination
- Returns resource attributes back to Terraform for state storage

### Resource Lifecycle

Every resource goes through these states:

```
                    terraform apply
[Not Existing] ──────────────────────▶ [Created]
                                         │
                    terraform apply      │ attribute change
                    (update in-place)    │
                         ◀───────────────┘
                                         │
                    terraform apply      │ immutable attribute change
                    (destroy + recreate) │
                         ◀───────────────┘
                                         │
                    terraform destroy     │
[Destroyed] ◀────────────────────────────┘
                                         │
                    apply fails midway   │
[Tainted] ◀──────────────────────────────┘
  │
  │ next terraform apply
  └──────▶ Terraform destroys the tainted resource and recreates it
```

## 5. Project Implementation — Evolving Project

### Phase 1 Starting Point: Basic VPC + EC2

This is the beginning of your evolving project. By Phase 5, this simple setup will become a fully modularized, CI/CD-deployed, multi-environment production system.

**File: `providers.tf`**
```hcl
terraform {
  required_version = ">= 1.11.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = var.aws_region

  default_tags {
    tags = {
      Project     = "terraform-learning"
      ManagedBy   = "terraform"
      Environment = var.environment
    }
  }
}
```

**File: `variables.tf`**
```hcl
variable "aws_region" {
  type        = string
  description = "AWS region to deploy into"
  default     = "us-east-1"
}

variable "environment" {
  type        = string
  description = "Environment name"
  default     = "dev"
}

variable "vpc_cidr" {
  type        = string
  description = "CIDR block for the VPC"
  default     = "10.0.0.0/16"

  validation {
    condition     = can(cidrhost(var.vpc_cidr, 0))
    error_message = "Must be a valid CIDR block."
  }
}
```

**File: `main.tf`**
```hcl
# ---------------------------------------------------------
# Data Sources — query existing AWS information
# ---------------------------------------------------------
data "aws_availability_zones" "available" {
  state = "available"
}

# ---------------------------------------------------------
# VPC — the network foundation
# ---------------------------------------------------------
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_support   = true
  enable_dns_hostnames = true

  tags = {
    Name = "${var.environment}-vpc"
  }
}

# ---------------------------------------------------------
# Subnets — 2 public, 2 private across AZs
# ---------------------------------------------------------
resource "aws_subnet" "public" {
  count                   = 2
  vpc_id                  = aws_vpc.main.id
  cidr_block              = cidrsubnet(var.vpc_cidr, 8, count.index)
  availability_zone       = data.aws_availability_zones.available.names[count.index]
  map_public_ip_on_launch = true

  tags = {
    Name = "${var.environment}-public-${count.index + 1}"
    Tier = "public"
  }
}

resource "aws_subnet" "private" {
  count             = 2
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(var.vpc_cidr, 8, count.index + 10)
  availability_zone = data.aws_availability_zones.available.names[count.index]

  tags = {
    Name = "${var.environment}-private-${count.index + 1}"
    Tier = "private"
  }
}

# ---------------------------------------------------------
# Internet Gateway — public internet access
# ---------------------------------------------------------
resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id

  tags = {
    Name = "${var.environment}-igw"
  }
}

# ---------------------------------------------------------
# Route Table — direct public subnets to the internet
# ---------------------------------------------------------
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.main.id
  }

  tags = {
    Name = "${var.environment}-public-rt"
  }
}

resource "aws_route_table_association" "public" {
  count          = 2
  subnet_id      = aws_subnet.public[count.index].id
  route_table_id = aws_route_table.public.id
}
```

**File: `outputs.tf`**
```hcl
output "vpc_id" {
  description = "ID of the VPC"
  value       = aws_vpc.main.id
}

output "public_subnet_ids" {
  description = "IDs of public subnets"
  value       = aws_subnet.public[*].id
}

output "private_subnet_ids" {
  description = "IDs of private subnets"
  value       = aws_subnet.private[*].id
}
```

### What To Observe

Run these commands and observe:

```bash
# 1. Initialize — watch providers download
terraform init

# 2. Plan — read every line. This is your contract.
terraform plan

# 3. Apply — watch resources create in dependency order
terraform apply

# 4. Plan again — this MUST say "No changes"
terraform plan
# Output: No changes. Your infrastructure matches the configuration.

# 5. This proves idempotency. Same code = same result = no changes.
```

## 6. Real-World Mapping

### In AWS

| Terraform Concept | AWS Equivalent |
|-------------------|---------------|
| `terraform plan` | CloudFormation Change Set (but better — readable diff) |
| `terraform apply` | CloudFormation deploy / AWS CLI create commands |
| State file | CloudFormation stack state (managed by AWS, not your problem) |
| Refresh | CloudFormation drift detection |
| Provider | AWS SDK under the hood |

### In CI/CD Pipelines

```
Developer writes .tf code
        ↓
Opens Pull Request
        ↓
CI runs: terraform init → terraform plan
        ↓
Plan output posted as PR comment (the review artifact)
        ↓
Team reviews the plan (not the code — the PLAN)
        ↓
PR merged to main branch
        ↓
CD runs: terraform apply -auto-approve
        ↓
State updated, infrastructure changes live
```

### In Multi-Team Environments

- Each team owns a set of Terraform configurations
- Teams share modules but have **separate state files**
- The plan output is the **change review document** — you review plans, not just code
- State locking prevents two team members from applying simultaneously

## 7. Failure Scenarios — Production Focus

### Scenario 1: Partial Apply Failure

| | Details |
|--|---------|
| **Symptom** | `terraform apply` fails at resource 5 of 10. Error: "Rate exceeded" or "ConflictException" |
| **Root Cause** | AWS API throttling or a resource dependency not yet ready |
| **Terraform Behavior** | Resources 1-4 are created and in state. Resource 5 failed. Resources 6-10 that depend on 5 are skipped. |
| **State** | Partially updated — reflects only resources 1-4 |
| **How to Debug** | 1. Read the error message carefully. 2. Run `terraform state list` to see what was created. 3. Run `terraform plan` to see what's still needed. 4. Fix the root cause (increase API limits, fix config). 5. Run `terraform apply` again — it only creates the missing resources. |

### Scenario 2: State File Out of Sync

| | Details |
|--|---------|
| **Symptom** | `terraform plan` shows it wants to create a VPC — but you know the VPC exists |
| **Root Cause** | Someone deleted the state file, or the resource was created outside Terraform |
| **Terraform Behavior** | Without state, Terraform doesn't know the VPC exists. It plans to create a new one. |
| **How to Debug** | 1. `terraform state list` — is the resource there? 2. If not: `terraform import aws_vpc.main vpc-0abc123` to bring it into state. 3. Run `terraform plan` — should show no changes if your code matches real config. |

### Scenario 3: "State Lock" Error

| | Details |
|--|---------|
| **Symptom** | `Error: Error acquiring the state lock` |
| **Root Cause** | A previous apply crashed or was killed, leaving a stale lock |
| **Terraform Behavior** | Refuses to proceed (safety mechanism) |
| **How to Debug** | 1. Verify no one else is running an apply. 2. Check if a CI job is still running. 3. If confirmed stale: `terraform force-unlock <LOCK_ID>`. 4. **Never** force-unlock if there's any chance an apply is in progress. |

### Scenario 4: "Resource Already Exists" Error

| | Details |
|--|---------|
| **Symptom** | `Error: creating S3 Bucket: BucketAlreadyExists` |
| **Root Cause** | Resource exists in AWS but not in Terraform state (manual creation or state loss) |
| **Terraform Behavior** | Tries to create, fails because it already exists |
| **How to Debug** | 1. `terraform import aws_s3_bucket.main my-bucket-name` 2. Run `terraform plan` — adjust code until "No changes." |

## 8. Debugging Playbook

### When Something Breaks — Step-by-Step

```
STEP 1: READ THE ERROR
  └── Terraform errors are descriptive. Read the FULL message.
  └── Look for: resource address, API error code, attribute name

STEP 2: CHECK STATE
  └── terraform state list          # What does Terraform know about?
  └── terraform state show <resource>  # What attributes does it have?

STEP 3: COMPARE STATE VS REAL WORLD
  └── Open AWS console → find the resource
  └── Compare attributes with terraform state show output
  └── Differences = drift

STEP 4: ENABLE VERBOSE LOGGING
  └── TF_LOG=DEBUG terraform plan 2> debug.log
  └── Search for: HTTP request/response, error codes, retry attempts
  └── TF_LOG levels: TRACE > DEBUG > INFO > WARN > ERROR

STEP 5: RUN A TARGETED PLAN
  └── terraform plan -target=aws_vpc.main
  └── Isolate the problem resource
  └── WARNING: -target is for debugging only, never for regular use

STEP 6: CHECK PROVIDER VERSION
  └── terraform version
  └── Is your provider version compatible with the AWS feature?
  └── Try: terraform init -upgrade
```

### Common Commands for Debugging

```bash
# View state
terraform state list
terraform state show aws_vpc.main

# Verbose logging
TF_LOG=DEBUG terraform plan

# Target specific resource
terraform plan -target=aws_vpc.main

# Validate syntax
terraform validate

# Format check
terraform fmt -check -recursive

# Show dependency graph
terraform graph | dot -Tpng > graph.png
```

## 10. Interview Preparation

### Q1: *"What happens when you run `terraform apply`?"* (Screening)

**Strong Answer:**
"Terraform first refreshes its state file by querying the cloud provider APIs to see the current real-world state. Then it compares the desired state in the .tf files against the refreshed state. The diff becomes the execution plan. Apply then executes that plan — making API calls to create, update, or destroy resources — and updates the state file after each successful operation. If it fails midway, the state reflects the partial reality, and the next plan will only show the remaining changes."

### Q2: *"Is it safe to run `terraform apply` twice?"* (Screening)

**Strong Answer:**
"Yes, due to idempotency. If nothing changed in the code or the real world, the second apply produces no changes. This is a fundamental property of declarative systems — you describe the desired end state, and Terraform converges toward it. However, if someone manually changed infrastructure between the two applies, the second apply will correct that drift."

### Q3: *"Your `terraform apply` failed halfway. What do you do?"* (Senior)

**Strong Answer:**
"First, I don't panic and re-run apply blindly. I read the error to understand what failed and why. Then I run `terraform state list` to see what was successfully created. I run `terraform plan` to see what Terraform thinks needs to happen next. I fix the root cause of the failure, then run apply again — Terraform will only attempt the remaining changes because successfully created resources are already in state."

### Q4: *"What is Terraform state and why can't Terraform work without it?"* (Screening)

**Strong Answer:**
"State is Terraform's memory — it maps each resource block in code to a real-world resource ID. Without state, Terraform has no way to know that `aws_vpc.main` in your code corresponds to `vpc-0abc123` in AWS. It would try to create a new VPC every time. State enables the diff calculation: desired minus known equals the plan."

### Q5: *"How is Terraform different from a shell script that calls AWS CLI?"* (Mid)

**Strong Answer:**
"Three fundamental differences: First, Terraform is declarative — you describe what should exist, not the steps to create it. Second, idempotency — running the same Terraform code twice is safe; running the same shell script twice creates duplicates. Third, state management — Terraform tracks what it created and can detect drift, update, and destroy. A shell script has no memory of what it did."

### Q6: *"What does the plan show that the code doesn't?"* (Mid)

**Strong Answer:**
"Plan shows the actual changes that will be made — creates, updates, and destroys. Code shows the desired state, but plan shows the delta between desired and current reality. A small code change might trigger a resource destroy-and-recreate that isn't obvious from reading the code. That's why you review plans, not just code, in production."

### Q7: *"Can you explain the `refresh` phase? When would you skip it?"* (Senior)

**Strong Answer:**
"Refresh queries every resource in state against the cloud provider API to update the in-memory state with real-world values. This is how drift is detected. You'd skip it with `-refresh=false` in very large environments where refreshing hundreds of resources is slow, and you're confident no manual changes were made. But skipping refresh risks working with stale state — only do it when you need speed and accept the risk."

## 11. Advanced Insights

### Trade-Offs of the Execution Model

| Benefit | Trade-Off |
|---------|-----------|
| Plan shows exactly what will change | Large environments = slow refresh (API call per resource) |
| Idempotency makes re-runs safe | Partial failures leave state in intermediate state |
| Dependency graph enables parallelism | Circular dependencies are impossible (sometimes frustrating) |
| State tracks everything | State becomes a critical file that must be secured and backed up |

### When NOT to Use Terraform

- **One-off scripts**: If you need to run something once and never again, a CLI command is simpler
- **Highly dynamic resources**: Auto-scaling groups dynamically add/remove instances — Terraform fights this unless you use `ignore_changes`
- **Rapidly iterating applications**: Deploying application code (Docker images, Lambda code) through Terraform is slow — use proper CI/CD tools for app deployment, Terraform for infrastructure
- **Secrets management**: Terraform stores everything in state. Use AWS Secrets Manager + external tools for secrets rotation

### Scaling Challenges

- **Refresh time**: 500 resources × 1 API call each = slow plan. Split into multiple smaller state files.
- **State file size**: Large state files are slow to read/write. Keep state files under 50MB by splitting infrastructure into components.
- **Blast radius**: One state file = all resources are planned/applied together. A typo in a tag affects everything in the plan. Split by blast radius (networking, compute, database = separate state files).

## 12. Common Mistakes

### Beginner Mistakes

| Mistake | Why It's Wrong | Fix |
|---------|---------------|-----|
| Treating Terraform like a script | Running apply without reading plan | Always run `plan` first. Read every line. |
| Editing `.tfstate` manually | Corrupts state, loses resource tracking | Use `terraform state` commands |
| Not running `terraform init` after changes | Provider/module not downloaded | Run `init` after any provider or backend change |
| Ignoring plan output | Apply destroys a database you didn't expect | The plan is a contract. Read it. |

### Intermediate Mistakes

| Mistake | Why It's Wrong | Fix |
|---------|---------------|-----|
| Using `-auto-approve` in production | Skips the safety net of plan review | Only use in CI after plan has been reviewed |
| Running `terraform destroy` without `-plan` first | Destroys everything immediately | Run `terraform plan -destroy` first |
| Using `-target` regularly | Creates state drift and incomplete applies | `-target` is for debugging, not workflow |

### Production Mistakes

| Mistake | Why It's Wrong | Fix |
|---------|---------------|-----|
| Running apply from laptops | No audit trail, no review, inconsistent environments | Use CI/CD pipeline exclusively |
| Not backing up state | Losing state = losing track of all infrastructure | Remote state with versioning |
| Ignoring partial apply failures | Next apply may create duplicates or conflicts | Always investigate and resolve before re-applying |

## 13. Cheat Sheet

### Essential Commands

```bash
terraform init              # Download providers, initialize backend
terraform plan              # Preview changes (ALWAYS do this first)
terraform apply             # Execute the plan
terraform destroy           # Destroy all managed resources
terraform validate          # Check syntax without API calls
terraform fmt               # Auto-format .tf files
terraform fmt -check        # Check formatting (CI use)
terraform version           # Show Terraform + provider versions
terraform graph             # Output dependency graph in DOT format
```

### Key Rules

1. **Terraform is declarative** — describe WHAT, not HOW
2. **Plan before apply** — always, no exceptions
3. **Read the plan** — the plan is the contract
4. **Idempotent** — same code + same state = no changes
5. **State is sacred** — never edit manually, always back up
6. **Refresh happens automatically** — during plan and apply
7. **Dependency graph** — Terraform parallelizes independent resources
8. **Partial failures** — state reflects partial reality; re-running is safe

### The Mental Model in One Sentence

> **Terraform reads your code (desired), checks reality (actual), diffs against memory (state), and makes reality match your code.**
