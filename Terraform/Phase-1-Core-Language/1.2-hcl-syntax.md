## 1. First Principles — Start Simple

### What Problem Does This Solve?

You now understand that Terraform compares desired state to actual state and produces a plan. But **how do you express the desired state?**

You need a language that can:
- **Declare** infrastructure resources (VPCs, EC2 instances, databases)
- **Parameterize** configurations so the same code works for dev and prod
- **Reuse** computed values to avoid repetition
- **Expose** important information after deployment (IPs, DNS names, IDs)

HCL (HashiCorp Configuration Language) gives you exactly four constructs to do this:

| Construct | Purpose | Analogy |
|-----------|---------|---------|
| **Resource** | "I want this thing to exist" | A blueprint item (e.g., "build a wall here") |
| **Variable** | "The caller decides this value" | A form field (e.g., "enter wall color") |
| **Local** | "A computed value I reuse internally" | A sticky note with a calculation |
| **Output** | "After building, report this value" | The delivery receipt |

### Real-World Analogy

Think of ordering a custom laptop online:

| Laptop Order | Terraform HCL |
|-------------|---------------|
| The laptop itself (CPU, RAM, storage) | **Resource** — `aws_instance` with instance_type, AMI |
| Configuration form (choose color, storage size) | **Variable** — caller chooses values |
| Calculated price shown on the form | **Local** — derived from selections |
| Order confirmation with serial number | **Output** — ID/ARN exposed after creation |

### Why Terraform Needs This Concept

Without these four constructs:
- **No resources** → Terraform has nothing to create
- **No variables** → Everything is hardcoded (useless for multiple environments)
- **No locals** → Repeated expressions everywhere (DRY violation)
- **No outputs** → No way to pass values between configurations or see results

## 2. Terraform Mental Model

### How HCL Fits Into the Framework

```
                         ┌──────────────────┐
                         │  variables.tf    │
                         │  (INPUTS)        │
                         │  vpc_cidr = ?    │
                         │  environment = ? │
                         └────────┬─────────┘
                                  │
                                  ▼
┌──────────────┐    ┌──────────────────────┐    ┌──────────────┐
│  locals.tf   │───▶│     main.tf          │───▶│  outputs.tf  │
│ (COMPUTED)   │    │   (RESOURCES)        │    │  (EXPOSED)   │
│ common_tags  │    │   aws_vpc.main       │    │  vpc_id      │
│ name_prefix  │    │   aws_subnet.public  │    │  subnet_ids  │
└──────────────┘    └──────────────────────┘    └──────────────┘
                              │
                              ▼
                    ┌────────────────────┐
                    │  Dependency Graph  │
                    │  VPC → Subnet      │
                    │  (automatic)       │
                    └────────────────────┘
```

### Desired State Expression

Each resource block **is** a declaration of desired state:

```hcl
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}
```

This says: *"I want a VPC with this CIDR to exist."* Terraform then:
1. Checks state — does `aws_vpc.main` already exist?
2. If no → plan to create
3. If yes but different → plan to update
4. If yes and same → no change

### Idempotency Through Declaration

You never write "create a VPC." You write "a VPC should exist with these properties." This is why:
- Running apply twice is safe (resource already exists = no change)
- Deleting a resource block from code triggers destroy (desired state no longer includes it)
- Changing an attribute triggers update (desired state differs from known state)

## 3. Core Concept — Deep Dive

### Construct 1: Resources — What You Want to Exist

**Syntax:**
```hcl
resource "<provider>_<type>" "<local_name>" {
  argument1 = value1
  argument2 = value2

  nested_block {
    nested_argument = value
  }
}
```

**Anatomy of a resource:**

```hcl
resource "aws_vpc" "main" {
#        ▲           ▲
#        │           └── Local name (your reference)
#        └── Resource type (provider_resource)
#
# Together: aws_vpc.main is the resource ADDRESS
# Used for references: aws_vpc.main.id, aws_vpc.main.cidr_block

  cidr_block           = "10.0.0.0/16"    # Required argument
  enable_dns_support   = true               # Optional argument
  enable_dns_hostnames = true               # Optional argument

  tags = {                                  # Map argument
    Name        = "production-vpc"
    Environment = "prod"
  }
}
```

**Key rules:**
- `resource "type" "name"` — the type+name combination must be **unique per configuration**
- Resource type determines which provider API to call (`aws_` → AWS, `azurerm_` → Azure)
- Arguments are **declared, not executed** — order within a resource doesn't matter
- Terraform docs list which arguments are **required** vs **optional** for each resource

**Referencing resources:**
```hcl
# Reference format: <resource_type>.<name>.<attribute>
aws_vpc.main.id                    # The VPC's provider-assigned ID
aws_vpc.main.cidr_block            # "10.0.0.0/16"
aws_vpc.main.default_route_table_id  # Computed by AWS, known after apply
```

### Construct 2: Variables — Parameterize Your Code

**Syntax:**
```hcl
variable "<name>" {
  type        = <type>
  description = "What this variable does"
  default     = <default_value>      # Optional
  sensitive   = true/false           # Optional, default false
  nullable    = true/false           # Optional, default true

  validation {
    condition     = <boolean_expression>
    error_message = "Error shown when condition is false"
  }
}
```

**Common types:**

```hcl
# String
variable "environment" {
  type        = string
  description = "Deployment environment"
  default     = "dev"
}

# Number
variable "instance_count" {
  type        = number
  description = "Number of instances to create"
  default     = 2
}

# Boolean
variable "enable_monitoring" {
  type        = bool
  description = "Enable detailed monitoring"
  default     = true
}

# List (ordered collection of same type)
variable "availability_zones" {
  type        = list(string)
  description = "AZs to deploy into"
  default     = ["us-east-1a", "us-east-1b"]
}

# Map (key-value pairs of same type)
variable "instance_types" {
  type = map(string)
  description = "Instance type per environment"
  default = {
    dev     = "t3.micro"
    staging = "t3.small"
    prod    = "t3.medium"
  }
}

# Object (structured type with named attributes)
variable "vpc_config" {
  type = object({
    cidr            = string
    public_subnets  = list(string)
    private_subnets = list(string)
    enable_nat      = bool
  })
  description = "VPC configuration object"
}
```

**How to set variable values (precedence order, highest wins):**

```
1. -var 'name=value'       CLI flag (highest precedence)
2. -var-file=custom.tfvars  Explicit var file
3. *.auto.tfvars            Auto-loaded var files
4. terraform.tfvars         Auto-loaded var file
5. TF_VAR_name=value        Environment variable
6. default = value          Default in variable block (lowest precedence)
7. Interactive prompt       If no value found and no default
```

**Variable validation (production essential):**

```hcl
variable "environment" {
  type        = string
  description = "Must be dev, staging, or prod"

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}

variable "vpc_cidr" {
  type        = string
  description = "VPC CIDR block"

  validation {
    condition     = can(cidrhost(var.vpc_cidr, 0))
    error_message = "Must be a valid CIDR block."
  }
}

variable "instance_type" {
  type        = string
  description = "EC2 instance type"

  validation {
    condition     = can(regex("^t[23]\\.", var.instance_type))
    error_message = "Only t2 and t3 instance families are allowed."
  }
}
```

### Construct 3: Locals — Computed Internal Values

**Syntax:**
```hcl
locals {
  name_prefix = "${var.project}-${var.environment}"

  common_tags = {
    Project     = var.project
    Environment = var.environment
    ManagedBy   = "terraform"
    Owner       = var.team_name
  }

  # Conditional logic
  is_production = var.environment == "prod"

  # Derived computations
  subnet_count = length(var.availability_zones)
}
```

**When to use locals vs variables:**

| Use Case | Variable | Local |
|----------|----------|-------|
| Value comes from outside (caller) | ✅ | ❌ |
| Value is computed from other values | ❌ | ✅ |
| Value is used multiple times internally | ❌ | ✅ |
| Value should be configurable per environment | ✅ | ❌ |
| Value is a constant internal to this module | ❌ | ✅ |

**Anti-pattern:** Don't use locals as an indirection for variables. If the value is just `var.x`, reference `var.x` directly.

### Construct 4: Outputs — Expose Values After Apply

**Syntax:**
```hcl
output "<name>" {
  description = "What this output exposes"
  value       = <expression>
  sensitive   = true/false  # Optional
}
```

**Essential outputs:**

```hcl
output "vpc_id" {
  description = "ID of the VPC"
  value       = aws_vpc.main.id
}

output "public_subnet_ids" {
  description = "List of public subnet IDs"
  value       = aws_subnet.public[*].id
}

output "private_subnet_ids" {
  description = "List of private subnet IDs"
  value       = aws_subnet.private[*].id
}

# Sensitive outputs — redacted from CLI output
output "db_password" {
  description = "Database password"
  value       = random_password.db.result
  sensitive   = true
}
```

**Three uses of outputs:**
1. **Human consumption**: After apply, see critical values (IPs, endpoints)
2. **Module interface**: Parent modules access child module values via `module.vpc.vpc_id`
3. **Remote state**: Other Terraform configurations can read outputs via `terraform_remote_state`

### Bonus Construct: Data Sources — Read Existing Infrastructure

Data sources are **read-only** queries. They don't create anything — they fetch information.

```hcl
# Fetch the latest Amazon Linux 2 AMI
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["amzn2-ami-hvm-*-x86_64-gp2"]
  }
}

# Fetch available AZs in the current region
data "aws_availability_zones" "available" {
  state = "available"
}

# Fetch the current AWS account ID
data "aws_caller_identity" "current" {}

# Usage:
resource "aws_instance" "web" {
  ami               = data.aws_ami.amazon_linux.id
  instance_type     = "t3.micro"
  availability_zone = data.aws_availability_zones.available.names[0]
}
```

**Data source vs resource:**
| | Resource | Data Source |
|---|---------|------------|
| Creates infrastructure? | ✅ Yes | ❌ No (read-only) |
| Appears in state? | ✅ Yes (with full attributes) | ✅ Yes (but read-only snapshot) |
| Can be destroyed? | ✅ Yes (`terraform destroy`) | ❌ No (nothing to destroy) |
| When to use | You want Terraform to manage it | It already exists or is managed elsewhere |

## 4. Under the Hood — Critical

### How Terraform Processes HCL Files

```
Step 1: FILE DISCOVERY
  └── Terraform loads ALL .tf files in the current directory
  └── File names don't matter — Terraform treats them as ONE config
  └── Convention: main.tf, variables.tf, outputs.tf, providers.tf

Step 2: PARSE
  └── HCL parser converts text to Abstract Syntax Tree (AST)
  └── Validates syntax (matching braces, valid types)

Step 3: GRAPH CONSTRUCTION
  └── Build dependency graph from references:
      aws_subnet uses aws_vpc.main.id → VPC must exist first
  └── Check for circular dependencies → error if found

Step 4: VARIABLE RESOLUTION
  └── Load variable values in precedence order
  └── Run validations
  └── Substitute values into expressions

Step 5: PLAN EXECUTION
  └── Walk the graph in dependency order
  └── For each resource: compare desired vs known state
  └── Generate the plan diff
```

### State Interaction for Each Construct

| Construct | Stored in State? | Details |
|-----------|-----------------|---------|
| Resource | ✅ Yes | Full resource attributes, provider ID, metadata |
| Variable | ❌ No | Values are evaluated at plan time, not persisted |
| Local | ❌ No | Computed at plan time, not persisted |
| Output | ✅ Yes | Values stored in state (used by remote_state data) |
| Data Source | ✅ Yes | Read values cached in state for the plan |

### Provider Interaction

When Terraform encounters `resource "aws_vpc" "main"`:

```
1. Looks up "aws_vpc" in the AWS provider schema
2. Provider tells Terraform:
   - Required arguments: cidr_block
   - Optional arguments: enable_dns_support, tags, ...
   - Computed attributes: id, arn, default_route_table_id, ...
3. During apply, provider:
   - Translates to: CreateVpc API call
   - Passes cidr_block, tags, etc.
   - Returns: VPC ID, ARN, all computed attributes
4. Terraform stores ALL returned attributes in state
```

### Expression Evaluation

```hcl
# String interpolation
name = "${var.environment}-vpc"
# Evaluated: "dev-vpc" (if var.environment = "dev")

# Conditional expression
instance_type = var.environment == "prod" ? "t3.large" : "t3.micro"
# Evaluated: "t3.micro" (if var.environment = "dev")

# Function call
cidr_block = cidrsubnet(var.vpc_cidr, 8, 0)
# Evaluated: "10.0.0.0/24" (if var.vpc_cidr = "10.0.0.0/16")

# Splat expression (shorthand for collecting attributes)
subnet_ids = aws_subnet.public[*].id
# Equivalent to: [for s in aws_subnet.public : s.id]
```

  common_tags = {

**File: `main.tf`** (enhanced)
```hcl
# ---------------------------------------------------------
# Data Sources
# ---------------------------------------------------------
data "aws_availability_zones" "available" {
  state = "available"
}

data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["amzn2-ami-hvm-*-x86_64-gp2"]
  }
}

# ---------------------------------------------------------
# VPC
# ---------------------------------------------------------
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_support   = true
  enable_dns_hostnames = true

  tags = {
    Name = "${local.name_prefix}-vpc"
  }
}

# ---------------------------------------------------------
# Subnets
# ---------------------------------------------------------
resource "aws_subnet" "public" {
  count                   = local.az_count
  vpc_id                  = aws_vpc.main.id
  cidr_block              = cidrsubnet(var.vpc_cidr, 8, count.index)
  availability_zone       = data.aws_availability_zones.available.names[count.index]
  map_public_ip_on_launch = true

  tags = {
    Name = "${local.name_prefix}-public-${count.index + 1}"
    Tier = "public"
  }
}

resource "aws_subnet" "private" {
  count             = local.az_count
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(var.vpc_cidr, 8, count.index + 10)
  availability_zone = data.aws_availability_zones.available.names[count.index]

  tags = {
    Name = "${local.name_prefix}-private-${count.index + 1}"
    Tier = "private"
  }
}

# ---------------------------------------------------------
# Internet Gateway + Route Table
# ---------------------------------------------------------
resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id

  tags = {
    Name = "${local.name_prefix}-igw"
  }
}

resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.main.id
  }

  tags = {
    Name = "${local.name_prefix}-public-rt"
  }
}

resource "aws_route_table_association" "public" {
  count          = local.az_count
  subnet_id      = aws_subnet.public[count.index].id
  route_table_id = aws_route_table.public.id
}

# ---------------------------------------------------------
# Security Group for EC2
# ---------------------------------------------------------
resource "aws_security_group" "web" {
  name        = "${local.name_prefix}-web-sg"
  description = "Security group for web server"
  vpc_id      = aws_vpc.main.id

  # HTTP from anywhere
  ingress {
    description = "HTTP"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  # SSH from allowed CIDRs only
  dynamic "ingress" {
    for_each = length(var.allowed_ssh_cidrs) > 0 ? [1] : []
    content {
      description = "SSH"
      from_port   = 22
      to_port     = 22
      protocol    = "tcp"
      cidr_blocks = var.allowed_ssh_cidrs
    }
  }

  # All outbound
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name = "${local.name_prefix}-web-sg"
  }
}

# ---------------------------------------------------------
# EC2 Instance
# ---------------------------------------------------------
resource "aws_instance" "web" {
  ami                    = data.aws_ami.amazon_linux.id
  instance_type          = var.instance_type
  subnet_id              = aws_subnet.public[0].id
  vpc_security_group_ids = [aws_security_group.web.id]

  tags = {
    Name = "${local.name_prefix}-web"
  }
}
```

## 6. Real-World Mapping

### In AWS

| HCL Construct | How AWS Uses It |
|---------------|----------------|
| Resource `aws_vpc` | Creates a VPC via `CreateVpc` API |
| Variable `vpc_cidr` | Parameterizes the CIDR — same code, different networks per env |
| Local `common_tags` | Ensures every resource has cost allocation + compliance tags |
| Output `vpc_id` | Consumed by EKS/RDS modules that need the VPC ID |
| Data `aws_ami` | Queries EC2 `DescribeImages` to find latest AMI — no hardcoding |
| Data `aws_availability_zones` | Queries AZs — code works in any region |

### In CI/CD Pipelines

```yaml
# terraform.tfvars files per environment
# environments/dev.tfvars
environment     = "dev"
vpc_cidr        = "10.0.0.0/16"
instance_type   = "t3.micro"
allowed_ssh_cidrs = ["10.0.0.0/8"]

# environments/prod.tfvars
environment     = "prod"
vpc_cidr        = "10.1.0.0/16"
instance_type   = "t3.small"
allowed_ssh_cidrs = []   # No SSH in prod
```

```bash
# CI pipeline uses -var-file for each environment
terraform plan -var-file=environments/dev.tfvars
terraform plan -var-file=environments/prod.tfvars
```

### In Multi-Team Environments

- **Variables** are the module's API — teams consume your module by setting variables
- **Outputs** are what other teams' modules can reference
- **Locals** are internal — teams shouldn't need to know about them
- **Naming conventions** (`local.name_prefix`) prevent resource name collisions between teams

## 7. Failure Scenarios — Production Focus

### Scenario 1: Missing Required Variable

| | Details |
|--|---------|
| **Symptom** | `Error: No value for required variable` during plan |
| **Root Cause** | Variable has no default and no value was provided |
| **Terraform Behavior** | Interactive prompt (CLI) or immediate error (CI/CD) |
| **How to Fix** | Add to `terraform.tfvars`, set via `TF_VAR_name`, or add a default |

### Scenario 2: Variable Validation Failure

| | Details |
|--|---------|
| **Symptom** | `Error: Invalid value for variable` with your custom message |
| **Root Cause** | Value doesn't pass the `validation` block's condition |
| **Terraform Behavior** | Fails during plan — no API calls made, nothing changed |
| **How to Fix** | Correct the value. This is **by design** — validation prevents bad deployments |
| **Why It Matters** | Catching "staging" typo as "stagging" here prevents deploying to wrong environment |

### Scenario 3: Incorrect Resource Reference

| | Details |
|--|---------|
| **Symptom** | `Error: Reference to undeclared resource` |
| **Root Cause** | Typo in resource address (e.g., `aws_vpc.Main` instead of `aws_vpc.main`) |
| **Terraform Behavior** | Fails during parse — before plan even starts |
| **How to Fix** | Fix the reference. Remember: resource names are case-sensitive |

### Scenario 4: Data Source Returns No Results

| | Details |
|--|---------|
| **Symptom** | `Error: no AMI matching the filters found` |
| **Root Cause** | Data source filters too restrictive, or wrong owner, or wrong region |
| **Terraform Behavior** | Fails during the refresh phase of plan |
| **How to Fix** | Check filters in AWS console. Verify region. Verify owner account ID. |

### Scenario 5: Hardcoded Values Break in New Region

| | Details |
|--|---------|
| **Symptom** | `Error: invalid availability zone us-east-1a` (when deploying to us-west-2) |
| **Root Cause** | Hardcoded AZ names instead of using `data.aws_availability_zones` |
| **Terraform Behavior** | Fails during apply — API rejects the AZ |
| **How to Fix** | Replace hardcoded AZs with data source. See project code above. |

## 8. Debugging Playbook

### When HCL code isn't working — Step-by-Step

```
STEP 1: VALIDATE SYNTAX
  └── terraform validate
  └── Catches: missing braces, invalid attribute names, type mismatches
  └── No API calls — instant feedback

STEP 2: CHECK FORMATTING
  └── terraform fmt -diff
  └── Shows what would change. Misformatted code can mask logic errors.

STEP 3: CHECK VARIABLE VALUES
  └── terraform console
  └── > var.environment          → "dev"
  └── > var.vpc_cidr             → "10.0.0.0/16"
  └── > local.name_prefix        → "terraform-learning-dev"
  └── > cidrsubnet(var.vpc_cidr, 8, 0)  → "10.0.0.0/24"

STEP 4: INSPECT PLAN OUTPUT
  └── terraform plan
  └── Look for unexpected changes (especially destroy+recreate)
  └── Verify computed values match expectations

STEP 5: CHECK PROVIDER DOCS
  └── registry.terraform.io → search for resource type
  └── Verify: required vs optional arguments
  └── Verify: which attributes are computed
```

## 10. Interview Preparation

### Q1: *"What are the four core HCL constructs and when do you use each?"* (Screening)

**Strong Answer:**
"Resources declare what infrastructure should exist — they're the core of Terraform. Variables parameterize the configuration, making it reusable across environments. Locals are computed values used internally to avoid repetition — the DRY principle for Terraform. Outputs expose values after apply — used by humans, CI/CD pipelines, and other Terraform configurations that need to reference this infrastructure."

### Q2: *"What's the difference between a variable and a local?"* (Screening)

**Strong Answer:**
"A variable is an input — its value comes from outside the configuration (tfvars files, CLI flags, environment variables). A local is a computed value — it's derived from variables, other locals, or expressions, and cannot be set from outside. Think of variables as function parameters and locals as internal calculations."

### Q3: *"Why should you never hardcode an AMI ID in a resource?"* (Mid)

**Strong Answer:**
"AMI IDs are region-specific — `ami-0abc` in us-east-1 doesn't exist in us-west-2. They also go stale as new versions are published. Instead, use a `data 'aws_ami'` data source with filters to always fetch the latest AMI dynamically. This makes your code portable across regions and always up-to-date."

### Q4: *"How do you prevent someone from deploying to production with a dev variable file?"* (Senior)

**Strong Answer:**
"Multiple layers: First, variable validation that enforces allowed values. Second, separate state files per environment — even if you use the wrong tfvars, you're applying to the wrong state file and Terraform will show unexpected changes in the plan. Third, CI/CD pipeline that enforces which variable file maps to which environment and which branch. Fourth, separate AWS accounts per environment — even if everything else fails, IAM permissions prevent cross-environment changes."

### Q5: *"Walk me through how you'd design the variables.tf for a VPC module used by 10 teams."* (Senior)

**Strong Answer:**
"I'd start with required variables for things that MUST differ per team: vpc_cidr, environment, and team_name. I'd add optional variables with sensible defaults for things teams might customize: enable_nat_gateway (default true), single_nat_gateway (default false for prod HA), subnet count. Every variable gets a description, a type constraint, and validation where applicable. The module should work with just the required variables — everything else has production-safe defaults."

### Q6: *"Data source vs resource — when do you use each?"* (Mid)

**Strong Answer:**
"A resource tells Terraform 'I want you to manage this.' A data source tells Terraform 'I want you to read this — it exists elsewhere and I don't manage it.' Example: I create an RDS instance (resource) in a VPC that was created by another team's Terraform config. I use a data source to look up their VPC ID because my config doesn't own it."

### Q7: *"What happens to outputs in the state file?"* (Trap)

**Strong Answer:**
"Output values are stored in the state file — including sensitive ones. This is important because `terraform_remote_state` data source reads outputs from another config's state file. If you mark an output as sensitive, Terraform redacts it from CLI output, but it's still plaintext in state. This is why state encryption and access control are critical."

## 11. Advanced Insights

### Trade-Offs

| Decision | Benefit | Risk |
|----------|---------|------|
| Variables with defaults | Easy to use, less config per env | Wrong default in prod = deployed mistake |
| Many small variables | Fine-grained control | API is complex, caller must set 20 variables |
| One object variable | Clean API, one variable groups related config | Changes to the object type break all callers |
| Data sources for everything | No hardcoding, fully dynamic | More API calls, slower plan, possible flakiness |
| Strict validation | Prevents bad deployments | Over-validation frustrates callers |

### When NOT to Use Each Construct

- **Don't use a variable** when the value should never change (use a local)
- **Don't use a local** when the caller should control the value (use a variable)
- **Don't use a data source** when you control the resource (use a resource and reference its attributes)
- **Don't use an output** for internal-only values (outputs pollute the state file and the interface)

### Scaling Considerations

- **100+ variables** = your module is too big. Split into smaller modules.
- **Variables without descriptions** = impossible for other teams to use your module
- **Variables without validation** = guaranteed misuse in production

## 12. Common Mistakes

### Beginner Mistakes

| Mistake | Why It's Wrong | Fix |
|---------|---------------|-----|
| Hardcoding values in resources | Breaks in different environments/regions | Use variables and data sources |
| No variable descriptions | Team members can't understand the interface | Always add descriptions |
| Using `string` for everything | Loses type safety | Use proper types: `number`, `bool`, `list`, `map`, `object` |
| Putting everything in `main.tf` | Unreadable at scale | Split: `main.tf`, `variables.tf`, `outputs.tf`, `locals.tf` |

### Intermediate Mistakes

| Mistake | Why It's Wrong | Fix |
|---------|---------------|-----|
| No variable validation | Bad values reach AWS and cause confusing errors | Add `validation` blocks |
| Locals as variable proxies | `local.x = var.x` adds indirection without value | Reference `var.x` directly |
| Outputs without descriptions | Consumers can't understand the interface | Always add descriptions |
| Hardcoded AMI IDs | Break across regions, go stale | Use `data "aws_ami"` |

### Production Mistakes

| Mistake | Why It's Wrong | Fix |
|---------|---------------|-----|
| Variables with dangerous defaults | `environment = "prod"` as default → accidental prod deploy | Default to `"dev"` or no default |
| Sensitive values in tfvars in git | Secrets exposed in version history forever | Use `TF_VAR_` env vars, never commit secrets |
| Missing outputs | Downstream consumers can't reference your infrastructure | Output all IDs, ARNs, endpoints |
| No `default_tags` in provider | Resources created without mandatory tags | Always use `default_tags` block |

## 13. Cheat Sheet

### Variable Precedence (Highest → Lowest)

```
1. -var 'name=value'          (CLI flag)
2. -var-file=file.tfvars      (explicit file)
3. *.auto.tfvars              (auto-loaded, alphabetical)
4. terraform.tfvars           (auto-loaded)
5. TF_VAR_name                (environment variable)
6. default = value            (in variable block)
7. Interactive prompt         (if no value found)
```

### Type System Quick Reference

```hcl
string    "hello"
number    42
bool      true
list(T)   ["a", "b"]         # Ordered, same type
map(T)    { key = "value" }  # Key-value, same value type
set(T)    toset(["a", "b"])  # Unordered, unique, same type
object    { name = string, age = number }  # Named attributes
tuple     [string, number]   # Fixed-length, mixed types
any       # Accepts anything — use sparingly
```

### Essential Functions

```hcl
# String
format("Hello, %s!", var.name)         → "Hello, world!"
join(", ", ["a", "b", "c"])            → "a, b, c"
replace("hello-world", "-", "_")       → "hello_world"
upper("hello")                         → "HELLO"

# Collection
length(["a", "b", "c"])                → 3
contains(["a", "b"], "a")             → true
merge({a=1}, {b=2})                   → {a=1, b=2}
lookup({dev="t3.micro"}, "dev", "t3.small") → "t3.micro"
keys({a=1, b=2})                      → ["a", "b"]
values({a=1, b=2})                    → [1, 2]

# Network
cidrsubnet("10.0.0.0/16", 8, 0)       → "10.0.0.0/24"
cidrsubnet("10.0.0.0/16", 8, 1)       → "10.0.1.0/24"
cidrhost("10.0.0.0/24", 5)            → "10.0.0.5"

# Type
can(cidrhost(var.vpc_cidr, 0))         → true/false
try(var.optional_value, "default")     → value or "default"
tostring(42)                          → "42"
tolist(toset(["b", "a"]))             → ["a", "b"]

# Filesystem
file("${path.module}/scripts/init.sh") → file contents
templatefile("template.sh", {name=var.name}) → rendered template
```

### File Convention

```
main.tf          # Resources (the infrastructure)
variables.tf     # Input variables (the API)
outputs.tf       # Output values (the interface)
locals.tf        # Computed values (internal)
providers.tf     # Provider configuration
versions.tf      # Version constraints (sometimes combined with providers.tf)
data.tf          # Data sources (optional, can be in main.tf)
terraform.tfvars # Variable values (DO NOT commit if contains secrets)
```
