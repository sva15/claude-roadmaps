## 2. Core Concept — Deep Dive

### Splat Expressions

The `[*]` syntax is a shorthand for collecting an attribute from all instances of a resource.

```hcl
# SPLAT: Get the ID of every public subnet
output "subnet_ids" {
  value = aws_subnet.public[*].id
}
# If 3 subnets exist → ["subnet-aaa", "subnet-bbb", "subnet-ccc"]

# EQUIVALENT for expression:
output "subnet_ids" {
  value = [for s in aws_subnet.public : s.id]
}
```

**When splat works:**
```hcl
# ✅ Works with count-based resources
resource "aws_subnet" "public" {
  count = 3
}
output "ids" { value = aws_subnet.public[*].id }

# ❌ Does NOT work with for_each-based resources
resource "aws_subnet" "public" {
  for_each = var.subnets
}
# aws_subnet.public[*].id → ERROR
# Use: values(aws_subnet.public)[*].id
# Or:  [for s in aws_subnet.public : s.id]
```

**Splat vs for expression:**

| | Splat `[*]` | For expression |
|--|-----------|----------------|
| Works with count | ✅ | ✅ |
| Works with for_each | ❌ (use `values()` wrapper) | ✅ |
| Can filter | ❌ | ✅ (`if` clause) |
| Can transform | ❌ (one attribute only) | ✅ (arbitrary expressions) |
| Readability | Cleaner for simple cases | Better for complex cases |

```hcl
# Splat: simple attribute collection
aws_subnet.public[*].id

# For: filtering + transformation
[for s in aws_subnet.public : s.id if s.availability_zone == "us-east-1a"]
```

### String Templates and Interpolation

```hcl
# INTERPOLATION: embed expressions in strings
name = "vpc-${var.environment}"

# When you DON'T need interpolation (common mistake):
region = var.aws_region              # ✅ Direct reference
region = "${var.aws_region}"         # ❌ Unnecessary wrapping (works, but anti-pattern)

# DIRECTIVE: conditionals and loops in strings
user_data = <<-EOF
  #!/bin/bash
  %{ if var.enable_monitoring }
  yum install -y cloudwatch-agent
  %{ endif }
  echo "ENVIRONMENT=${var.environment}" >> /etc/environment
  %{ for port in var.open_ports }
  iptables -A INPUT -p tcp --dport ${port} -j ACCEPT
  %{ endfor }
EOF
```

**Rule:** If the entire value is a single variable reference, **don't** wrap it in `"${}"`. Only use interpolation when embedding a variable inside a larger string.

### Type System — Variable Type Constraints

```hcl
# PRIMITIVE types
variable "name"   { type = string }     # "hello"
variable "count"  { type = number }     # 42
variable "active" { type = bool }       # true

# COLLECTION types
variable "zones"    { type = list(string) }        # ["us-east-1a", "us-east-1b"]
variable "ports"    { type = list(number) }        # [80, 443]
variable "tags"     { type = map(string) }         # { Name = "web", Tier = "public" }
variable "enabled"  { type = set(string) }         # like list but unique + unordered

# STRUCTURAL types
variable "vpc_config" {
  type = object({
    cidr            = string
    public_subnets  = list(string)
    private_subnets = list(string)
    enable_nat      = bool
  })
}

# TUPLE type (fixed-length, mixed types)
variable "rule" {
  type = tuple([string, number, bool])    # ["tcp", 80, true]
}

# ANY type (accept anything — loses type safety)
variable "settings" {
  type = any                               # Accepts any type
}

# OPTIONAL attributes in objects (Terraform 1.3+)
variable "config" {
  type = object({
    name     = string
    port     = optional(number, 8080)     # Default: 8080 if not provided
    ssl      = optional(bool, false)      # Default: false if not provided
  })
}
```

**Type hierarchy:**
```
any
├── string, number, bool       (primitives)
├── list(type), set(type)      (sequences)
├── map(type)                  (key-value)
├── object({...})              (structured)
└── tuple([...])               (fixed structure)
```

### `depends_on` — When You Actually Need It

Terraform automatically detects dependencies from attribute references. `depends_on` is for **implicit dependencies** that Terraform can't see.

```hcl
# ❌ UNNECESSARY — Terraform already knows this dependency
resource "aws_subnet" "public" {
  vpc_id = aws_vpc.main.id     # ← Terraform sees this reference
  depends_on = [aws_vpc.main]  # ← Redundant! Remove this.
}

# ✅ NECESSARY — IAM side effects
# The IAM policy must be ATTACHED before the instance launches
# But the instance doesn't reference the policy attachment directly
resource "aws_iam_role_policy_attachment" "ssm" {
  role       = aws_iam_role.app.name
  policy_arn = "arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore"
}

resource "aws_instance" "web" {
  ami                  = data.aws_ami.al2023.id
  instance_type        = "t3.micro"
  iam_instance_profile = aws_iam_instance_profile.app.name

  depends_on = [aws_iam_role_policy_attachment.ssm]
  # Without this: instance might launch before IAM permissions propagate
  # The instance doesn't reference the policy attachment → Terraform can't see the dependency
}

# ✅ NECESSARY — Module-level dependencies
module "eks" {
  source = "./modules/eks"
  # ...
  depends_on = [module.vpc]
  # When the entire module depends on another module's side effects
}
```

**The three legitimate `depends_on` scenarios:**
1. **IAM propagation** — policy must exist before the resource that needs it
2. **Network side effects** — NAT gateway must be routable before instances in private subnets
3. **Module boundaries** — when module outputs don't capture the real dependency

**Anti-pattern:** Using `depends_on` everywhere "just to be safe" → slows down plans (forces serial execution instead of parallel).

## 10. Interview Preparation

### Q1: *"What's a splat expression and when would you use it?"* (Mid)

**Strong Answer:**
"Splat `[*]` collects a single attribute from all instances of a count-based resource. `aws_subnet.public[*].id` returns a list of all subnet IDs. It's syntactic sugar for `[for s in aws_subnet.public : s.id]`. I use splat for simple attribute collection, but `for` expressions when I need filtering or transformation. Note: splat doesn't work directly with `for_each` resources — you need `values()` first or use a `for` expression."

### Q2: *"When do you need `depends_on`?"* (Senior — trap question)

**Strong Answer:**
"Almost never. Terraform detects dependencies automatically from attribute references. `depends_on` is only needed for implicit dependencies that Terraform can't infer — like when an IAM policy attachment must complete before an instance launches, even though the instance doesn't reference the attachment. The three real scenarios: IAM propagation delays, network side-effect ordering, and module-level dependencies. Overusing `depends_on` slows plans by forcing serial resource creation."

### Q3: *"What's the difference between `list(string)` and `list(any)`?"* (Mid)

**Strong Answer:**
"`list(string)` enforces that every element is a string — Terraform errors if you pass a number. `list(any)` accepts any type but all elements must still be the same type — Terraform infers the type from the first element. `any` is useful for generic modules where the type is determined by the caller. In production, I prefer explicit types for documentation and early error detection."

## 13. Cheat Sheet

```hcl
# Splat
resource.name[*].attribute              # All instances, one attribute
values(resource.name)[*].attribute      # for_each wrapper

# For (filtering + transforming)
[for item in collection : item.id]                          # Map to list
[for item in collection : item.id if item.tier == "public"] # Filter
{ for item in list : item.key => item.value }               # List → Map

# Interpolation (only when embedding in string)
"prefix-${var.name}-suffix"   # ✅ Needed
var.name                      # ✅ Direct (no interpolation needed)

# Type constraints
type = string | number | bool
type = list(string) | map(string) | set(string)
type = object({ key = type, opt = optional(type, default) })

# depends_on (ONLY for implicit dependencies)
depends_on = [aws_iam_role_policy_attachment.x]
```
