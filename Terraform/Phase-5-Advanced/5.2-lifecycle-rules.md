# Topic 5.2 — Lifecycle Rules: Control Resource Replacement Behaviour

> **Phase:** 5 — Advanced Patterns  
> **Topic:** prevent_destroy, create_before_destroy, ignore_changes, replace_triggered_by  
> **Priority:** 🔴 Must-Know (Production Safety)  
> **Previous:** [5.1 — Dynamic Blocks](./5.1-dynamic-blocks.md)

---

## 1. First Principles — Start Simple

### What Problem Does This Solve?

Some Terraform behaviors are dangerous by default:
- Changing an instance's AMI destroys and recreates it → **downtime**
- `terraform destroy` can delete a production database → **data loss**
- An auto-scaler changes instance count outside Terraform → **constant drift noise**
- Renaming a resource triggers destroy+recreate → **unnecessary recreation**

Lifecycle rules tell Terraform: **"I know the default behavior, but override it for this resource."**

## 2. Core Concept — Deep Dive

### The Four Lifecycle Rules

```hcl
resource "aws_instance" "web" {
  # ... resource config ...

  lifecycle {
    create_before_destroy = true    # Create new before destroying old
    prevent_destroy       = true    # Block terraform destroy
    ignore_changes        = [tags]  # Don't track these attributes
    replace_triggered_by  = [       # Force replacement when this changes
      aws_ami.custom.id
    ]
  }
}
```

### Rule 1: `create_before_destroy`

**Default behavior:** Terraform destroys the old resource, then creates the new one.  
**With `create_before_destroy`:** Terraform creates the new one first, verifies it's healthy, then destroys the old one.

```hcl
resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = var.instance_type

  lifecycle {
    create_before_destroy = true
  }
}
```

**When to use:**
- Resources that must maintain availability during replacement (instances behind an ALB)
- Resources with unique constraints (name/IP) might conflict — test carefully
- Launch configurations (old ones must not be deleted before the new ASG is updated)

**Gotcha:** If the resource has a unique name, create_before_destroy will fail because both resources temporarily exist with the same name. Solution: use `name_prefix` instead of `name`.

### Rule 2: `prevent_destroy`

**Blocks `terraform destroy` for this resource.**

```hcl
resource "aws_db_instance" "production" {
  identifier = "prod-database"
  # ... config ...

  lifecycle {
    prevent_destroy = true
  }
}
```

```bash
terraform destroy
# Error: Instance aws_db_instance.production has lifecycle.prevent_destroy
# set, but the plan calls for this resource to be destroyed.
```

**When to use:**
- Production databases (RDS, DynamoDB)
- S3 buckets with irreplaceable data
- KMS keys
- Any resource where accidental deletion = data loss

**Important:** This only prevents `destroy`. If you remove the resource block from code, Terraform errors instead of destroying. To actually destroy: remove `prevent_destroy`, apply, then destroy.

### Rule 3: `ignore_changes`

**Tells Terraform to ignore differences in specified attributes.**

```hcl
resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = var.instance_type

  lifecycle {
    ignore_changes = [
      ami,             # Don't recreate when a new AMI is available
      tags["Updated"], # Don't revert tags modified by automation
    ]
  }
}

# Ignore ALL changes (treat as "create once, never update")
resource "aws_iam_role" "legacy" {
  lifecycle {
    ignore_changes = all
  }
}
```

**When to use:**
- Auto-scaling modifies instance count → ignore `desired_capacity`
- AMI updates should be controlled separately → ignore `ami`
- External systems add tags → ignore specific tags
- ECS task definitions updated by CI/CD → ignore `task_definition`

**When NOT to use:**
- As a band-aid for drift you should fix — this hides problems
- For everything (defeats the purpose of IaC)

### Rule 4: `replace_triggered_by`

**Force resource replacement when a referenced value changes.**

```hcl
resource "aws_instance" "web" {
  ami           = var.ami_id
  instance_type = var.instance_type

  lifecycle {
    replace_triggered_by = [
      null_resource.force_replacement,
      aws_security_group.web.id,
    ]
  }
}

# Changing this resource forces the instance to be replaced
resource "null_resource" "force_replacement" {
  triggers = {
    rotation_date = "2024-01-15"
  }
}
```

**When to use:**
- Force instance replacement on AMI rotation schedules
- Recreate resources when dependent configuration changes
- Implement blue-green deployment patterns

## 3. Real-World Patterns

### Production Database Protection

```hcl
resource "aws_db_instance" "prod" {
  identifier           = "production-postgres"
  engine               = "postgres"
  instance_class       = "db.r5.large"
  allocated_storage    = 100
  deletion_protection  = true     # AWS-level protection
  skip_final_snapshot  = false    # Snapshot before any deletion

  lifecycle {
    prevent_destroy = true        # Terraform-level protection
    ignore_changes  = [
      password,                   # Rotated externally
    ]
  }
}
```

### Auto-Scaling Group

```hcl
resource "aws_autoscaling_group" "web" {
  desired_capacity = 3
  min_size         = 2
  max_size         = 10

  lifecycle {
    ignore_changes = [
      desired_capacity,    # Auto-scaling changes this
      target_group_arns,   # Modified by deployments
    ]
  }
}
```

### Zero-Downtime Launch Template Updates

```hcl
resource "aws_launch_template" "web" {
  name_prefix   = "web-"       # Use prefix, not name (for create_before_destroy)
  image_id      = data.aws_ami.amazon_linux.id
  instance_type = var.instance_type

  lifecycle {
    create_before_destroy = true
  }
}
```

## 10. Interview Preparation

### Q1: *"How do you prevent accidental deletion of a production database in Terraform?"* (Senior)

**Strong Answer:**
"Three layers: First, `lifecycle { prevent_destroy = true }` in Terraform — blocks `terraform destroy` for that resource. Second, `deletion_protection = true` on the RDS instance itself — blocks deletion even via AWS console or CLI. Third, `skip_final_snapshot = false` — creates a backup snapshot before any deletion. Together, you need to intentionally disable all three protections before the database can be deleted."

### Q2: *"When is `ignore_changes` appropriate vs when is it hiding a problem?"* (Senior)

**Strong Answer:**
"Appropriate when something legitimately changes outside Terraform by design — auto-scaling adjusting `desired_capacity`, CI/CD updating ECS `task_definition`, or external systems adding tags. It's hiding a problem when you use it to suppress drift caused by manual changes that should have been done in Terraform. The test: if someone asks 'why is this ignored?', can you explain a legitimate operational reason? If the reason is 'because it keeps showing changes in the plan,' that's a problem, not a reason."

## 13. Cheat Sheet

```hcl
lifecycle {
  create_before_destroy = true    # New first, then destroy old
  prevent_destroy       = true    # Block destroy entirely
  ignore_changes        = [attr]  # Don't track specific attributes
  ignore_changes        = all     # Never update (create-only)
  replace_triggered_by  = [ref]   # Force replacement on change
}

# Common patterns:
# Database:    prevent_destroy = true
# ASG:         ignore_changes = [desired_capacity]
# ECS:         ignore_changes = [task_definition]
# Templates:   create_before_destroy = true + name_prefix
```
