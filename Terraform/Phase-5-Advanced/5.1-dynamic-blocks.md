## 1. First Principles — Start Simple

### What Problem Does This Solve?

Some resources have **repeating nested blocks** whose count varies based on configuration. Security groups have variable numbers of ingress/egress rules. IAM policies have variable numbers of statements. ALBs have variable numbers of listener rules.

Without dynamic blocks, you'd hardcode every possible combination or copy-paste blocks.

---

## 2. Core Concept — Deep Dive

### Dynamic Blocks

A `dynamic` block generates zero or more **nested blocks** from a collection:

```hcl
# BEFORE (hardcoded — inflexible)
resource "aws_security_group" "web" {
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# AFTER (dynamic — data-driven)
variable "ingress_rules" {
  type = list(object({
    port        = number
    protocol    = string
    cidr_blocks = list(string)
    description = string
  }))
  default = [
    { port = 80,  protocol = "tcp", cidr_blocks = ["0.0.0.0/0"], description = "HTTP" },
    { port = 443, protocol = "tcp", cidr_blocks = ["0.0.0.0/0"], description = "HTTPS" },
  ]
}

resource "aws_security_group" "web" {
  name   = "${local.name_prefix}-web-sg"
  vpc_id = var.vpc_id

  dynamic "ingress" {
    for_each = var.ingress_rules
    content {
      description = ingress.value.description
      from_port   = ingress.value.port
      to_port     = ingress.value.port
      protocol    = ingress.value.protocol
      cidr_blocks = ingress.value.cidr_blocks
    }
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

### Dynamic Block Syntax

```hcl
dynamic "<block_label>" {
  for_each = <collection>    # List, map, or set
  iterator = <name>          # Optional — defaults to block_label
  content {
    # Use <name>.key and <name>.value
    # Or <block_label>.key and <block_label>.value
  }
}
```

### Conditional Dynamic Blocks

```hcl
# Generate SSH ingress ONLY if CIDRs are provided
dynamic "ingress" {
  for_each = length(var.allowed_ssh_cidrs) > 0 ? [1] : []
  content {
    description = "SSH"
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = var.allowed_ssh_cidrs
  }
}
```

### Complex for Expressions

```hcl
# Flatten nested structures
variable "subnet_config" {
  default = {
    public  = ["10.0.0.0/24", "10.0.1.0/24"]
    private = ["10.0.10.0/24", "10.0.11.0/24"]
  }
}

locals {
  # Flatten map of lists into a flat list of objects
  all_subnets = flatten([
    for tier, cidrs in var.subnet_config : [
      for i, cidr in cidrs : {
        key  = "${tier}-${i}"
        tier = tier
        cidr = cidr
      }
    ]
  ])

  # Convert to map for for_each
  subnet_map = { for s in local.all_subnets : s.key => s }
}

resource "aws_subnet" "all" {
  for_each   = local.subnet_map
  vpc_id     = aws_vpc.main.id
  cidr_block = each.value.cidr

  tags = {
    Name = each.key
    Tier = each.value.tier
  }
}
```

### Conditional Resources and Expressions

```hcl
# Ternary operator
instance_type = var.environment == "prod" ? "t3.medium" : "t3.micro"

# Conditional resource creation
resource "aws_nat_gateway" "main" {
  count = var.enable_nat ? 1 : 0
  # ...
}

# Conditional reference
nat_gateway_id = var.enable_nat ? aws_nat_gateway.main[0].id : null

# try() for optional attributes
name = try(var.config.name, "default-name")

# coalesce — first non-null value
region = coalesce(var.override_region, var.default_region, "us-east-1")
```

## 3. When to Use Dynamic Blocks

| Scenario | Use Dynamic Block? |
|----------|--------------------|
| Variable number of ingress rules | ✅ Yes |
| Fixed number of known blocks | ❌ No — hardcode for clarity |
| 1 or 2 conditional blocks | ⚠️ Maybe — simple conditional is clearer |
| Complex nested iteration | ✅ Yes, with `flatten()` helper |

**Anti-pattern warning:** Don't use dynamic blocks to generate the **entire** resource. If every attribute is dynamic, you're probably building a module instead.

## 10. Interview Preparation

### Q1: *"When do you use dynamic blocks vs separate resources?"* (Senior)

**Strong Answer:**
"Dynamic blocks for **nested blocks within a resource** — like security group rules, IAM policy statements, or listener rules. Separate resources with `for_each` for **multiple instances of a resource**. Don't use dynamic blocks to create multiple resources — that's what `for_each` is for. And don't overuse dynamic blocks — if the block structure is static and known, hardcoding is clearer."

### Q2: *"Explain the `flatten()` + `for_each` pattern."* (Senior)

**Strong Answer:**
"When you have nested data — like a map of lists — you can't use it directly with `for_each` because Terraform needs a flat map. The pattern is: use `flatten()` with nested `for` expressions to create a flat list of objects, then convert that list to a map using `{ for item in list : item.key => item }`. This gives for_each a flat map with unique string keys, and each entry carries all the context from the nested structure."

## 13. Cheat Sheet

```hcl
# Dynamic block
dynamic "block_name" {
  for_each = collection
  content {
    attr = block_name.value.attr
  }
}

# Flatten nested data
locals {
  flat = flatten([for k, v in var.nested : [for item in v : { key = "${k}-${item}", ... }]])
  map  = { for item in local.flat : item.key => item }
}

# Conditionals
value = condition ? true_val : false_val
try(expression, default)
coalesce(val1, val2, val3)
```
