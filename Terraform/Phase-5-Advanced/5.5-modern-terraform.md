## 1. First Principles — Start Simple

### What Problem Does This Solve?

Three long-standing Terraform problems now have built-in solutions:

| Problem | Old Workaround | Modern Solution |
|---------|---------------|-----------------|
| Secrets stored in state | Accept it, encrypt state | **Ephemeral values** — exist only during plan/apply, never written to state |
| Post-deploy validation | External test scripts | **Check blocks** — continuous assertions that run on every plan |
| `null_resource` requires null provider | Install extra provider | **`terraform_data`** — built-in resource for triggers and exec |

## 2. Ephemeral Values (Terraform 1.10+)

### The State-Secrets Problem — Finally Solved

In Topic 4.3, we documented honestly: *"The password is still in terraform.tfstate. This is Terraform's limitation."*

Ephemeral values change this. They exist **only in memory** during plan/apply and are **never written to state**.

### Ephemeral Variables

```hcl
# Declare an ephemeral variable
variable "db_password" {
  type      = string
  ephemeral = true    # ← NEW: value never touches state
  sensitive = true    # Still redacted from CLI output
}

# Pass to a resource that supports write-only arguments
resource "aws_db_instance" "main" {
  engine         = "postgres"
  instance_class = "db.t3.micro"
  username       = "admin"
  password       = var.db_password  # Ephemeral → NOT stored in state
}
```

**What `ephemeral = true` does:**
- ✅ Value exists during plan/apply (can be used in resources)
- ✅ Never written to state file
- ✅ Never shown in `terraform state show`
- ❌ Cannot be used in outputs (outputs are stored in state)
- ❌ Cannot be used in `locals` that feed into state-stored resources

### Ephemeral Data Sources

```hcl
# Read a secret ephemerally — value never hits state
ephemeral "aws_secretsmanager_secret_version" "db_password" {
  secret_id = aws_secretsmanager_secret.db.id
}

# Use the secret value (only exists during plan/apply)
resource "aws_db_instance" "main" {
  password = ephemeral.aws_secretsmanager_secret_version.db_password.secret_string
}
```

### Write-Only Arguments

Some resource arguments are **write-only** — they're sent to the API but never stored in state:

```hcl
resource "aws_db_instance" "main" {
  # password is a write-only argument in newer provider versions
  # It's sent to the RDS API but NOT stored in state
  password = var.db_password
}
```

### Ephemeral vs Sensitive

| | `sensitive = true` | `ephemeral = true` |
|--|-------------------|-------------------|
| Redacted from CLI | ✅ | ✅ |
| Stored in state | ✅ Yes (plaintext) | ❌ Never |
| Available in outputs | ✅ (if output is sensitive) | ❌ |
| Requires provider support | No | Yes (write-only args) |
| Available since | Terraform 0.14 | Terraform 1.10 |

## 3. Check Blocks (Terraform 1.5+)

### Continuous Post-Deploy Validation

Check blocks define **assertions that run on every plan/apply**. They query data sources and validate conditions — like infrastructure smoke tests built into Terraform.

```hcl
check "website_health" {
  # Query the deployed website
  data "http" "health" {
    url = "https://${aws_lb.main.dns_name}/health"
  }

  # Assert it's healthy
  assert {
    condition     = data.http.health.status_code == 200
    error_message = "Website health check failed! Status: ${data.http.health.status_code}"
  }
}
```

### Key Behaviors

- **Check blocks produce warnings, not errors** — they don't block apply
- **Run on every plan** — continuous validation, not one-time
- **Can use data sources** — query APIs, HTTP endpoints, DNS records
- **Independent from resources** — they validate the overall system, not individual resources

### Real-World Examples

```hcl
# Validate certificate is not expiring soon
check "certificate_expires" {
  data "aws_acm_certificate" "main" {
    domain = var.domain_name
  }

  assert {
    condition     = timecmp(data.aws_acm_certificate.main.not_after, timeadd(timestamp(), "720h")) > 0
    error_message = "TLS certificate expires within 30 days!"
  }
}

# Validate DNS is resolving
check "dns_resolution" {
  data "dns_a_record_set" "main" {
    host = var.domain_name
  }

  assert {
    condition     = length(data.dns_a_record_set.main.addrs) > 0
    error_message = "DNS record for ${var.domain_name} is not resolving!"
  }
}

# Validate RDS is accessible
check "database_accessible" {
  data "aws_db_instance" "main" {
    db_instance_identifier = aws_db_instance.main.identifier
  }

  assert {
    condition     = data.aws_db_instance.main.db_instance_status == "available"
    error_message = "RDS instance is not in 'available' status!"
  }
}
```

### Check Blocks vs Variable Validation vs terraform test

| | Variable Validation | Check Block | terraform test |
|--|-------------------|-------------|---------------|
| When runs | On input | Every plan | On demand |
| What validates | Variable values | System state | Resource behavior |
| Scope | Single variable | Cross-resource | Full config |
| Blocking | ✅ Errors | ⚠️ Warnings | ✅ Pass/Fail |
| Use case | Input validation | Smoke tests | Full test suite |

## 4. terraform_data Resource (Terraform 1.4+)

### Replacement for null_resource

Before Terraform 1.4, you needed the `null` provider for trigger-based local executions. `terraform_data` is built-in — no provider required.

```hcl
# ❌ OLD: null_resource (requires hashicorp/null provider)
resource "null_resource" "bootstrap" {
  triggers = {
    instance_id = aws_instance.web.id
  }
  provisioner "local-exec" {
    command = "ansible-playbook -i ${aws_instance.web.public_ip}, playbook.yml"
  }
}

# ✅ NEW: terraform_data (built-in, no provider needed)
resource "terraform_data" "bootstrap" {
  triggers_replace = [aws_instance.web.id]

  provisioner "local-exec" {
    command = "ansible-playbook -i ${aws_instance.web.public_ip}, playbook.yml"
  }
}
```

### terraform_data Features

```hcl
# Store arbitrary data
resource "terraform_data" "config" {
  input = {
    deployed_at = timestamp()
    version     = var.app_version
  }
}

# Reference stored data
output "deployment_info" {
  value = terraform_data.config.output
}

# Force replacement when specific values change
resource "terraform_data" "rotation" {
  triggers_replace = [
    var.certificate_arn,        # Recreate when cert changes
    var.force_rotation_date,    # Manual rotation trigger
  ]

  provisioner "local-exec" {
    command = "echo 'Certificate rotated at ${timestamp()}'"
  }
}
```

### terraform_data vs null_resource

| | `null_resource` | `terraform_data` |
|--|----------------|------------------|
| Provider needed | ✅ hashicorp/null | ❌ Built-in |
| Trigger attribute | `triggers` (map) | `triggers_replace` (list) |
| Store data | ❌ | ✅ `input` / `output` |
| Plan behavior | Shows as `null_resource` | Shows as `terraform_data` |
| Recommendation | Legacy | **Use this** |

## 5. Provisioners — Why They're Last Resort (and When They're Okay)

Provisioners execute scripts on local or remote machines during resource creation/destruction. **Terraform explicitly discourages them** because they break the declarative model.

```hcl
# local-exec: runs on the machine running 'terraform apply'
resource "aws_instance" "web" {
  ami           = data.aws_ami.al2023.id
  instance_type = "t3.micro"

  provisioner "local-exec" {
    command = "echo ${self.public_ip} >> inventory.txt"
  }
}

# remote-exec: runs on the created resource via SSH
resource "aws_instance" "web" {
  ami           = data.aws_ami.al2023.id
  instance_type = "t3.micro"

  connection {
    type        = "ssh"
    user        = "ec2-user"
    private_key = file("~/.ssh/id_rsa")
    host        = self.public_ip
  }

  provisioner "remote-exec" {
    inline = [
      "sudo yum update -y",
      "sudo yum install -y httpd",
      "sudo systemctl start httpd",
    ]
  }
}
```

### Why Provisioners Are Discouraged

| Problem | Explanation |
|---------|------------|
| **Not in plan** | Provisioners are not shown in `terraform plan` — you don't see what they'll do |
| **Break idempotency** | Running apply twice may re-execute the script, causing errors |
| **No rollback** | If the script fails halfway, you have a half-configured resource |
| **State doesn't track** | Terraform doesn't know what the provisioner did — can't detect drift |
| **Tight coupling** | SSH keys, network access, script dependencies — fragile |

### The Two Acceptable Uses

```hcl
# 1. One-time bootstrap (local-exec, not remote-exec)
resource "terraform_data" "ansible" {
  triggers_replace = [aws_instance.web.id]

  provisioner "local-exec" {
    command = "ansible-playbook -i '${aws_instance.web.public_ip},' site.yml"
  }
}

# 2. Registering with external system on destroy
resource "aws_instance" "web" {
  # ...
  provisioner "local-exec" {
    when    = destroy
    command = "curl -X DELETE https://monitoring.example.com/hosts/${self.id}"
  }
}
```

### What to Use Instead

| Instead of... | Use... |
|---------------|--------|
| remote-exec for software install | `user_data` script (AWS handles execution) |
| remote-exec for config management | Ansible/Chef/Puppet (separate tool) |
| Provisioner for AMI customization | Packer (build AMI with software pre-installed) |
| local-exec for API calls | HTTP provider or `terraform_data` with local-exec |

## 10. Interview Preparation

### Q1: *"What are ephemeral values and what problem do they solve?"* (Senior)

**Strong Answer:**
"Ephemeral values in Terraform 1.10+ exist only during plan/apply and are never written to state. This solves the fundamental state-secrets problem — where `sensitive = true` redacts from CLI but the value is still plaintext in state. With `ephemeral = true` on a variable or using an ephemeral data source, secrets like database passwords are used during resource creation but never persisted. Combined with write-only resource arguments, you can finally have zero secrets in state."

### Q2: *"What's a check block and how is it different from validation?"* (Senior)

**Strong Answer:**
"Check blocks run **continuous assertions on every plan** — they query data sources and validate system state. Unlike variable validation (which validates inputs), check blocks validate outcomes: is the website returning 200? Is the certificate expiring? Is the database in 'available' status? Check blocks produce warnings, not errors, so they don't block apply. They're like built-in smoke tests — if you deploy and something breaks, the next plan warns you."

### Q3: *"What replaced null_resource?"* (Mid)

**Strong Answer:**
"`terraform_data`, built-in since Terraform 1.4. It doesn't need the hashicorp/null provider. It has `triggers_replace` to force recreation when values change, and `input`/`output` attributes to store arbitrary data in state. Use it for trigger-based local-exec provisioners, dependency anchoring, and storing deployment metadata."

### Q4: *"Why are provisioners discouraged?"* (Mid — screening answer)

**Strong Answer:**
"Three reasons: they're not shown in plan (no preview of what they'll do), they break idempotency (running apply twice may re-execute scripts), and Terraform can't track what they did (no drift detection on provisioner changes). Better alternatives: `user_data` for instance bootstrap, Packer for AMI customization, Ansible for configuration management. The only acceptable use is `local-exec` for orchestration — calling Ansible, notifying an API, or one-time setup."

## 13. Cheat Sheet

```hcl
# ─── EPHEMERAL (1.10+) ───
variable "secret" {
  type      = string
  ephemeral = true      # Never in state
  sensitive = true      # Also redacted from CLI
}

ephemeral "aws_secretsmanager_secret_version" "x" {
  secret_id = "my-secret"
}

# ─── CHECK BLOCKS (1.5+) ───
check "name" {
  data "http" "x" { url = "https://..." }
  assert {
    condition     = data.http.x.status_code == 200
    error_message = "Health check failed"
  }
}

# ─── TERRAFORM_DATA (1.4+) ───
resource "terraform_data" "x" {
  triggers_replace = [some.resource.id]
  input = { key = "value" }
  provisioner "local-exec" {
    command = "echo done"
  }
}

# ─── PROVISIONERS (last resort) ───
# Prefer: user_data, Packer, Ansible
# Acceptable: local-exec for orchestration, destroy-time cleanup
```
