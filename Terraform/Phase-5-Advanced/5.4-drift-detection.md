## 1. First Principles — Start Simple

### What Problem Does This Solve?

Configuration **drift** is when real infrastructure differs from what Terraform code defines. It happens constantly in production:

| How Drift Happens | Example |
|-------------------|---------|
| **Manual console changes** | Engineer adds a security group rule via AWS Console during an incident |
| **Another tool** | Ansible modifies an instance's configuration |
| **AWS auto-operations** | AWS patches an RDS instance, changing its engine version |
| **Auto-scaling** | ASG adjusts `desired_capacity` based on load |
| **Incident response** | Someone opens port 22 for emergency debugging, forgets to close it |

**Why it matters:**
- **Security risk**: An opened port stays open indefinitely
- **Compliance violation**: Resources drift out of compliance posture
- **Surprise in Terraform plan**: Next `terraform apply` may revert a critical hotfix
- **State of confusion**: Nobody knows the real state of infrastructure

## 2. Core Concept — Deep Dive

### How Terraform Detects Drift

```
terraform plan:
  1. REFRESH — Query AWS APIs for current state of every resource
  2. COMPARE — Diff refreshed state against desired state (code)
  3. REPORT — Any differences = drift that plan will fix

The plan output IS your drift report.
```

```bash
$ terraform plan

# No drift:
No changes. Your infrastructure matches the configuration.

# Drift detected:
aws_security_group.web will be updated in-place
  ~ ingress {
    + cidr_blocks = ["0.0.0.0/0"]     # Manually added!
      from_port   = 22
      to_port     = 22
      protocol    = "tcp"
  }
```

### Scheduled Drift Detection

```yaml
# .github/workflows/drift-detection.yml
name: Drift Detection

on:
  schedule:
    - cron: '0 6 * * *'    # Every day at 6 AM UTC
  workflow_dispatch: {}     # Manual trigger

jobs:
  drift-check:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        environment: [dev, staging, prod]
    
    steps:
      - uses: actions/checkout@v4
      
      - uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: "1.11.0"
      
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets[format('TF_ROLE_{0}', matrix.environment)] }}
          aws-region: us-east-1
      
      - name: Terraform Init
        working-directory: environments/${{ matrix.environment }}
        run: terraform init -input=false
      
      - name: Detect Drift
        id: drift
        working-directory: environments/${{ matrix.environment }}
        run: |
          terraform plan -detailed-exitcode -input=false -no-color \
            2>&1 | tee drift_output.txt
          EXIT_CODE=$?
          if [ $EXIT_CODE -eq 2 ]; then
            echo "drift_detected=true" >> $GITHUB_OUTPUT
          else
            echo "drift_detected=false" >> $GITHUB_OUTPUT
          fi
        continue-on-error: true
      
      - name: Alert on Drift
        if: steps.drift.outputs.drift_detected == 'true'
        uses: slackapi/slack-github-action@v1
        with:
          payload: |
            {
              "text": "⚠️ Terraform drift detected in *${{ matrix.environment }}*!\nCheck: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}"
            }
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}
```

### Key: `terraform plan -detailed-exitcode`

```bash
terraform plan -detailed-exitcode

# Exit codes:
# 0 = No changes (no drift)
# 1 = Error
# 2 = Changes detected (drift!)

# Use in scripts:
terraform plan -detailed-exitcode
case $? in
  0) echo "No drift" ;;
  1) echo "Error — investigate" ;;
  2) echo "DRIFT DETECTED — alert team" ;;
esac
```

### Refresh-Only Mode (Terraform 1.2+)

```bash
# Only refresh state without planning changes
terraform apply -refresh-only

# This updates state to match reality WITHOUT changing infrastructure
# Use case: after a manual change you WANT to keep
# State is updated to reflect the manual change
# Your code still shows the old value → update code next
```

**Workflow for accepting a manual change:**
```bash
# 1. Someone manually changed the instance type during an incident
# 2. You want to KEEP the change

terraform apply -refresh-only
# State now reflects the real instance type

# 3. Update code to match
# instance_type = "t3.large"  (match the manual change)

terraform plan
# "No changes" — code, state, and reality are aligned
```

## 3. Drift Remediation Strategies

### Strategy 1: Revert (Code is Source of Truth)

```bash
# Drift: someone manually changed a tag
terraform plan
# Shows: ~ tags = { ~ "Environment" = "PRODUCTION" -> "prod" }

terraform apply
# Reverts the manual change. Code wins.
```

**When to use:** Unauthorized changes, accidental modifications, compliance violations.

### Strategy 2: Adopt (Accept the Reality)

```bash
# Drift: incident response changed security group rule
terraform apply -refresh-only
# State updated to match reality

# Update code to include the new rule
# Commit the code change
terraform plan
# "No changes" — aligned

# OR: if the change should be temporary, schedule a revert
```

**When to use:** Legitimate changes made during incidents, auto-scaling adjustments, AWS-initiated operations.

### Strategy 3: Ignore (Expected Drift)

```hcl
resource "aws_autoscaling_group" "web" {
  desired_capacity = 3

  lifecycle {
    ignore_changes = [desired_capacity]
  }
}
# Auto-scaling changes desired_capacity → Terraform ignores it
```

**When to use:** Attributes that are intentionally managed by another system.

### Strategy 4: Detect and Alert (Monitor Continuously)

Use the scheduled pipeline above. Don't auto-remediate — alert, investigate, then decide.

**When to use:** Production environments where blind remediation could cause outages.

## 4. Under the Hood — How Refresh Works

```
For EACH resource in state:

terraform plan:
  1. Read resource attributes from state (last known values)
  2. Call AWS API: DescribeVpcs(VpcId: vpc-abc123)
  3. Compare API response to state values
  4. If different: update in-memory state
  5. Compare updated state to code
  6. Generate plan diff

The REFRESH step is implicit in plan/apply.
You rarely need to run it separately.
```

### Performance at Scale

| State Size | Refresh Time | Why |
|------------|-------------|-----|
| 50 resources | ~10 seconds | 50 API calls |
| 500 resources | ~2 minutes | 500 API calls (some parallelized) |
| 5000 resources | ~10+ minutes | API throttling, pagination |

**Optimization:**
```bash
# Skip refresh (use when you KNOW nothing changed externally)
terraform plan -refresh=false

# Target specific resources (debug drift on one resource)
terraform plan -target=aws_security_group.web
```

## 5. AWS-Native Drift Detection Tools

### AWS Config

```hcl
# Deploy AWS Config rules via Terraform to monitor drift
resource "aws_config_config_rule" "sg_open_to_world" {
  name = "restricted-ssh"

  source {
    owner             = "AWS"
    source_identifier = "INCOMING_SSH_DISABLED"
  }
}

resource "aws_config_config_rule" "encrypted_volumes" {
  name = "encrypted-volumes"

  source {
    owner             = "AWS"
    source_identifier = "ENCRYPTED_VOLUMES"
  }
}
```

**AWS Config vs Terraform drift detection:**

| | Terraform Plan | AWS Config |
|---|---------------|-----------|
| Scope | Only Terraform-managed resources | All AWS resources |
| Trigger | On-demand (plan) or scheduled | Continuous (event-driven) |
| Baseline | Your .tf code | AWS Config rules |
| Remediation | `terraform apply` | AWS SSM Automation |
| Cost | Free (just API calls) | AWS Config pricing |

### CloudTrail for Drift Investigation

```bash
# WHO changed the security group?
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=ResourceName,AttributeValue=sg-abc123 \
  --start-time 2024-01-01 \
  --end-time 2024-01-02 \
  --query 'Events[].{Time:EventTime,User:Username,Event:EventName}'
```

## 10. Interview Preparation

### Q1: *"How do you handle configuration drift in production?"* (Senior)

**Strong Answer:**
"Four-part strategy: First, detect — scheduled `terraform plan -detailed-exitcode` runs nightly, alerting to Slack/PagerDuty on exit code 2. Second, investigate — use CloudTrail to find who made the change and why. Third, decide — either revert (apply to enforce code), adopt (update code to match reality), or ignore (lifecycle `ignore_changes` for expected drift). Fourth, prevent — restrict console access, use CI/CD as the only apply path, and enforce SCPs that require IaC tags on all resources."

### Q2: *"At 3 AM, your on-call engineer opened port 22 on a production security group via the console. What happens next?"* (Senior)

**Strong Answer:**
"The nightly drift detection catches it the next morning. The alert shows the SG change in the plan output. I investigate via CloudTrail — see the on-call engineer's action, confirm it was for an incident. Then I decide: if the incident is resolved, I revert by running `terraform apply` (the code doesn't have port 22, so Terraform removes it). If port 22 is still needed, I add it to code with `allowed_ssh_cidrs` restricted to our VPN CIDR, not 0.0.0.0/0. I also follow up: why was console access needed? Can we provide better tooling so future incidents don't require manual changes?"

### Q3: *"terraform plan shows 20 unexpected changes. Nothing was modified manually. What happened?"* (Trap)

**Strong Answer:**
"Several possibilities: Provider upgrade changed the schema or default values — check if the provider was recently upgraded. AWS made changes — some resources get modified by AWS (RDS maintenance, EKS version updates). State is stale — if refresh was skipped or state wasn't updated after an interrupted apply, these could be stale diffs. Lock file mismatch —different provider version resolves attributes differently. I'd check each changed resource to categorize: is it a real drift, a provider quirk, or a state issue?"

### Q4: *"What's the difference between `terraform apply -refresh-only` and regular `terraform apply`?"* (Mid)

**Strong Answer:**
"`-refresh-only` updates the state file to match current reality WITHOUT making any infrastructure changes. It says 'accept what AWS says as the new known state.' Regular apply changes infrastructure to match code. Use refresh-only when you WANT to keep a manual change — update state first, then update code to match. Use regular apply when you want code to be the source of truth and override manual changes."

## 12. Common Mistakes

| Mistake | Fix |
|---------|-----|
| No scheduled drift detection | Set up nightly `terraform plan -detailed-exitcode` |
| Auto-remediating all drift | Some drift is intentional (auto-scaling) — investigate first |
| Using `ignore_changes = all` on too many resources | Defeats IaC purpose — only ignore specific, justified attributes |
| Not using CloudTrail to investigate | Always know WHO and WHY before deciding how to remediate |
| Skipping `-refresh` to speed up plans | Hides real drift — only skip when you're certain |

## 13. Cheat Sheet

```bash
# Detect drift
terraform plan -detailed-exitcode    # Exit 0=clean, 2=drift
terraform plan -refresh=true         # Default: always refresh

# Accept manual changes into state
terraform apply -refresh-only        # Update state only

# Investigate drift
terraform plan -target=<resource>    # Check specific resource
terraform state show <resource>      # See what state thinks
# Compare with AWS console/CLI

# Prevent drift (CI/CD + scheduled detection)
# Cron: terraform plan -detailed-exitcode → alert on exit 2
```

```
Drift Response Decision Tree:
  Is the drift a security risk?  → Revert immediately
  Was it an authorized change?   → Adopt (update code)
  Is it from auto-scaling/AWS?   → Ignore (lifecycle rule)
  Is it unknown?                 → Investigate via CloudTrail
```
