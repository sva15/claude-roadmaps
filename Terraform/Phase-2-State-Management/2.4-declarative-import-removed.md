## 1. First Principles — Start Simple

### What Problem Does This Solve?

Topic 2.3 covered `terraform import` (CLI) and `terraform state rm` (CLI). These are **imperative** — one person runs the command, and the state change isn't visible to the team until after the fact. 

Modern Terraform (1.5+) replaces these with **declarative blocks in code**:

| Old (Imperative CLI) | New (Declarative Code) | Since |
|---------------------|----------------------|-------|
| `terraform import aws_vpc.main vpc-123` | `import { to = aws_vpc.main; id = "vpc-123" }` | Terraform 1.5 |
| `terraform state rm aws_instance.old` | `removed { from = aws_instance.old }` | Terraform 1.7 |
| (manual code writing for import) | `terraform plan -generate-config-out=generated.tf` | Terraform 1.5 |

**Why declarative is better:**
- ✅ Visible in pull requests → team reviews the import/removal
- ✅ Shown in `terraform plan` → no surprise state changes
- ✅ Tracked in git → auditability
- ✅ Works in CI/CD → no manual CLI commands needed

## 2. Core Concept — Deep Dive

### The `import` Block (Terraform 1.5+)

```hcl
# Step 1: Write the resource block matching the real resource
resource "aws_vpc" "imported" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_support   = true
  enable_dns_hostnames = true

  tags = {
    Name = "production-vpc"
  }
}

# Step 2: Add the import block (same file or separate imports.tf)
import {
  to = aws_vpc.imported
  id = "vpc-0abc123def456"
}
```

```bash
# Step 3: Plan — import is shown in the plan output
terraform plan

# Output:
# aws_vpc.imported will be imported
#   resource "aws_vpc" "imported" {
#     + arn          = "arn:aws:ec2:us-east-1:123:vpc/vpc-0abc123"
#     + cidr_block   = "10.0.0.0/16"
#     ~ tags         = {
#         + "Environment" = "prod"   # ← attribute you missed in code
#       }
#   }

# Step 4: Fix code to match reality, then apply
terraform apply

# Step 5: Remove the import block (it's one-time)
# Commit the change
```

### Auto-Generating Code for Imports (`-generate-config-out`)

**The biggest time saver for brownfield adoption.** Terraform reads the real resource and generates the HCL code for you.

```bash
# Step 1: Write ONLY the import block (NOT the resource block)
# imports.tf:
# import {
#   to = aws_vpc.production
#   id = "vpc-0abc123def456"
# }

# Step 2: Generate the resource code
terraform plan -generate-config-out=generated.tf

# Terraform produces generated.tf:
# resource "aws_vpc" "production" {
#   cidr_block           = "10.0.0.0/16"
#   enable_dns_support   = true
#   enable_dns_hostnames = true
#   tags = {
#     "Name"        = "production-vpc"
#     "Environment" = "prod"
#   }
# }
```

```bash
# Step 3: Review generated.tf — clean up unnecessary attributes
# Remove computed-only attributes (arn, id, owner_id)
# The generated code includes EVERYTHING — trim to what you want to manage

# Step 4: Move the cleaned-up resource to the appropriate .tf file
# Remove or merge generated.tf

# Step 5: Plan, verify, apply
terraform plan   # Should show import only, no attribute changes
terraform apply  # Import executes
# Remove import block after successful apply
```

**What `-generate-config-out` does NOT do perfectly:**
- Generates overly verbose code (includes computed attributes you shouldn't manage)
- May not handle complex nested blocks perfectly
- Doesn't organize code into your module structure
- **Always review and clean the generated code** before committing

### Importing Multiple Resources

```hcl
# Import an entire set of subnets at once
import {
  to = aws_subnet.public["us-east-1a"]
  id = "subnet-aaa111"
}

import {
  to = aws_subnet.public["us-east-1b"]
  id = "subnet-bbb222"
}

import {
  to = aws_subnet.private["us-east-1a"]
  id = "subnet-ccc333"
}

# All shown in a single terraform plan → reviewable
```

### Using for_each with Import Blocks (Terraform 1.7+)

```hcl
# Bulk import with for_each
locals {
  existing_subnets = {
    "us-east-1a" = "subnet-aaa111"
    "us-east-1b" = "subnet-bbb222"
    "us-east-1c" = "subnet-ccc333"
  }
}

import {
  for_each = local.existing_subnets
  to       = aws_subnet.public[each.key]
  id       = each.value
}
```

### The `removed` Block (Terraform 1.7+)

The declarative replacement for `terraform state rm`. Removes a resource from Terraform management **without destroying it**.

```hcl
# Scenario: you no longer want Terraform to manage this instance
# but the instance should keep running in AWS

removed {
  from = aws_instance.legacy_app

  lifecycle {
    destroy = false  # Keep the resource in AWS, just stop managing it
  }
}
```

```bash
terraform plan
# Output:
# aws_instance.legacy_app will no longer be managed by Terraform
# (the real AWS resource will NOT be destroyed)

terraform apply
# Resource removed from state. AWS instance still running.
# Remove the `removed` block after apply (it's one-time)
```

**With `destroy = true` (default):**
```hcl
# This is the same as removing the resource block from code
# The resource WILL be destroyed
removed {
  from = aws_instance.temp

  lifecycle {
    destroy = true   # Default — destroys the real resource
  }
}
```

### `removed` vs `terraform state rm`

| | `terraform state rm` (CLI) | `removed` block |
|--|---------------------------|-----------------|
| Declarative | ❌ | ✅ |
| In version control | ❌ | ✅ |
| Shown in plan | ❌ (immediate state write) | ✅ (planned) |
| Team-friendly | ❌ (one person runs) | ✅ (PR-reviewed) |
| Cleanup reminder | ❌ (easy to forget the resource) | ✅ (block reminds you) |

### Module Removal

```hcl
# Remove an entire module from management
removed {
  from = module.old_monitoring

  lifecycle {
    destroy = false  # Keep the resources, just stop managing
  }
}

# Terraform removes ALL resources within the module from state
```

## 3. Complete Brownfield Import Workflow

```
BROWNFIELD = existing infrastructure that wasn't built with Terraform

Step 1: INVENTORY
  └── aws ec2 describe-vpcs, describe-subnets, etc.
  └── List all resource types and IDs

Step 2: WRITE IMPORT BLOCKS
  └── One import {} block per resource
  └── Commit imports.tf to a branch

Step 3: GENERATE CODE
  └── terraform plan -generate-config-out=generated.tf
  └── Review and clean generated code
  └── Move to proper .tf files (main.tf, modules, etc.)

Step 4: PLAN & VERIFY
  └── terraform plan → should show "import" only, no changes
  └── If attribute changes shown → fix code to match reality

Step 5: PR REVIEW
  └── Team reviews the import plan
  └── Verify resource code matches real infrastructure

Step 6: APPLY
  └── terraform apply → resources imported into state

Step 7: CLEANUP
  └── Remove import {} blocks (one-time)
  └── Commit cleaned code
```

## 10. Interview Preparation

### Q1: *"How do you import existing infrastructure into Terraform?"* (Mid)

**Strong Answer:**
"In Terraform 1.5+, I use `import` blocks in code — `import { to = aws_vpc.main; id = 'vpc-123' }`. This is declarative, appears in the plan, and is PR-reviewable. For large imports, I use `-generate-config-out` to auto-generate resource code from the real infrastructure, then clean up the generated code. For bulk imports, I use `for_each` on import blocks. The old `terraform import` CLI still works but isn't team-friendly since it modifies state without a plan."

### Q2: *"You need to stop managing a resource without destroying it. What do you do?"* (Mid)

**Strong Answer:**
"In Terraform 1.7+, I use a `removed` block with `lifecycle { destroy = false }`. This declares the removal in code, shows in the plan, and goes through PR review. The old `terraform state rm` CLI command does the same thing but imperatively. I prefer `removed` blocks in team environments because the removal is tracked in git and the team can review it before it takes effect."

### Q3: *"Explain `-generate-config-out`. When is it useful?"* (Senior)

**Strong Answer:**
"It auto-generates Terraform resource blocks from real infrastructure during import. You write only the `import` block, then run `terraform plan -generate-config-out=generated.tf` — Terraform queries AWS and writes the full resource configuration. It's a massive time saver for brownfield adoption — importing 50 resources without manually writing each block. The caveat: the generated code is verbose and includes computed attributes. I always review and clean it before committing."

## 13. Cheat Sheet

```hcl
# ─── IMPORT (Terraform 1.5+) ───
import {
  to = aws_vpc.main
  id = "vpc-0abc123"
}

# Bulk import (1.7+)
import {
  for_each = local.resource_ids
  to       = aws_subnet.public[each.key]
  id       = each.value
}

# Auto-generate resource code:
# terraform plan -generate-config-out=generated.tf

# ─── REMOVED (Terraform 1.7+) ───
removed {
  from = aws_instance.old
  lifecycle { destroy = false }  # Keep in AWS, remove from state
}

# ─── WORKFLOW ───
# 1. Write import/removed blocks
# 2. terraform plan (shows import/removal in plan)
# 3. PR review
# 4. terraform apply
# 5. Remove import/removed blocks (one-time)
```
