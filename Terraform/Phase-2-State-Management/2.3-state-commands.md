> **Priority:** 🔴 Must-Know (Production Skill)

## 1. First Principles — Start Simple

### What Problem Does This Solve?

Production infrastructure doesn't stay neat. Things happen:

| Situation | Why State Surgery Is Needed |
|-----------|---------------------------|
| Colleague created a VPC manually (console/CLI) | Need to **import** it into Terraform management |
| You renamed a resource in code (`aws_subnet.old` → `aws_subnet.new`) | Need to **move** it in state to avoid destroy+recreate |
| A resource should no longer be Terraform-managed | Need to **remove** it from state without deleting it from AWS |
| Modules were restructured | Need to **move** resources between module namespaces |
| Need to debug what Terraform thinks it manages | Need to **list** and **show** state entries |

These operations are **routine in production**. Every experienced Terraform operator has done state surgery. It's not a last resort — it's a skill.

### Real-World Analogy

Think of state commands as **property registry operations**:

| Registry Operation | Terraform State Command |
|-------------------|------------------------|
| Look up all properties → `state list` | List everything Terraform manages |
| View property details → `state show` | See all attributes of one resource |
| Transfer ownership → `state mv` | Resource renamed/moved in code, update state to match |
| Remove from registry (property still exists) → `state rm` | Stop managing a resource without destroying it |
| Register a pre-existing property → `import` | Bring a manually-created resource under Terraform management |

## 2. Terraform Mental Model

### State Commands and the Three-Way Reconciliation

```
BEFORE state surgery:
  Code: aws_subnet.new_name     (renamed in code)
  State: aws_subnet.old_name    (old name in state)
  AWS: subnet-abc123             (real resource)
  
  Plan: DESTROY aws_subnet.old_name + CREATE aws_subnet.new_name
  Reality: destroys and recreates a real subnet (outage!)

AFTER state mv:
  Code: aws_subnet.new_name     
  State: aws_subnet.new_name    (moved via state mv)
  AWS: subnet-abc123            (untouched)
  
  Plan: No changes ✅
```

**The key insight:** State commands change the **mapping** between code and real resources. They never touch real infrastructure. They never modify your .tf files. They only modify the state file.

```bash
aws_subnet.public["us-east-1a"]

```

**Filtering:**
```bash
# Find all subnet resources
terraform state list | Select-String subnet

# Find all resources in a module
terraform state list | Select-String "module.vpc"

# Count resources
terraform state list | Measure-Object -Line
```

### Command 2: `terraform state show` — Inspect One Resource

```bash
$ terraform state show aws_vpc.main

# aws_vpc.main:
resource "aws_vpc" "main" {
    arn                  = "arn:aws:ec2:us-east-1:123456789:vpc/vpc-0abc123"
    cidr_block           = "10.0.0.0/16"
    enable_dns_hostnames = true
    enable_dns_support   = true
    id                   = "vpc-0abc123def456"
    instance_tenancy     = "default"
    tags                 = {
        "Environment" = "dev"
        "ManagedBy"   = "terraform"
        "Name"        = "terraform-learning-dev-vpc"
    }
    tags_all             = { ... }
}
```

**Use cases:**
- Compare state attributes to AWS console (spot drift)
- Find the real resource ID for AWS CLI commands
- Debug why plan shows unexpected changes

### Command 3: `terraform state mv` — Rename/Move Without Recreating

**Syntax:** `terraform state mv <source> <destination>`

**Scenario 1: Rename a resource**
```bash
# You renamed in code: aws_subnet.public → aws_subnet.public_subnet
# Without state mv: Terraform destroys old + creates new (outage!)

# Fix: rename in state to match code
terraform state mv aws_subnet.public aws_subnet.public_subnet

# Now plan shows: No changes ✅
```

**Scenario 2: Move resource into a module**
```bash
# You're refactoring: moving aws_vpc.main into module.vpc
terraform state mv aws_vpc.main module.vpc.aws_vpc.main
terraform state mv 'aws_subnet.public["us-east-1a"]' 'module.vpc.aws_subnet.public["us-east-1a"]'
```

**Scenario 3: Move resource between modules**
```bash
terraform state mv module.old_module.aws_instance.web module.new_module.aws_instance.web
```

**Alternative: `moved` blocks (Terraform 1.1+) — declarative approach:**
```hcl
# In your .tf code — Terraform handles the state rename automatically
moved {
  from = aws_subnet.public
  to   = aws_subnet.public_subnet
}

moved {
  from = aws_vpc.main
  to   = module.vpc.aws_vpc.main
}

# Run terraform plan → Terraform shows the move, not destroy+recreate
# After apply, the moved blocks can be removed (they're one-time)
```

### Command 4: `terraform state rm` — Forget a Resource

**Syntax:** `terraform state rm <resource_address>`

```bash
# Remove from Terraform's management — resource STILL EXISTS in AWS
terraform state rm aws_instance.temp

# Removed aws_instance.temp
# The EC2 instance is still running in AWS!
# Terraform just stopped tracking it.
```

**When to use:**
- Resource will be managed by a different tool (another Terraform config, CloudFormation, manual)
- Resource was imported by mistake
- A resource is causing persistent errors and you need to exclude it temporarily

**Danger:** The real resource continues to exist and incur costs. It becomes "unmanaged" — invisible to Terraform.

### Command 5: `terraform import` — Adopt Existing Resources

**Syntax:** `terraform import <resource_address> <cloud_resource_id>`

```bash
# Step 1: A VPC was created manually in AWS Console
# The VPC ID is: vpc-0xyz789abc123

# Step 2: Write the resource block in your .tf code
# (You must write the code BEFORE importing)

# Step 3: Import
terraform import aws_vpc.imported vpc-0xyz789abc123

# Step 4: Run plan to check alignment
terraform plan
# If plan shows changes → your code doesn't match the real resource
# Adjust code until plan shows "No changes"
```

**Import for different resource types:**
```bash
# VPC
terraform import aws_vpc.main vpc-0abc123

# Subnet
terraform import 'aws_subnet.public["us-east-1a"]' subnet-0def456

# Security Group
terraform import aws_security_group.web sg-0ghi789

# EC2 Instance
terraform import aws_instance.web i-0jkl012

# S3 Bucket
terraform import aws_s3_bucket.main my-bucket-name

# IAM Role
terraform import aws_iam_role.app my-role-name

# RDS Instance
terraform import aws_db_instance.main my-db-identifier
```

**Import with `import` block (Terraform 1.5+) — declarative approach:**
```hcl
# In your .tf code — no CLI command needed
import {
  to = aws_vpc.imported
  id = "vpc-0xyz789abc123"
}

# Run terraform plan → Terraform shows the import plan
# Run terraform apply → resource is imported into state
# After apply, remove the import block (one-time)
```

### Command 6: `terraform state pull` / `terraform state push`

```bash
# Download full state as JSON
terraform state pull > state_backup.json

# Upload state (DANGEROUS — use only for recovery)
terraform state push recovered_state.json
```

**`state pull` use cases:**
- Create a backup before risky operations
- Inspect full state JSON for debugging
- Compare states between environments

**`state push` rules:**
- Only use for disaster recovery
- Verify the serial number is correct (prevents accidental overwrite)
- Always backup current state first

## 4. Under the Hood — Critical

### How State Commands Interact with Locking

```
Every state-modifying command (mv, rm, import):
  1. Acquire lock
  2. Read state from backend
  3. Modify state in memory
  4. Write state to backend
  5. Release lock

Read-only commands (list, show, pull):
  1. Acquire lock (briefly)
  2. Read state from backend
  3. Display results
  4. Release lock
```

### State Serial Number

Every state write increments the `serial` number:
```
Initial:  serial = 1
After apply:  serial = 2
After state mv:  serial = 3
After import:  serial = 4
```

If you try to `state push` with a lower serial than what's in the backend, Terraform refuses (conflict protection).

### moved Blocks vs state mv

| Aspect | `terraform state mv` | `moved` block |
|--------|---------------------|---------------|
| Declarative? | No (imperative CLI) | Yes (in code) |
| Versioned? | No (state change not in git) | Yes (code change in git) |
| Team-friendly? | No (everyone must run the command) | Yes (plan shows the move automatically) |
| When executed | Immediately on CLI run | During next plan/apply |
| Remove after? | N/A | Remove after apply |

**Production recommendation:** Use `moved` blocks for team environments. Use `state mv` for emergencies or solo operations.

## 5. Project Implementation — Evolving Project

### Demonstrating State Surgery in the Project

**Scenario: Rename a resource**

```bash
# Current code has: aws_security_group.web
# You want to rename to: aws_security_group.web_server

# Option A: state mv (imperative)
terraform state mv aws_security_group.web aws_security_group.web_server
# Then rename in code

# Option B: moved block (declarative)
```

```hcl
# Add to main.tf temporarily
moved {
  from = aws_security_group.web
  to   = aws_security_group.web_server
}

# Rename the resource block
resource "aws_security_group" "web_server" {
  # ... same config as before
}

# Update references
resource "aws_instance" "web" {
  vpc_security_group_ids = [aws_security_group.web_server.id]  # Updated
}
```

```bash
terraform plan
# Shows: aws_security_group.web has moved to aws_security_group.web_server
# No destroy, no create — just a rename in state

terraform apply
# State updated. Remove the moved block after this.
```

**Scenario: Import an existing resource**

```hcl
# Someone created an S3 bucket manually. Import it.
# Step 1: Write the resource block
resource "aws_s3_bucket" "logs" {
  bucket = "my-manually-created-logs-bucket"
}

# Step 2: import block
import {
  to = aws_s3_bucket.logs
  id = "my-manually-created-logs-bucket"
}
```

```bash
terraform plan
# Shows: aws_s3_bucket.logs will be imported
# May also show attribute changes if your code doesn't match reality

terraform apply
# Resource imported. Remove the import block.
# Run plan again — should show "No changes" (or fix code to match).
```

## 6. Real-World Mapping

### When Each Command Is Used in Production

| Command | Production Scenario | Frequency |
|---------|-------------------|-----------|
| `state list` | Debugging, auditing, inventory | Daily |
| `state show` | Comparing state vs console, finding resource IDs | Daily |
| `state mv` | Code refactoring, module restructuring | Monthly |
| `state rm` | Removing orphaned resources, changing management tooling | Monthly |
| `import` | Onboarding pre-existing infrastructure to Terraform | During migrations |
| `state pull` | Backup before risky operations, debugging | As needed |
| `force-unlock` | CI crash recovery | Rare (hopefully!) |

### The Import Workflow for Brownfield

"Brownfield" = existing infrastructure that wasn't built with Terraform.

```
1. Inventory existing resources (AWS console, CLI)
2. Write Terraform code matching each resource
3. Import each resource: terraform import <address> <id>
4. Run terraform plan → fix code until "No changes"
5. Repeat for all resources
6. Now Terraform manages everything

For large environments (100+ resources):
- Use terraformer or import-into-terraform tools
- Import in batches (networking first, then compute, then database)
- Test imports in a non-production environment first
```

## 7. Failure Scenarios — Production Focus

### Scenario 1: state mv to Wrong Address

| | Details |
|--|---------|
| **Symptom** | Plan shows destroy of moved resource + create of a new one |
| **Root Cause** | Code address doesn't match the state mv destination |
| **How to Fix** | Run another `state mv` to correct, or undo by moving back to original address |
| **Prevention** | Always run `terraform plan` immediately after `state mv` to verify |

### Scenario 2: Import Without Matching Code

| | Details |
|--|---------|
| **Symptom** | `Error: resource address "aws_vpc.main" does not exist in configuration` |
| **Root Cause** | You ran `terraform import` before writing the resource block |
| **How to Fix** | Write the resource block first, then import |

### Scenario 3: state rm Forgotten Resource

| | Details |
|--|---------|
| **Symptom** | AWS resources running (costing money) but not in any Terraform state |
| **Root Cause** | Used `state rm` but forgot to delete the real resource |
| **Impact** | Unmanaged resource — invisible to Terraform, still incurring costs |
| **How to Fix** | Either: re-import into Terraform, or manually delete from AWS. Set up AWS billing alerts. |

### Scenario 4: Import Attribute Mismatch

| | Details |
|--|---------|
| **Symptom** | After import, `terraform plan` shows changes you didn't write |
| **Root Cause** | Your code doesn't exactly match the real resource's attributes |
| **How to Fix** | Run `terraform state show <resource>` to see all real attributes. Update your code to match. Iterate until plan is clean. |

## 8. Debugging Playbook

```
THE STATE SURGERY WORKFLOW (always follow this):

1. BACKUP
   └── terraform state pull > backup-$(date +%s).json

2. PLAN (before surgery)
   └── terraform plan
   └── Understand current state before changing anything

3. PERFORM SURGERY
   └── terraform state mv / rm / import

4. VERIFY
   └── terraform plan
   └── Expected: "No changes" (or only intended changes)
   └── If unexpected changes: undo surgery, investigate

5. COMMIT
   └── If using moved/import blocks: commit the code change
   └── If using CLI commands: document what was done in a commit message

GOLDEN RULE: Always run terraform plan after any state command
```

## 10. Interview Preparation

### Q1: *"You renamed a resource. Plan shows destroy+recreate. How do you fix it?"* (Mid — most common state question)

**Strong Answer:**
"I'd use `terraform state mv` to rename the resource address in state to match the new code name. For example, `terraform state mv aws_subnet.old aws_subnet.new`. After the move, plan should show no changes. In a team environment, I'd prefer using a `moved` block in code so the rename is versioned and tracked in git — when teammates pull the change, Terraform automatically handles the state move during their next plan."

### Q2: *"A colleague created 20 AWS resources manually during an incident. How do you bring them under Terraform management?"* (Senior)

**Strong Answer:**
"The import workflow: First, inventory the resources — list their types and IDs. Write Terraform resource blocks matching each one. Import each: `terraform import <address> <id>`. Run plan after each import to ensure code matches reality. Fix attribute mismatches until plan is clean. For 20 resources, this takes a few hours. I'd do it in a branch, verify with plan, then PR for team review. For Terraform 1.5+, I'd use `import` blocks in code so the entire import is declarative and reviewable."

### Q3: *"What's the difference between `terraform state rm` and `terraform destroy`?"* (Trap)

**Strong Answer:**
"`state rm` removes a resource from Terraform's tracking — the real AWS resource continues to exist and incur costs. `terraform destroy` (or removing the resource from code and applying) deletes the real AWS resource and removes it from state. `state rm` is for when you want the resource to keep existing but be managed differently. `destroy` is for when you want it gone entirely."

### Q4: *"How would you refactor a flat Terraform configuration into modules without downtime?"* (Senior)

**Strong Answer:**
"I'd use `terraform state mv` to move each resource from its flat address to the module namespace. For example, `terraform state mv aws_vpc.main module.vpc.aws_vpc.main`. Then restructure the code into the module directory. After all moves, run plan — it should show zero changes. In Terraform 1.1+, I'd use `moved` blocks to make this declarative. The key is that no real infrastructure changes — only the state mapping moves."

### Q5: *"When would you use `terraform state rm` in production?"* (Senior)

**Strong Answer:**
"Three main scenarios: First, when migrating a resource to be managed by a different Terraform configuration — remove from the old state, import in the new state. Second, when a resource should no longer be managed by Terraform at all — maybe it's been adopted by another IaC tool. Third, during an emergency when a resource is causing persistent plan/apply failures and you need to exclude it temporarily while you investigate."

## 11. Advanced Insights

### Bulk State Operations

For large refactoring (moving 50+ resources):

```bash
# Generate state mv commands
terraform state list | grep "aws_subnet" | while read line; do
  new=$(echo $line | sed 's/aws_subnet./module.vpc.aws_subnet./')
  echo "terraform state mv '$line' '$new'"
done

# Review the generated commands, then execute
```

### State Push — Last Resort Recovery

```bash
# 1. Current state is corrupted
# 2. Download the last known-good state from S3 version history
aws s3api get-object \
  --bucket my-state-bucket \
  --key dev/main/terraform.tfstate \
  --version-id "PREVIOUS_VERSION" \
  good-state.json

# 3. Push it (increment serial manually if needed)
terraform state push good-state.json

# 4. Plan to verify
terraform plan
```

### import Block vs terraform import CLI

| Aspect | `terraform import` (CLI) | `import` block (1.5+) |
|--------|--------------------------|----------------------|
| Declarative | No | Yes |
| In code/VCS | No | Yes |
| Team-friendly | No (everyone must run) | Yes (plan shows import) |
| Plan preview | No (direct state write) | Yes (shown in plan) |
| Rollback | Manual | Terraform handles |

## 12. Common Mistakes

### Beginner Mistakes

| Mistake | Fix |
|---------|-----|
| Importing before writing the resource block | Write code first, then import |
| Confusing `state rm` with `destroy` | `state rm` = forget resource. `destroy` = delete resource. |
| Not running `plan` after state surgery | Always verify with plan |

### Intermediate Mistakes

| Mistake | Fix |
|---------|-----|
| Using `state mv` in team environments without communication | Use `moved` blocks instead — tracked in git |
| Importing a resource with wrong type | Double-check resource type matches the real AWS resource |
| Not backing up state before surgery | Always `terraform state pull > backup.json` first |

### Production Mistakes

| Mistake | Fix |
|---------|-----|
| Bulk state operations without testing | Test the workflow in a dev environment first |
| `state rm` without deleting/re-importing the resource | Track unmanaged resources, set cost alerts |
| `state push` without verifying serial number | Can overwrite concurrent changes |

## 13. Cheat Sheet

### State Commands

```bash
# READ operations
terraform state list                    # All resources
terraform state show <address>          # One resource's attributes
terraform state pull                    # Full state as JSON

# WRITE operations (always backup first!)
terraform state mv <from> <to>          # Rename/move
terraform state rm <address>            # Forget (don't destroy)
terraform import <address> <id>         # Adopt existing resource
terraform state push <file>             # Upload state (dangerous)
terraform force-unlock <lock_id>        # Release stale lock
```

### Declarative Alternatives (Preferred)

```hcl
# Instead of: terraform state mv
moved {
  from = aws_subnet.old
  to   = aws_subnet.new
}

# Instead of: terraform import
import {
  to = aws_vpc.main
  id = "vpc-abc123"
}
```

### The Golden Workflow

```
1. terraform state pull > backup.json       # Backup
2. terraform state mv/rm/import ...         # Surgery
3. terraform plan                           # Verify: expect "No changes"
4. Commit code changes                      # Track in git
```
