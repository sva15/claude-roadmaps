# Topic 3.2 — Module Composition: Passing Outputs Between Modules

> **Phase:** 3 — Modules  
> **Topic:** Wiring modules together via outputs → variables  
> **Priority:** 🔴 Must-Know (Core Skill)  
> **Previous:** [3.1 — Writing Modules](./3.1-writing-modules.md)

---

## 1. First Principles — Start Simple

### What Problem Does This Solve?

In Topic 3.1, you created a VPC module and a compute module. But the compute module needs to know:
- Which VPC to deploy into → needs VPC ID
- Which subnet to use → needs subnet ID
- Which security groups to attach → needs SG ID

How does Module B use values from Module A? **Through outputs wired to inputs.**

### Real-World Analogy

Think of **LEGO blocks**: each block has studs (outputs) and holes (inputs). You connect block A's studs into block B's holes. The blocks don't know about each other's internals — only about the connection points.

```
Module A (VPC)          Module B (Compute)
  outputs:                variables:
    vpc_id    ────────▶    vpc_id
    subnet_ids ───────▶    subnet_id
```

---

## 2. Terraform Mental Model

### Data Flow Between Modules

```
Root module (orchestrator)
   │
   ├── module "vpc" {
   │     source = "./modules/vpc"
   │     vpc_cidr = "10.0.0.0/16"          # Variable IN
   │   }
   │     │
   │     └── outputs: vpc_id, subnet_ids    # Data OUT
   │
   ├── module "compute" {
   │     source = "./modules/compute"
   │     vpc_id    = module.vpc.vpc_id       # Wire A→B
   │     subnet_id = module.vpc.public_subnet_id_list[0]
   │   }
   │
   └── module "database" {
         source = "./modules/database"
         vpc_id     = module.vpc.vpc_id       # Wire A→C
         subnet_ids = module.vpc.private_subnet_id_list
         sg_id      = module.compute.app_sg_id # Wire B→C
       }
```

**Key insight:** The root module is the **wiring layer**. It doesn't create resources directly — it connects modules by passing one module's outputs as another module's inputs.

## 3. Core Concept — Deep Dive

### The Composition Pattern

**Step 1: Module A exposes outputs**
```hcl
# modules/vpc/outputs.tf
output "vpc_id" {
  value = aws_vpc.main.id
}
output "public_subnet_id_list" {
  value = [for s in aws_subnet.public : s.id]
}
output "private_subnet_id_list" {
  value = [for s in aws_subnet.private : s.id]
}
```

**Step 2: Module B declares variables matching those outputs**
```hcl
# modules/compute/variables.tf
variable "vpc_id" {
  type        = string
  description = "VPC ID to deploy instances into"
}
variable "subnet_id" {
  type        = string
  description = "Subnet ID for the instance"
}
variable "instance_type" {
  type    = string
  default = "t3.micro"
}
```

**Step 3: Root module wires them together**
```hcl
# root main.tf
module "vpc" {
  source       = "./modules/vpc"
  vpc_cidr     = "10.0.0.0/16"
  environment  = var.environment
  project_name = var.project_name
}

module "compute" {
  source        = "./modules/compute"
  vpc_id        = module.vpc.vpc_id                    # ← Wiring
  subnet_id     = module.vpc.public_subnet_id_list[0]  # ← Wiring
  instance_type = var.instance_type
  environment   = var.environment
  project_name  = var.project_name
}
```

### Cross-Stack References with terraform_remote_state

When modules are in **different Terraform configurations** (separate state files):

```hcl
# In compute config: read networking config's outputs
data "terraform_remote_state" "networking" {
  backend = "s3"
  config = {
    bucket = "my-company-terraform-state"
    key    = "prod/networking/terraform.tfstate"
    region = "us-east-1"
  }
}

resource "aws_instance" "web" {
  subnet_id = data.terraform_remote_state.networking.outputs.public_subnet_ids[0]
}
```

### Module for_each — Multiple Instances of a Module

```hcl
# Create one VPC per environment
module "vpc" {
  for_each = {
    dev     = { cidr = "10.0.0.0/16" }
    staging = { cidr = "10.1.0.0/16" }
    prod    = { cidr = "10.2.0.0/16" }
  }

  source       = "./modules/vpc"
  vpc_cidr     = each.value.cidr
  environment  = each.key
  project_name = var.project_name
}

# Reference: module.vpc["dev"].vpc_id
# Reference: module.vpc["prod"].public_subnet_id_list
```

## 4. Under the Hood — Critical

### Dependency Graph with Composed Modules

```
Terraform resolves inter-module dependencies through the wiring:

module.vpc.aws_vpc.main
  ↓ (module.compute references module.vpc.vpc_id)
module.compute.aws_security_group.web
  ↓
module.compute.aws_instance.web

The modules don't directly depend on each other —
the ROOT MODULE'S wiring creates the dependency.
```

### Module Output Types — What You Can Pass

| Type | Example Output | Example Wiring |
|------|---------------|----------------|
| `string` | `vpc_id = "vpc-abc"` | `vpc_id = module.vpc.vpc_id` |
| `list(string)` | `subnet_ids = ["sub-1", "sub-2"]` | `subnets = module.vpc.subnet_ids` |
| `map(string)` | `{"us-east-1a" = "sub-1"}` | `subnet_map = module.vpc.subnet_map` |
| `object` | `{ id = "vpc-abc", cidr = "10.0.0.0/16" }` | `vpc = module.vpc.vpc_details` |

```hcl
variable "vpc_id" {
  type        = string

  type        = string

  type    = string

}

  type = string
}

  type = string
}

```

```hcl
# modules/compute/main.tf
locals {
  name_prefix = "${var.project_name}-${var.environment}"
}

data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]
  filter {
    name   = "name"
    values = ["amzn2-ami-hvm-*-x86_64-gp2"]
  }
}

resource "aws_security_group" "web" {
  name        = "${local.name_prefix}-web-sg"
  description = "Security group for web server"
  vpc_id      = var.vpc_id    # ← Received from parent

  ingress {
    description = "HTTP"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = { Name = "${local.name_prefix}-web-sg" }
}

resource "aws_instance" "web" {
  ami                    = data.aws_ami.amazon_linux.id
  instance_type          = var.instance_type
  subnet_id              = var.subnet_id    # ← Received from parent
  vpc_security_group_ids = [aws_security_group.web.id]

  tags = { Name = "${local.name_prefix}-web" }
}
```

```hcl
# modules/compute/outputs.tf
output "instance_id" {
  value = aws_instance.web.id
}
output "public_ip" {
  value = aws_instance.web.public_ip
}
output "security_group_id" {
  value = aws_security_group.web.id
}
```

### Root Module Wiring

```hcl
# root main.tf — clean orchestration
module "vpc" {
  source       = "./modules/vpc"
  vpc_cidr     = var.vpc_cidr
  environment  = var.environment
  project_name = var.project_name
}

module "compute" {
  source        = "./modules/compute"
  vpc_id        = module.vpc.vpc_id
  subnet_id     = module.vpc.public_subnet_id_list[0]
  instance_type = var.instance_type
  environment   = var.environment
  project_name  = var.project_name
}
```

```hcl
# root outputs.tf — expose what matters
output "vpc_id" {
  value = module.vpc.vpc_id
}
output "web_public_ip" {
  value = module.compute.public_ip
}
```

## 6. Real-World Mapping

### In Production Architectures

```
module "networking" → outputs: vpc_id, subnet_ids, security_group_ids
    ↓
module "eks" → inputs: vpc_id, subnet_ids → outputs: cluster_endpoint, cluster_ca
    ↓  
module "app" → inputs: cluster_endpoint → deploys K8s resources
```

### terraform_remote_state vs Module Composition

| | Same Config (Module Composition) | Different Configs (Remote State) |
|--|-----------------------------------|----------------------------------|
| Speed | Fast (single plan) | Requires two separate applies |
| Coupling | Tighter (same apply cycle) | Looser (independent lifecycle) |
| Team ownership | Same team manages both | Different teams can own each |
| Blast radius | Changes affect everything in one plan | Changes isolated to one config |
| When to use | Components that deploy together | Components with independent lifecycles |

## 7. Failure Scenarios

### Scenario 1: Module Output Not Declared
**Symptom:** `Error: Unsupported attribute "subnet_ids" — object has no attribute`  
**Fix:** Add the missing output to the child module's `outputs.tf`.

### Scenario 2: Type Mismatch Between Output and Variable
**Symptom:** `Error: Invalid value for variable — string required`  
**Root Cause:** Module A outputs `list(string)`, Module B expects `string`.  
**Fix:** Index the list (`module.vpc.subnet_ids[0]`) or change the variable type.

### Scenario 3: Circular Module Dependency
**Symptom:** `Error: Cycle`  
**Root Cause:** Module A references Module B, and Module B references Module A.  
**Fix:** Extract the shared dependency into a third module, or restructure to eliminate the cycle.

## 8–9. Debugging & Practice

**Debug:** Use `terraform output` to see root-level outputs. Use `terraform state show module.vpc.aws_vpc.main` to inspect module resources.

**Practice:** Wire your VPC and compute modules. Add a third module (`modules/database`) that takes `vpc_id` and `private_subnet_ids` from the VPC module.

## 10. Interview Preparation

### Q1: *"How do modules communicate in Terraform?"* (Mid)

**Strong Answer:**
"Through outputs and variables. Module A exposes values via `output` blocks. The root module passes those as variable values to Module B. The root module is the wiring layer — it orchestrates data flow. Modules never reference each other directly; the root module always mediates."

### Q2: *"Module composition vs terraform_remote_state — when to use each?"* (Senior)

**Strong Answer:**
"Module composition within a single config for components that deploy together and share a lifecycle. Remote state for components owned by different teams with independent lifecycles. Remote state adds operational complexity — separate plan/apply cycles, eventual consistency between states, potential for stale references. Use it when the isolation benefits outweigh the complexity."

---

## 11-12. Advanced Insights & Common Mistakes

- **Don't pass everything through**: Only pass what modules actually need
- **Output everything useful**: Better to over-output than under-output
- **Avoid deep chains**: A→B→C→D means D can't deploy without A,B,C
- **Common mistake**: Forgetting to output a value, then needing to add it later and re-apply
