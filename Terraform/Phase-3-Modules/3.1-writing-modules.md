## 1. First Principles — Start Simple

### What Problem Does This Solve?

Your Phase 1-2 project has all resources in one flat directory. It works, but:

- **No reuse**: If you need a second VPC (staging), you copy-paste everything
- **No encapsulation**: Every variable, output, and resource is exposed at the top level
- **No separation of concerns**: Networking, compute, and security are all tangled
- **Scale problem**: At 500 resources, one flat config is unmanageable

A module solves this by being a **reusable, encapsulated unit of infrastructure** — like a function in programming.

### Real-World Analogy

| Programming | Terraform Modules |
|-------------|------------------|
| Function with parameters and return values | Module with variables (inputs) and outputs (return values) |
| Function hides implementation details | Module hides resource configuration details |
| Call the same function with different arguments | Use the same module with different variable values |
| `result = createVPC(cidr="10.0.0.0/16")` | `module "vpc" { source = "./modules/vpc"; vpc_cidr = "10.0.0.0/16" }` |

## 2. Terraform Mental Model

### Module = Directory of .tf Files

**Every Terraform configuration is already a module.** Your project root is the "root module." When you create a `modules/vpc/` directory with its own `.tf` files, that's a "child module."

```
Root Module calls → Child Module
     │                    │
  main.tf              main.tf
  variables.tf         variables.tf   (module's public API)
  outputs.tf           outputs.tf     (module's return values)
```

### How Modules Change the Dependency Graph

```
WITHOUT modules:
  aws_vpc → aws_subnet → aws_instance (flat graph)

WITH modules:
  module.vpc (encapsulates: aws_vpc, aws_subnet, aws_igw, aws_route_table)
       ↓
  module.compute (encapsulates: aws_instance, aws_security_group)

Terraform still builds a resource-level graph internally,
but YOU think at the module level — higher abstraction.
```

## 3. Core Concept — Deep Dive

### Module Directory Structure

```
modules/
  vpc/
    main.tf           # Resources (the implementation)
    variables.tf      # Inputs (the API)
    outputs.tf        # Outputs (the return values)
    versions.tf       # Provider version requirements
    README.md         # Documentation (always write this)
```

**Rule of thumb:** A module should do ONE thing well (single responsibility). A VPC module creates networking. A compute module creates EC2/EKS. Don't create a "everything" module.

### Designing the Module Interface

**variables.tf — The Module's Public API**

Design variables like you're designing a function API:
- **Required inputs**: Things the caller must provide (no default)
- **Optional inputs**: Sensible defaults that work for most cases
- **Validation**: Reject invalid values early

```hcl
# modules/vpc/variables.tf

# --- REQUIRED (no default) ---
variable "vpc_cidr" {
  type        = string
  description = "CIDR block for the VPC"

  validation {
    condition     = can(cidrhost(var.vpc_cidr, 0))
    error_message = "Must be a valid CIDR block."
  }
}

variable "environment" {
  type        = string
  description = "Environment name (dev, staging, prod)"

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Must be dev, staging, or prod."
  }
}

variable "project_name" {
  type        = string
  description = "Project name for resource naming and tagging"
}

# --- OPTIONAL (sensible defaults) ---
variable "az_count" {
  type        = number
  description = "Number of AZs to deploy subnets across"
  default     = 2
}

variable "enable_nat_gateway" {
  type        = bool
  description = "Create NAT Gateway for private subnet internet access"
  default     = false
}

variable "single_nat_gateway" {
  type        = bool
  description = "Use a single NAT (cheaper, less HA) vs one per AZ"
  default     = true
}

variable "additional_tags" {
  type        = map(string)
  description = "Additional tags to merge with default tags"
  default     = {}
}
```

**outputs.tf — What the Caller Can Access**

```hcl
# modules/vpc/outputs.tf

output "vpc_id" {
  description = "ID of the VPC"
  value       = aws_vpc.main.id
}

output "vpc_cidr" {
  description = "CIDR block of the VPC"
  value       = aws_vpc.main.cidr_block
}

output "public_subnet_ids" {
  description = "Map of AZ to public subnet ID"
  value       = { for k, s in aws_subnet.public : k => s.id }
}

output "private_subnet_ids" {
  description = "Map of AZ to private subnet ID"
  value       = { for k, s in aws_subnet.private : k => s.id }
}

output "public_subnet_id_list" {
  description = "List of public subnet IDs"
  value       = [for s in aws_subnet.public : s.id]
}

output "private_subnet_id_list" {
  description = "List of private subnet IDs"
  value       = [for s in aws_subnet.private : s.id]
}

output "internet_gateway_id" {
  description = "ID of the Internet Gateway"
  value       = aws_internet_gateway.main.id
}
```

**main.tf — The Implementation (Hidden From Callers)**

```hcl
# modules/vpc/main.tf

locals {
  name_prefix = "${var.project_name}-${var.environment}"

  common_tags = merge({
    Environment = var.environment
    Project     = var.project_name
    ManagedBy   = "terraform"
    Module      = "vpc"
  }, var.additional_tags)
}

data "aws_availability_zones" "available" {
  state = "available"
}

resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_support   = true
  enable_dns_hostnames = true

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-vpc"
  })
}

resource "aws_subnet" "public" {
  for_each = {
    for i, az in slice(data.aws_availability_zones.available.names, 0, var.az_count) :
    az => { cidr = cidrsubnet(var.vpc_cidr, 8, i), index = i }
  }

  vpc_id                  = aws_vpc.main.id
  cidr_block              = each.value.cidr
  availability_zone       = each.key
  map_public_ip_on_launch = true

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-public-${each.key}"
    Tier = "public"
  })
}

resource "aws_subnet" "private" {
  for_each = {
    for i, az in slice(data.aws_availability_zones.available.names, 0, var.az_count) :
    az => { cidr = cidrsubnet(var.vpc_cidr, 8, i + 10), index = i }
  }

  vpc_id            = aws_vpc.main.id
  cidr_block        = each.value.cidr
  availability_zone = each.key

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-private-${each.key}"
    Tier = "private"
  })
}

resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-igw"
  })
}

resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.main.id
  }

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-public-rt"
  })
}

resource "aws_route_table_association" "public" {
  for_each       = aws_subnet.public
  subnet_id      = each.value.id
  route_table_id = aws_route_table.public.id
}
```

### Calling the Module from Root

```hcl
# root main.tf
module "vpc" {
  source       = "./modules/vpc"
  vpc_cidr     = var.vpc_cidr
  environment  = var.environment
  project_name = var.project_name
  az_count     = 2
}

# Reference module outputs
resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = var.instance_type
  subnet_id     = module.vpc.public_subnet_id_list[0]

  tags = { Name = "${var.project_name}-${var.environment}-web" }
}
```

## 4. Under the Hood — Critical

### How Modules Appear in State

```json
{
  "resources": [
    {
      "module": "module.vpc",          // ← Module namespace
      "type": "aws_vpc",
      "name": "main",
      "instances": [{ "attributes": { "id": "vpc-abc123" } }]
    },
    {
      "module": "module.vpc",
      "type": "aws_subnet",
      "name": "public",
      "instances": [
        { "index_key": "us-east-1a", "attributes": { "id": "subnet-111" } },
        { "index_key": "us-east-1b", "attributes": { "id": "subnet-222" } }
      ]
    }
  ]
}
```

**State addresses with modules:**
```bash
terraform state list
# module.vpc.aws_vpc.main
# module.vpc.aws_subnet.public["us-east-1a"]
# module.vpc.aws_subnet.public["us-east-1b"]
# aws_instance.web (root module, no prefix)
```

### How Terraform Resolves Module References

```
module "vpc" {
  source   = "./modules/vpc"   ← Terraform reads .tf files from this directory
  vpc_cidr = "10.0.0.0/16"    ← Passed as var.vpc_cidr inside the module
}

module.vpc.vpc_id              ← Reads the vpc module's output named "vpc_id"
```

## 5. Project Implementation — Evolving Project

### Phase 3: Restructure Into Modules

**Before (Phase 2 — flat):**
```
Project/
├── backend.tf
├── providers.tf
├── variables.tf
├── locals.tf
├── main.tf          # ALL resources (VPC, subnets, SG, EC2)
└── outputs.tf
```

**After (Phase 3 — modular):**
```
Project/
├── backend.tf
├── providers.tf
├── variables.tf
├── main.tf          # Module calls only
├── outputs.tf
├── modules/
│   ├── vpc/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   ├── outputs.tf
│   │   └── versions.tf
│   └── compute/
│       ├── main.tf
│       ├── variables.tf
│       ├── outputs.tf
│       └── versions.tf
```

**Root `main.tf` (clean — only module calls):**
```hcl
module "vpc" {
  source       = "./modules/vpc"
  vpc_cidr     = var.vpc_cidr
  environment  = var.environment
  project_name = var.project_name
}

module "compute" {
  source        = "./modules/compute"
  environment   = var.environment
  project_name  = var.project_name
  vpc_id        = module.vpc.vpc_id
  subnet_id     = module.vpc.public_subnet_id_list[0]
  instance_type = var.instance_type
}
```

## 6. Real-World Mapping

### Module Design Patterns in Production

| Pattern | When to Use | Example |
|---------|-------------|---------|
| **Infrastructure module** | Reusable cloud components | `modules/vpc`, `modules/rds`, `modules/eks` |
| **Service module** | Application-specific | `modules/api-service` (ALB + ECS + RDS) |
| **Composition module** | Combines infrastructure modules | `modules/platform` (VPC + EKS + Monitoring) |

### Module as API — The Team Contract

```
YOUR TEAM provides:                OTHER TEAMS consume:
modules/vpc/                       module "vpc" {
  ├── variables.tf  ← API           source = "git::https://..."
  ├── outputs.tf    ← Return        vpc_cidr = "10.0.0.0/16"
  ├── main.tf       ← Hidden        environment = "prod"
  └── README.md     ← Docs        }
```

## 7. Failure Scenarios

### Scenario 1: Missing Output

| | Details |
|--|---------|
| **Symptom** | `Error: Unsupported attribute. This object has no attribute named "vpc_arn"` |
| **Root Cause** | Module doesn't expose the needed attribute as an output |
| **Fix** | Add the output to the module's `outputs.tf`, run `terraform apply` |

### Scenario 2: Module Source Change Requires Re-init

| | Details |
|--|---------|
| **Symptom** | `Error: Module not installed` after changing `source` |
| **Fix** | Run `terraform init` to download the updated module |

## 10. Interview Preparation

### Q1: *"What makes a good Terraform module?"* (Senior)

**Strong Answer:**
"A good module does one thing well. Its `variables.tf` is a clean API — required inputs have no defaults, optional inputs have safe defaults, everything has descriptions and validations. It outputs every ID and ARN downstream consumers might need. It never hardcodes region, account, or AZ. It works when instantiated multiple times with different inputs. And it has a README."

### Q2: *"Root module vs child module?"* (Mid)

**Strong Answer:**
"The root module is your working directory — where you run `terraform apply`. Child modules are directories called via `module` blocks. The root module is the orchestrator that wires child modules together. Child modules are reusable components. Resources in child modules appear in state under the `module.name` namespace."

### Q3: *"How do you migrate a flat config into modules without downtime?"* (Senior)

**Strong Answer:**
"Use `moved` blocks to declaratively remap state addresses from flat to module-namespaced. For example, `moved { from = aws_vpc.main; to = module.vpc.aws_vpc.main }`. Plan should show moves with zero creates or destroys. After applying, remove the moved blocks."

## 11. Advanced Insights

- **Module depth**: Avoid more than 2 levels of nesting (root → module → sub-module max). Deeper nesting makes debugging harder.
- **Variable passthrough anti-pattern**: Don't create a module that just passes all variables through to resources. That adds abstraction without value.
- **Testing**: Modules should be independently testable. A module that works only when called from one specific root config is too tightly coupled.

---

## 12. Common Mistakes

| Mistake | Fix |
|---------|-----|
| Module with 100+ resources | Split into smaller, focused modules |
| No outputs | Always output IDs, ARNs, endpoints |
| Hardcoded values inside module | Use variables for everything configurable |
| No README | Always document inputs, outputs, and usage |
