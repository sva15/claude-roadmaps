## 1. First Principles — Start Simple

### What Problem Does This Solve?

In a team environment, modules evolve:
- Someone adds a new variable (breaking change if required)
- Someone removes an output (breaks all callers)
- Someone changes resource behavior (unexpected in production)

Without versioning, every `terraform init` pulls the latest module code. A teammate's change to the shared VPC module could silently break your compute config. In production, this causes unplanned resource changes.

Versioning lets you **pin** a module to a specific version and **upgrade intentionally**.

## 2. Terraform Mental Model

### Module Sources — Where Modules Come From

```
module "vpc" {
  source = ???
}

Source options:
1. Local path:       "./modules/vpc"              ← Same repo
2. Terraform Registry: "terraform-aws-modules/vpc/aws"  ← Public registry
3. Git repository:   "git::https://github.com/org/modules.git//vpc?ref=v1.2.0"
4. S3 bucket:        "s3::https://bucket.s3.amazonaws.com/vpc.zip"
5. HTTP URL:         "https://example.com/modules/vpc.zip"
```

| Source | Versioning | Use Case |
|--------|-----------|----------|
| Local (`./modules/vpc`) | Same as your code (git commits) | Components within same project |
| Registry | `version = "~> 5.0"` constraint | Community or private modules |
| Git | `?ref=v1.2.0` (tag/branch/commit) | Organization-internal shared modules |
| S3/HTTP | Download specific artifact | CI/CD-built module artifacts |

## 3. Core Concept — Deep Dive

### Terraform Registry Modules

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"    # Same pessimistic constraint as providers

  name = "my-vpc"
  cidr = "10.0.0.0/16"

  azs             = ["us-east-1a", "us-east-1b"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24"]

  enable_nat_gateway = true
  single_nat_gateway = true
}
```

**Version constraints (same as providers):**

```hcl
version = "5.1.0"      # Exact: only 5.1.0
version = "~> 5.0"     # Pessimistic: ≥5.0, <6.0
version = "~> 5.1.2"   # Pessimistic: ≥5.1.2, <5.2.0
version = ">= 5.0"     # Open: 5.0 and above (risky)
```

### Git-Based Module Sources

```hcl
# Tag (recommended for production)
module "vpc" {
  source = "git::https://github.com/myorg/terraform-modules.git//modules/vpc?ref=v1.2.0"
}

# Branch (for development)
module "vpc" {
  source = "git::https://github.com/myorg/terraform-modules.git//modules/vpc?ref=main"
}

# Specific commit (most precise)
module "vpc" {
  source = "git::https://github.com/myorg/terraform-modules.git//modules/vpc?ref=abc123def"
}

# SSH (for private repos)
module "vpc" {
  source = "git::ssh://git@github.com/myorg/terraform-modules.git//modules/vpc?ref=v1.2.0"
}
```

**The `//` separator:** Splits the repo URL from the path within the repo. `repo.git//modules/vpc` means "clone repo.git, use the `modules/vpc` subdirectory."

### Version Pinning Strategy

```
DEVELOPMENT:
  source = "./modules/vpc"             # Local path, changes immediately
  # OR
  source = "git::...?ref=feature-branch"  # Test changes from branch

STAGING:
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.13"                 # Allow patch updates

PRODUCTION:
  source  = "terraform-aws-modules/vpc/aws"
  version = "= 5.13.0"               # Exact pin (maximum stability)
  # OR
  version = "~> 5.13.0"              # Allow 5.13.x patches only
```

### terraform init and Module Downloads

```bash
terraform init
  ├── Downloads modules to .terraform/modules/
  ├── Records versions in .terraform/modules/modules.json
  └── Local modules: symlinked, not copied

terraform init -upgrade
  └── Re-downloads modules to newest version matching constraints
```

## 4. Under the Hood — Critical

### The Module Lock — .terraform/modules/modules.json

```json
{
  "Modules": [
    {
      "Key": "vpc",
      "Source": "registry.terraform.io/terraform-aws-modules/vpc/aws",
      "Version": "5.13.0",
      "Dir": ".terraform/modules/vpc"
    }
  ]
}
```

This file records what was downloaded. Unlike `.terraform.lock.hcl` (for providers), this file is NOT committed to git — it's regenerated by `init`.

### Semantic Versioning for Modules

```
v5.13.2
│  │  │
│  │  └── PATCH: bug fixes, no behavior change
│  └──── MINOR: new features, backward compatible
└─────── MAJOR: breaking changes (removed variables, changed behavior)

~> 5.0   allows: 5.0 → 5.1 → 5.13 → 5.99 (blocks 6.0)
~> 5.13  allows: 5.13 → 5.14 → 5.99 (blocks 6.0 AND stays in 5.13+)
~> 5.13.0 allows: 5.13.0 → 5.13.1 → 5.13.99 (blocks 5.14)
```

## 5. Project Implementation — Evolving Project

### Using a Registry Module Alongside Custom Modules

```hcl
# root main.tf — mixing local and registry modules

# Custom local module (your code)
module "compute" {
  source        = "./modules/compute"
  vpc_id        = module.vpc.vpc_id
  subnet_id     = module.vpc.public_subnets[0]
  environment   = var.environment
  project_name  = var.project_name
}

# Community registry module (battle-tested, feature-rich)
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.13"

  name = "${var.project_name}-${var.environment}"
  cidr = var.vpc_cidr

  azs             = ["us-east-1a", "us-east-1b"]
  public_subnets  = [cidrsubnet(var.vpc_cidr, 8, 0), cidrsubnet(var.vpc_cidr, 8, 1)]
  private_subnets = [cidrsubnet(var.vpc_cidr, 8, 10), cidrsubnet(var.vpc_cidr, 8, 11)]

  enable_nat_gateway = var.environment == "prod"
  single_nat_gateway = true

  tags = {
    Environment = var.environment
    Project     = var.project_name
    ManagedBy   = "terraform"
  }
}
```

|------|--------|-----------|---------|
| Community | Terraform Registry | `version = "~> X.Y"` | terraform-aws-modules/vpc/aws |

## 7. Failure Scenarios

### Scenario 1: Breaking Module Upgrade
**Symptom:** After `terraform init -upgrade`, plan shows unexpected destroys.  
**Cause:** Major version bump removed a resource or changed behavior.  
**Fix:** Pin to the previous version, review changelog, upgrade intentionally.

### Scenario 2: Module Source Unavailable
**Symptom:** `Error: Failed to download module` during init.  
**Cause:** Git repo access revoked, registry down, or URL changed.  
**Fix:** Check network/auth. For critical production, consider vendoring modules.

---

## 8–9. Debugging & Practice

**Debug:** `terraform init` failures → check source URL, authentication, version constraints.

**Practice:**
1. Replace your local VPC module with the community `terraform-aws-modules/vpc/aws` module. Pin to `~> 5.0`. Run `terraform init`.
2. Try `terraform init -upgrade` with different version constraints to see what gets installed.
3. Create a git repo for your modules, push, and use `git::` source with `?ref=` tags.

## 10. Interview Preparation

### Q1: *"How do you manage module versions in production?"* (Senior)

**Strong Answer:**
"For production, I pin modules tightly — `version = '~> 5.13.0'` for registry modules or `?ref=v1.2.0` for git modules. Upgrades are intentional: create a PR, change the version constraint, run `terraform plan` to review impact, test in staging first, then merge. I never use `terraform init -upgrade` in CI without review. For critical infrastructure, I pin to exact versions."

### Q2: *"Local module vs registry module — when to use each?"* (Mid)

**Strong Answer:**
"Local modules for project-specific code that only makes sense in this context. Registry modules for well-solved problems like VPCs, EKS clusters, and RDS instances — they handle edge cases and best practices you'd otherwise have to discover yourself. Organization git repos for internal modules shared across teams that contain company-specific patterns."

## 11-13. Advanced Insights, Mistakes, Cheat Sheet

### Module Source Quick Reference

```hcl
# Local
source = "./modules/vpc"

# Terraform Registry
source  = "terraform-aws-modules/vpc/aws"
version = "~> 5.0"

# Git (HTTPS)
source = "git::https://github.com/org/repo.git//path?ref=v1.0.0"

# Git (SSH)
source = "git::ssh://git@github.com/org/repo.git//path?ref=v1.0.0"

# GitHub shorthand
source = "github.com/org/repo//path?ref=v1.0.0"
```

### Common Mistakes

| Mistake | Fix |
|---------|-----|
| No version constraint on registry modules | Always pin: `version = "~> X.Y"` |
| Using `ref=main` in production | Always use tags: `ref=v1.2.0` |
| `init -upgrade` in CI without review | Only upgrade in dedicated PRs |
| Not reading module docs/changelog before upgrading | Always review changes between versions |
