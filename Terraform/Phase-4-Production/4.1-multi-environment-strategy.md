> **Phase:** 4 — Production Patterns  
> **Topic:** Managing dev/staging/prod with Terraform  
> **Priority:** 🔴 Must-Know (Production Architecture)  
> **Previous:** [3.3 — Module Versioning](../Phase-3-Modules/3.3-module-versioning.md)

## 1. First Principles — Start Simple

### What Problem Does This Solve?

Every production system has multiple environments:
- **dev** — rapid iteration, breakable
- **staging** — production-like, used for validation
- **prod** — real users, zero tolerance for mistakes

You need the **same infrastructure code** to deploy **different instances** of infrastructure. But how do you manage this without duplication?

Three approaches exist:

| Approach | Mechanism | Isolation Level |
|----------|-----------|----------------|
| **Workspaces** | Same code, different state files | Logical (shared backend, shared code) |
| **Directory-per-env** | Copies of code calling shared modules | Physical (separate directories, separate state) |
| **Var-file-per-env** | Same code, different `.tfvars` files | Configuration (same directory, separate applies) |

## 2. Terraform Mental Model

### Approach 1: Terraform Workspaces

```
Single directory:
  ├── main.tf
  ├── variables.tf
  └── backend.tf

Three workspaces → three state files:
  s3://bucket/env:/dev/terraform.tfstate
  s3://bucket/env:/staging/terraform.tfstate
  s3://bucket/env:/prod/terraform.tfstate
```

```bash
terraform workspace new dev
terraform workspace new staging
terraform workspace new prod

terraform workspace select dev
terraform apply -var-file=dev.tfvars

terraform workspace select prod
terraform apply -var-file=prod.tfvars
```

```hcl
# Access current workspace in code
resource "aws_instance" "web" {
  instance_type = terraform.workspace == "prod" ? "t3.medium" : "t3.micro"
  
  tags = {
    Environment = terraform.workspace
  }
}
```

### Approach 2: Directory-Per-Environment (Recommended)

```
environments/
├── dev/
│   ├── main.tf          # Calls shared modules
│   ├── backend.tf       # key = "dev/terraform.tfstate"
│   ├── terraform.tfvars # Dev-specific values
│   └── providers.tf
├── staging/
│   ├── main.tf          # Same module calls, different vars
│   ├── backend.tf       # key = "staging/terraform.tfstate"
│   ├── terraform.tfvars
│   └── providers.tf
├── prod/
│   ├── main.tf          # May have extra resources (monitoring, scaling)
│   ├── backend.tf       # key = "prod/terraform.tfstate"
│   ├── terraform.tfvars
│   └── providers.tf
└── modules/             # Shared modules
    ├── vpc/
    └── compute/
```

### Approach 3: Var-File-Per-Environment

```
project/
├── main.tf
├── variables.tf
├── backend.tf           # Uses workspace or variable for key
├── environments/
│   ├── dev.tfvars
│   ├── staging.tfvars
│   └── prod.tfvars
```

```bash
terraform apply -var-file=environments/dev.tfvars
terraform apply -var-file=environments/prod.tfvars
# Each uses a different backend key (configured via variables or workspaces)
```

## 3. Core Concept — Deep Dive

### Workspaces — Deep Analysis

**Pros:**
- Minimal setup — one directory, one codebase
- Good for testing variations (feature branches)
- `terraform.workspace` variable available in code

**Cons (why production teams avoid them):**
- All environments share the **same backend configuration**
- Easy to accidentally apply to wrong workspace
- No structural differences between environments (prod needs monitoring that dev doesn't)
- `terraform.workspace` conditionals scattered through code become hard to maintain
- No isolation of IAM permissions between workspaces

**When to use workspaces:**
- Temporary environments (feature branch environments)
- Testing the same module with different inputs
- Small teams with simple infrastructure

### Directory-Per-Environment — Deep Analysis

**Pros:**
- Complete isolation (separate state, separate backend config)
- Each environment can have structural differences (prod has CloudWatch alarms, dev doesn't)
- Different IAM roles per environment
- CI/CD pipeline is clearer (cd into env directory, apply)
- Separate AWS accounts per environment is natural

**Cons:**
- Some duplication in `main.tf` across environments
- Must keep module calls in sync (mitigated by shared modules)

**When to use (most production teams):**
- Any team with > 1 engineer
- Any infrastructure that touches production
- When environments need structural differences

### Production Strategy: Directory + Shared Modules

```hcl
# environments/dev/main.tf
module "vpc" {
  source       = "../../modules/vpc"
  vpc_cidr     = "10.0.0.0/16"
  environment  = "dev"
  project_name = var.project_name
  az_count     = 2
}

module "compute" {
  source        = "../../modules/compute"
  vpc_id        = module.vpc.vpc_id
  subnet_id     = module.vpc.public_subnet_id_list[0]
  instance_type = "t3.micro"      # Small for dev
  environment   = "dev"
  project_name  = var.project_name
}
```

```hcl
# environments/prod/main.tf
module "vpc" {
  source              = "../../modules/vpc"
  vpc_cidr            = "10.2.0.0/16"   # Different CIDR
  environment         = "prod"
  project_name        = var.project_name
  az_count            = 3               # More AZs for prod
  enable_nat_gateway  = true            # NAT for prod
}

module "compute" {
  source        = "../../modules/compute"
  vpc_id        = module.vpc.vpc_id
  subnet_id     = module.vpc.public_subnet_id_list[0]
  instance_type = "t3.medium"     # Larger for prod
  environment   = "prod"
  project_name  = var.project_name
}

# Prod-only resources
module "monitoring" {
  source      = "../../modules/monitoring"
  vpc_id      = module.vpc.vpc_id
  environment = "prod"
}
```

## 4. Under the Hood — Critical

### How Workspace State Isolation Works

```
Default backend key: dev/main/terraform.tfstate

With workspaces, Terraform modifies the key:
  workspace "dev"     → env:/dev/dev/main/terraform.tfstate
  workspace "staging" → env:/staging/dev/main/terraform.tfstate
  workspace "prod"    → env:/prod/dev/main/terraform.tfstate

All share the same S3 bucket, same IAM permissions.
```

### How Directory-Based Isolation Works

```
environments/dev/backend.tf:
  bucket = "my-state-bucket"
  key    = "dev/main/terraform.tfstate"

environments/prod/backend.tf:
  bucket = "prod-state-bucket"            # Different bucket!
  key    = "prod/main/terraform.tfstate"
  role_arn = "arn:aws:iam::PROD:role/tf"  # Different IAM role!

Complete isolation: different bucket, different credentials, different state.
```

## 6–9. Real-World, Failures, Debugging, Practice

**Biggest failure:** Applying prod changes with dev workspace selected.  
**Prevention:** Directory-per-env + CI/CD that cd's into the right directory.

**Practice:**
1. Create `environments/dev/` and `environments/prod/` directories
2. Both call the same modules with different variables
3. Apply each independently — verify separate state files

## 10. Interview Preparation

### Q1: *"Workspaces vs directories for multi-environment — which and why?"* (Senior)

**Strong Answer:**
"Directories per environment for production. Workspaces give logical isolation but share backend config and IAM — too risky for prod. With directories, each environment has its own `backend.tf` pointing to separate state, can use different AWS accounts and IAM roles, and can have structural differences. The trade-off is some duplication in module call configuration, but shared modules keep the actual infrastructure logic DRY."

### Q2: *"How do you prevent someone from deploying dev code to production?"* (Senior)

**Strong Answer:**
"Multiple layers: separate AWS accounts per environment with different IAM roles, separate state files with different backend configs, CI/CD pipeline that maps branches to environments (main→prod, develop→staging), PR approval requirements for prod changes, and plan output posted as PR comments for team review before apply."
