# Topic 4.3 — Secrets Handling in Terraform

> **Phase:** 4 — Production Patterns  
> **Topic:** Managing sensitive data without exposing secrets  
> **Priority:** 🔴 Must-Know (Security Requirement)  
> **Previous:** [4.2 — CI/CD Pipeline](./4.2-cicd-pipeline.md)

---

## 1. First Principles — Start Simple

### What Problem Does This Solve?

Infrastructure needs secrets: database passwords, API keys, TLS certificates. But:
- **State stores secrets in plaintext** — anything passed to a resource attribute ends up in state
- **tfvars files might be committed to git** — secrets in version history forever
- **Plan output shows values** — console logs in CI expose sensitive data

You need strategies to **minimize secrets in Terraform** and **protect secrets that must exist**.

---

## 2. Core Concept — Deep Dive

### The Secrets Hierarchy (Best → Worst)

```
1. BEST:  Don't manage the secret in Terraform at all
   └── Use AWS Secrets Manager / SSM Parameter Store
   └── Application reads secret at runtime

2. GOOD:  Generate the secret in Terraform, store in Secrets Manager
   └── random_password resource → aws_secretsmanager_secret
   └── No human ever sees the password

3. ACCEPTABLE: Pass secret via environment variable
   └── TF_VAR_db_password in CI/CD
   └── Never stored in files

4. BAD:   Pass secret via tfvars file
   └── Easy to accidentally commit
   └── Must be gitignored

5. WORST: Hardcode secret in .tf files
   └── In version history forever
   └── Anyone with repo access sees it
```

### Pattern 1: Generate + Store (Recommended)

```hcl
# Generate a random password — no human input
resource "random_password" "db" {
  length  = 24
  special = true
}

# Store in AWS Secrets Manager
resource "aws_secretsmanager_secret" "db_password" {
  name = "${local.name_prefix}/db-password"
}

resource "aws_secretsmanager_secret_version" "db_password" {
  secret_id     = aws_secretsmanager_secret.db_password.id
  secret_string = random_password.db.result
}

# Use in RDS — Terraform manages it, humans never see it
resource "aws_db_instance" "main" {
  engine         = "postgres"
  instance_class = "db.t3.micro"
  username       = "admin"
  password       = random_password.db.result  # Still in state!

  tags = { Name = "${local.name_prefix}-db" }
}
```

> ⚠️ **The password is still in terraform.tfstate.** This is Terraform's limitation. Encrypt and protect state.

### Pattern 2: Reference Existing Secret (Best Isolation)

```hcl
# Read a secret that was created OUTSIDE Terraform
data "aws_secretsmanager_secret_version" "db_password" {
  secret_id = "prod/db-password"
}

resource "aws_db_instance" "main" {
  password = data.aws_secretsmanager_secret_version.db_password.secret_string
}
# ⚠️ secret_string is STILL stored in state as a data source attribute
```

### Pattern 3: Environment Variables for CI/CD

```bash
# In GitHub Actions:
env:
  TF_VAR_db_password: ${{ secrets.DB_PASSWORD }}

# Terraform picks up TF_VAR_* automatically
# No files, no git history, no tfvars
```

```hcl
variable "db_password" {
  type      = string
  sensitive = true   # Redacts from plan output and CLI
}
```

### The `sensitive` Attribute

```hcl
variable "db_password" {
  type      = string
  sensitive = true
}

output "db_endpoint" {
  value = aws_db_instance.main.endpoint
}

output "db_password" {
  value     = var.db_password
  sensitive = true    # Required if the output contains sensitive data
}
```

**What `sensitive = true` does:**
- ✅ Redacts value from `terraform plan` CLI output (`<sensitive>`)
- ✅ Redacts from `terraform output` (shows `<sensitive>`)
- ✅ Prevents accidental display in CI logs
- ❌ Does NOT encrypt the value in state
- ❌ Does NOT prevent the value from being in state

### AWS SSM Parameter Store Alternative

```hcl
# Store config values (non-secret)
resource "aws_ssm_parameter" "api_url" {
  name  = "/${var.project}/${var.environment}/api-url"
  type  = "String"
  value = "https://api.example.com"
}

# Store secrets (encrypted)
resource "aws_ssm_parameter" "api_key" {
  name  = "/${var.project}/${var.environment}/api-key"
  type  = "SecureString"    # Encrypted with KMS
  value = var.api_key       # Still in Terraform state
}
```

## 3. State-Level Protection

### Encrypt State at Rest

```hcl
terraform {
  backend "s3" {
    bucket     = "my-terraform-state"
    key        = "prod/terraform.tfstate"
    region     = "us-east-1"
    encrypt    = true
    kms_key_id = "arn:aws:kms:us-east-1:123:key/mrk-abc"  # KMS encryption
    use_lockfile = true
  }
}
```

### Restrict State Access

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:PutObject"],
    "Resource": "arn:aws:s3:::my-terraform-state/prod/*",
    "Principal": {
      "AWS": "arn:aws:iam::123:role/terraform-prod-apply"
    }
  }]
}
```

## 10. Interview Preparation

### Q1: *"How do you handle secrets in Terraform?"* (Senior)

**Strong Answer:**
"I minimize secrets managed by Terraform. For database passwords, I use `random_password` to generate them and `aws_secretsmanager_secret` to store them — humans never see the password. Applications read from Secrets Manager at runtime, not from Terraform outputs. For secrets that must be passed to Terraform (API keys for third-party providers), I use `TF_VAR_` environment variables in CI/CD, never tfvars files. I always mark secrets with `sensitive = true` to prevent plan output leakage. And I accept that state will contain secrets — so state is encrypted with KMS and access is restricted via IAM."

### Q2: *"Sensitive is set to true but the secret is still in state. Is that a problem?"* (Trap)

**Strong Answer:**
"Yes, `sensitive = true` only redacts from CLI and plan output — it doesn't affect state storage. The value is plaintext in the state file. This is a fundamental Terraform limitation. Mitigation: encrypt state at rest with KMS, restrict state access to only CI/CD roles, enable S3 versioning with lifecycle policies to rotate old versions. For the most sensitive secrets, consider not managing them in Terraform at all — create them via a separate secure process and reference via data source."

## 13. Cheat Sheet

```
NEVER: Hardcode secrets in .tf files
NEVER: Commit terraform.tfvars with secrets
ALWAYS: Mark with sensitive = true
ALWAYS: Encrypt state (KMS)
ALWAYS: Restrict state access (IAM)
PREFER: random_password → Secrets Manager
PREFER: TF_VAR_ env vars for CI/CD
BEST: Don't manage the secret in Terraform at all
```
