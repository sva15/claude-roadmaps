## 1. First Principles вҖ” Start Simple

### What Problem Does This Solve?

Running `terraform apply` from your laptop is dangerous:
- No audit trail (who applied what, when?)
- No review process (did anyone check the plan?)
- No consistency (different CLI versions, different credentials)
- Human error (applying to the wrong environment)

CI/CD automates the Terraform lifecycle:
1. Developer pushes code вҶ’ PR created
2. CI runs `terraform plan` вҶ’ posts plan as PR comment
3. Team reviews the plan вҶ’ approves or requests changes
4. On merge вҶ’ CI runs `terraform apply` (automated, audited, consistent)

## 3. Core Concept вҖ” Deep Dive

### GitHub Actions Workflow вҖ” Complete Example

```yaml
# .github/workflows/terraform.yml
name: Terraform CI/CD

on:
  pull_request:
    branches: [main]
    paths:
      - 'environments/**'
      - 'modules/**'
  push:
    branches: [main]
    paths:
      - 'environments/**'
      - 'modules/**'

permissions:
  id-token: write    # For OIDC authentication
  contents: read
  pull-requests: write  # To post plan comments

env:
  TF_VERSION: "1.11.0"
  AWS_REGION: "us-east-1"

jobs:
  # в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  # PLAN (runs on pull requests)
  # в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  plan:
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    strategy:
      matrix:
        environment: [dev, staging, prod]
    
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: ${{ env.TF_VERSION }}

      - name: Configure AWS Credentials (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/github-terraform-${{ matrix.environment }}
          aws-region: ${{ env.AWS_REGION }}

      - name: Terraform Init
        working-directory: environments/${{ matrix.environment }}
        run: terraform init -input=false

      - name: Terraform Validate
        working-directory: environments/${{ matrix.environment }}
        run: terraform validate

      - name: Terraform Plan
        id: plan
        working-directory: environments/${{ matrix.environment }}
        run: |
          terraform plan -input=false -no-color -out=tfplan \
            2>&1 | tee plan_output.txt
          echo "plan_output<<EOF" >> $GITHUB_OUTPUT
          cat plan_output.txt >> $GITHUB_OUTPUT
          echo "EOF" >> $GITHUB_OUTPUT

      - name: Post Plan to PR
        uses: actions/github-script@v7
        with:
          script: |
            const output = `### Terraform Plan вҖ” ${{ matrix.environment }}
            
            \`\`\`
            ${{ steps.plan.outputs.plan_output }}
            \`\`\`
            
            *Pushed by: @${{ github.actor }}*`;
            
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: output
            });

  # в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  # APPLY (runs on merge to main)
  # в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  apply:
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    strategy:
      matrix:
        environment: [dev, staging, prod]
      max-parallel: 1  # Apply sequentially: dev вҶ’ staging вҶ’ prod

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: ${{ env.TF_VERSION }}

      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/github-terraform-${{ matrix.environment }}
          aws-region: ${{ env.AWS_REGION }}

      - name: Terraform Init
        working-directory: environments/${{ matrix.environment }}
        run: terraform init -input=false

      - name: Terraform Apply
        working-directory: environments/${{ matrix.environment }}
        run: terraform apply -input=false -auto-approve
```

### OIDC Authentication вҖ” No Long-Lived Credentials

```json
// IAM Trust Policy for GitHub OIDC
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Federated": "arn:aws:iam::123456789:oidc-provider/token.actions.githubusercontent.com"
    },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringEquals": {
        "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
      },
      "StringLike": {
        "token.actions.githubusercontent.com:sub": "repo:myorg/terraform-infra:*"
      }
    }
  }]
}
```

### Key Flags for CI/CD

```bash
-input=false      # Never prompt for input (CI has no stdin)
-auto-approve     # Skip "yes" confirmation (already reviewed in PR)
-no-color         # Clean output for PR comments
-out=tfplan       # Save plan to file for later apply
-compact-warnings # Reduce noise in CI output
```

## 4вҖ“5. Under the Hood & Project Implementation

### Pipeline Safety Features

| Feature | Why | How |
|---------|-----|-----|
| Plan on PR | Review before apply | `if: github.event_name == 'pull_request'` |
| Apply on merge only | Only reviewed changes hit infra | `if: github.ref == 'refs/heads/main'` |
| Sequential apply | dev before prod | `max-parallel: 1` in strategy |
| OIDC, not API keys | No long-lived secrets | `id-token: write` permission |
| Plan posted as comment | Visible to all reviewers | `actions/github-script` |
| Version-pinned Terraform | Consistency across runs | `terraform_version: ${{ env.TF_VERSION }}` |

## 10. Interview Preparation

### Q1: *"How do you implement CI/CD for Terraform?"* (Mid)

**Strong Answer:**
"Plan on pull requests with output posted as comments for review. Apply only on merge to main. Use OIDC for keyless AWS authentication вҖ” no stored secrets. Pin the Terraform version. Set `-input=false` and `-auto-approve` in CI. Apply environments sequentially вҖ” dev first, then staging, then prod. Failed dev apply blocks prod apply."

### Q2: *"How do you prevent a junior engineer from applying directly to production?"* (Senior)

**Strong Answer:**
"Remove humans from the apply path entirely. No one has direct AWS credentials for production вҖ” only CI has them via OIDC. Branch protection requires PR approval before merge. Production apply only runs from the main branch after PR approval. State bucket IAM restricts write access to the CI role only. Even a senior engineer can't manually apply to production вҖ” they must go through the reviewed pipeline."
