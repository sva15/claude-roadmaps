## 1. The Question This Answers

This is the #1 senior Terraform interview question. Your answer must cover: directory layout, module boundaries, dependency order, variable flow, state isolation, CI/CD integration, and secrets handling. This document is your complete reference.

## 2. Architecture Overview

### Target Infrastructure

```
Internet
   │
   ├── ALB (Application Load Balancer)
   │     ├── Target Group → ECS Service / EC2 ASG
   │     └── HTTPS (ACM certificate)
   │
   ├── ECS/EC2 (Compute)
   │     ├── Application containers / instances
   │     ├── Reads secrets from Secrets Manager
   │     └── Connects to RDS via private subnet
   │
   ├── RDS (Database)
   │     ├── PostgreSQL in private subnets
   │     ├── Multi-AZ for production
   │     └── Password in Secrets Manager
   │
   └── VPC (Networking)
         ├── Public subnets (ALB, NAT Gateway)
         ├── Private subnets (Compute, Database)
         ├── NAT Gateway (outbound internet for private subnets)
         └── Security Groups (least-privilege access)
```

## 4. Module Dependency Order

```
                  ┌──────────────────┐
                  │  module.secrets  │  ← Created first (no dependencies)
                  │  random_password │
                  │  secrets_manager │
                  └────────┬─────────┘
                           │
                  ┌────────▼─────────┐
                  │   module.vpc     │  ← VPC is the foundation
                  │   VPC, subnets   │
                  │   IGW, NAT, RTs  │
                  └──┬─────────────┬─┘
                     │             │
          ┌──────────▼──┐    ┌────▼──────────┐
          │ module.alb  │    │ module.database│
          │ ALB, TG     │    │ RDS, SG       │
          │ Listener    │    │ Subnet group  │
          └──────┬──────┘    └────┬──────────┘
                 │                │
          ┌──────▼────────────────▼──┐
          │     module.compute      │  ← Last (needs ALB TG + DB endpoint)
          │     EC2/ECS, ASG, SG    │
          │     User data (DB conn) │
          └─────────────────────────┘
```

## 5. Root Module Wiring (environments/prod/main.tf)

```hcl
# ────────────────────────────────────────────
# Secrets (no dependencies — created first)
# ────────────────────────────────────────────
module "secrets" {
  source       = "../../modules/secrets"
  project_name = var.project_name
  environment  = var.environment
}

# ────────────────────────────────────────────
# Networking
# ────────────────────────────────────────────
module "vpc" {
  source             = "../../modules/vpc"
  vpc_cidr           = var.vpc_cidr
  environment        = var.environment
  project_name       = var.project_name
  az_count           = 3                          # 3 AZs for prod
  enable_nat_gateway = true                       # Required for private subnet internet
  single_nat_gateway = false                      # One per AZ for HA in prod
}

# ────────────────────────────────────────────
# Database (depends on VPC + Secrets)
# ────────────────────────────────────────────
module "database" {
  source            = "../../modules/database"
  vpc_id            = module.vpc.vpc_id           # ← Wiring
  subnet_ids        = module.vpc.private_subnet_id_list  # ← Wiring
  db_password_arn   = module.secrets.db_password_arn      # ← Wiring
  engine            = "postgres"
  engine_version    = "16.3"
  instance_class    = "db.r6g.large"              # Prod-scale
  multi_az          = true                        # HA for prod
  environment       = var.environment
  project_name      = var.project_name
}

# ────────────────────────────────────────────
# Load Balancer (depends on VPC)
# ────────────────────────────────────────────
module "alb" {
  source          = "../../modules/alb"
  vpc_id          = module.vpc.vpc_id             # ← Wiring
  subnet_ids      = module.vpc.public_subnet_id_list     # ← Wiring
  certificate_arn = var.certificate_arn
  environment     = var.environment
  project_name    = var.project_name
}

# ────────────────────────────────────────────
# Compute (depends on VPC + ALB + Database)
# ────────────────────────────────────────────
module "compute" {
  source            = "../../modules/compute"
  vpc_id            = module.vpc.vpc_id           # ← Wiring
  subnet_ids        = module.vpc.private_subnet_id_list  # ← Wiring
  target_group_arn  = module.alb.target_group_arn         # ← Wiring
  db_endpoint       = module.database.endpoint            # ← Wiring
  db_secret_arn     = module.secrets.db_password_arn       # ← Wiring
  instance_type     = "t3.medium"                 # Prod-scale
  min_size          = 2
  max_size          = 10
  environment       = var.environment
  project_name      = var.project_name
}
```

## 6. Variable Flow: From tfvars to Resources

```
terraform.tfvars (or -var-file)
  │
  │  vpc_cidr = "10.2.0.0/16"
  │  environment = "prod"
  │  instance_type = "t3.medium"
  │
  ▼
environments/prod/variables.tf
  │
  │  variable "vpc_cidr" { type = string }
  │  variable "environment" { type = string }
  │
  ▼
environments/prod/main.tf (root module)
  │
  │  module "vpc" { vpc_cidr = var.vpc_cidr }
  │  module "compute" { vpc_id = module.vpc.vpc_id }
  │
  ▼
modules/vpc/variables.tf
  │
  │  variable "vpc_cidr" { type = string }
  │
  ▼
modules/vpc/main.tf
  │
  │  resource "aws_vpc" "main" { cidr_block = var.vpc_cidr }
  │
  ▼
AWS API: CreateVpc(CidrBlock: "10.2.0.0/16")
```

## 8. State Isolation

```
S3 State Bucket: company-terraform-state
├── dev/
│   └── main/terraform.tfstate        # Dev environment state
├── staging/
│   └── main/terraform.tfstate        # Staging environment state
└── prod/
    └── main/terraform.tfstate        # Prod environment state

Each environment:
  - Separate state file (blast radius isolation)
  - Separate backend.tf with unique key
  - Can use separate AWS accounts + IAM roles
  - Native S3 locking (use_lockfile = true)
```

## 9. CI/CD Integration

```
PR created (any env):
  ├── terraform fmt -check
  ├── terraform validate
  ├── tflint
  ├── checkov (security scan)
  └── terraform plan → posted as PR comment

Merge to main:
  ├── dev: auto-apply
  ├── staging: auto-apply (after dev succeeds)
  └── prod: manual approval gate → apply

Deploy order: always dev → staging → prod
A failure in dev blocks staging and prod.
```

## 10. Secrets Flow

```
module.secrets:
  ├── random_password.db (generated, never seen by humans)
  ├── aws_secretsmanager_secret (created)
  └── aws_secretsmanager_secret_version (password stored)
         │
         ▼
module.database:
  └── aws_db_instance.main
         password = random_password (via module wiring)
         
module.compute:
  └── EC2/ECS reads password from Secrets Manager at runtime
      (via IAM role, NOT from Terraform outputs)
```

## 11. Interview Answer Template

> *"Walk me through how you'd design Terraform for a 3-tier AWS application."*

**Structure your answer in this order:**

1. **Directory layout**: "I'd use directory-per-environment with shared modules. `environments/dev/`, `environments/prod/`, and a shared `modules/` directory."

2. **Module boundaries**: "Five modules: vpc (networking foundation), alb (load balancing), compute (EC2/ECS), database (RDS), secrets (Secrets Manager + random_password). Each has clear inputs (variables) and outputs."

3. **Dependency order**: "Secrets first (no deps), then VPC, then ALB and database in parallel (both need VPC), then compute last (needs ALB target group and DB endpoint)."

4. **Variable flow**: "Environment-specific `terraform.tfvars` → root module `variables.tf` → passed to modules via module block arguments. Dev gets `t3.micro`, prod gets `t3.medium`."

5. **State**: "Separate S3 state files per environment with native locking. `prod/main/terraform.tfstate` is isolated from dev. KMS encryption. Restricted IAM."

6. **CI/CD**: "GitHub Actions with OIDC. Plan on PR (posted as comment), apply on merge. Sequential: dev → staging → prod. Prod requires manual approval."

7. **Secrets**: "`random_password`→ Secrets Manager. DB password never exposed to humans. Application reads from Secrets Manager at runtime via IAM role."

8. **Safety**: "`prevent_destroy` on prod database. `default_tags` on provider. `checkov` security scanning in CI."

## 12. Project Checklist

Use this when starting a new Terraform project:

```
□ Directory structure (environments/ + modules/)
□ State backend (S3 + native locking + versioning + KMS)
□ Provider config (version pinned, default_tags)
□ Module interfaces (variables with validation, comprehensive outputs)
□ Module wiring (root main.tf connects modules via outputs → variables)
□ Environment tfvars (different values per env)
□ .gitignore (*.tfstate, .terraform/, sensitive *.tfvars)
□ CI/CD pipeline (plan on PR, apply on merge, OIDC auth)
□ Secrets handling (random_password + Secrets Manager)
□ Lifecycle rules (prevent_destroy on critical resources)
□ Security scanning (checkov/tfsec in CI)
□ README (for each module and each environment)
```
